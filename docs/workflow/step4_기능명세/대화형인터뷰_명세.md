# 기능명세 — 대화형 인터뷰
updated: 2026-05-28

> 이 문서는 MVP 확장(V2) 중 "대화형 인터뷰" 기능의 설계 기준이다.
> 기존 7필드 CLI 폼을 CEO 주도 대화형 인터뷰로 교체한다.
> 변경 시 ADR 작성 + 핑구 승인 필수.

---

## 개요

현재 시스템은 CLI에서 7개 필드(이름, 직업, 특기, 성격, 타겟연령대, SNS경험, 목표)를
순차적으로 입력받는다. 이 방식은 정보의 깊이가 얕고, 사용자가 무엇을 써야 할지
모르는 경우가 많다. 실제 기획사 첫 상담에서는 대화를 통해 자연스럽게 정보를 이끌어낸다.

대화형 인터뷰는 CEO 에이전트가 첫 상담자 역할로 10회 이상의 대화 턴을 통해
크리에이터의 정보를 자연스럽게 수집한다. 짧은 답변에는 후속 질문으로 파고들고,
상세한 답변에는 빠르게 다음 주제로 넘어간다. 하드코딩된 턴 수가 아니라
LLM이 정보 충분성을 스스로 판단하여 인터뷰를 종료한다.

수집 결과는 기존 7개 필드 + 확장 4개 필드(가용시간, 촬영환경, 카메라경험, 예산)를
포함하는 구조화된 딕셔너리로 반환되어, 기존 파이프라인과 완전히 호환된다.

## 클래스 설계

### InterviewEngine

```
InterviewEngine
├── 속성
│   ├── llm_client: LLMClient          # LLM 호출 클라이언트
│   ├── system_prompt: str             # CEO 인터뷰 시스템 프롬프트
│   ├── conversation: list[dict]       # 대화 이력 (role/content 쌍)
│   ├── extracted: dict                # 현재까지 추출된 필드
│   ├── max_turns: int = 25           # 안전장치 최대 턴 수
│   ├── min_turns: int = 5            # 최소 턴 수 (너무 빠른 종료 방지)
│   └── state: str                     # "interviewing" | "extracting" | "confirming" | "done"
│
├── 핵심 메서드
│   ├── start() -> str
│   │   # CEO 첫 인사 + 첫 질문 생성
│   │   # conversation에 assistant 메시지 추가
│   │   # 반환: CEO 첫 메시지 문자열
│   │
│   ├── reply(user_message: str) -> InterviewResponse
│   │   # 사용자 답변 처리 → LLM 호출 → 다음 질문 또는 요약
│   │   # conversation에 user + assistant 메시지 추가
│   │   # extracted 업데이트
│   │   # 반환: InterviewResponse 객체
│   │
│   ├── confirm(approved: bool, corrections: dict | None) -> dict
│   │   # 사용자가 추출 결과 확인/수정
│   │   # approved=True → 최종 subject dict 반환
│   │   # approved=False + corrections → 수정 반영 후 재확인 요청
│   │   # 반환: 최종 subject dict
│   │
│   ├── get_subject() -> dict
│   │   # 기존 7필드 + 확장 4필드 딕셔너리 반환
│   │   # 미수집 필드는 "정보 없음"으로 채움
│   │   # 반환: main.py의 collect_inputs()과 호환되는 dict
│   │
│   └── save_conversation(path: Path) -> None
│       # 대화 이력을 마크다운으로 저장 (장애 복구용)
│
└── 내부 메서드
    ├── _build_messages() -> list[dict]
    │   # system_prompt + conversation → LLM 호출용 메시지 배열
    │
    ├── _parse_llm_response(raw: str) -> dict
    │   # LLM JSON 응답 파싱 (type, message, extracted_so_far, sufficient)
    │   # JSON 파싱 실패 시 → 텍스트를 question으로 래핑
    │
    ├── _check_force_extract() -> bool
    │   # max_turns 도달 여부 확인
    │   # True → 강제 추출 단계 전환
    │
    └── _merge_extracted(new_fields: dict) -> None
        # 새로 추출된 필드를 기존 extracted에 병합
        # 기존 값이 "정보 없음"이고 새 값이 있으면 덮어씀
```

### InterviewResponse (데이터 클래스)

```python
@dataclass
class InterviewResponse:
    type: str            # "question" | "summary" | "error"
    message: str         # CEO가 사용자에게 보여줄 메시지
    extracted: dict      # 현재까지 추출된 필드 스냅샷
    sufficient: bool     # 정보 수집 충분 여부
    turn_count: int      # 현재 턴 수
```

## 인터뷰 플로우

### 전체 흐름도

```
[시작]
  │
  ├── InterviewEngine.start()
  │   └── CEO 첫 인사 + 첫 질문 → 사용자에게 표시
  │
  ├── [인터뷰 루프] ← state="interviewing"
  │   ├── 사용자 입력 수신
  │   ├── InterviewEngine.reply(user_input)
  │   │   ├── LLM 호출 (conversation + system_prompt)
  │   │   ├── JSON 응답 파싱
  │   │   │   ├── type="question" → 다음 질문 표시 → 루프 계속
  │   │   │   └── type="summary" + sufficient=true → 추출 단계로
  │   │   └── extracted_so_far 업데이트
  │   │
  │   ├── [안전장치] max_turns 도달 시
  │   │   └── 강제 추출 단계 전환
  │   │
  │   └── [특수 입력 처리]
  │       ├── "이미 다 말했어" → LLM에 충분성 재확인 요청
  │       ├── 빈 입력 → "조금 더 자세히 말씀해주시겠어요?" 재질문
  │       └── "그만" / "종료" → 현재까지 수집된 것으로 추출
  │
  ├── [추출 단계] ← state="extracting"
  │   ├── CEO가 수집 정보 요약 메시지 생성
  │   ├── 구조화된 필드 딕셔너리 생성
  │   └── 사용자에게 확인 요청 표시
  │
  ├── [확인 단계] ← state="confirming"
  │   ├── 사용자 "확인" → get_subject() 반환
  │   ├── 사용자 수정 요청 → corrections 반영 → 재확인
  │   └── 3회 수정 반복 시 → 현재 상태로 확정
  │
  └── [완료] ← state="done"
      └── subject dict 반환 → 기존 파이프라인 진입
```

### 턴별 상세

| 턴 | CEO 행동 | 목표 수집 필드 |
|----|---------|--------------|
| 1 | 인사 + 이름/직업 질문 | 이름, 직업 |
| 2~3 | 직업 심화 (경력, 전문 분야, 일과) | 특기, 가용시간 |
| 4~5 | 성격/소통 스타일 탐색 | 성격, 카메라경험 |
| 6~7 | SNS 경험 + 목표 탐색 | SNS경험, 목표 |
| 8~9 | 타겟/환경/예산 탐색 | 타겟연령대, 촬영환경, 예산 |
| 10+ | 부족한 필드 보강 또는 요약 | 미수집 필드 보완 |

위 순서는 가이드라인이며, LLM이 대화 흐름에 따라 자율적으로 조정한다.

## 프롬프트 설계

### 파일 위치
`src/prompts/ceo/interview.md`

### 프롬프트 구조

```markdown
# [ROLE]
당신은 인플루언서 기획사의 CEO입니다.
지금은 새로운 크리에이터와의 첫 상담 시간입니다.
10년 경력의 기획사 대표답게 따뜻하면서도 전문적인 태도로 대화합니다.
상대방이 편안하게 이야기할 수 있도록 자연스러운 대화를 이끕니다.

# [GOAL]
대화를 통해 크리에이터의 정보를 자연스럽게 수집합니다.
수집해야 할 필드:
- 필수 7개: 이름, 직업, 특기, 성격, 타겟연령대, SNS경험, 목표
- 확장 4개: 가용시간, 촬영환경, 카메라경험, 예산
정보가 충분히 수집되었다고 판단되면 sufficient=true를 반환합니다.

# [GUIDE]
- 한 번에 1~2개 질문만 합니다. 질문 폭격 금지.
- 상대방이 짧게 답하면 → 후속 질문으로 깊이 파고듭니다.
  예: "미용사요" → "어떤 분야를 주로 하세요? 컬러, 커트, 펌 중 특히 자신 있는 게 있나요?"
- 상대방이 상세하게 답하면 → 핵심을 정리하고 다음 주제로 넘어갑니다.
- 격려와 공감을 적절히 섞습니다. "오, 그거 정말 좋은 강점이네요!"
- 너무 사적인 질문은 피합니다 (나이, 수입 등 직접 묻지 않음).
- 예산은 "혹시 장비나 촬영에 투자할 수 있는 여유가 있으세요?" 수준으로 간접 질문합니다.

# [INPUT]
대화 이력이 messages 배열로 제공됩니다.
이전 턴에서 추출한 필드 정보가 함께 제공됩니다.

# [EXTRACTED_FIELDS]
아래 필드를 대화에서 추출합니다. 직접 묻지 않아도 대화 맥락에서 유추 가능하면 추출합니다.

| 필드 | 설명 | 추출 조건 |
|------|------|----------|
| 이름 | 크리에이터 이름 | 명시적 언급 |
| 직업 | 현재 직업/전문 분야 | 명시적 언급 |
| 특기 | 잘하는 것, 두드러지는 능력 | 명시적 또는 대화 맥락 유추 |
| 성격 | 외향/내향, 소통 스타일 | 대화 태도에서 유추 가능 |
| 타겟연령대 | 어필하고 싶은 대상 | 명시적 언급 |
| SNS경험 | 현재 운영 중인 채널 | 명시적 언급 |
| 목표 | 6개월~1년 목표 | 명시적 언급 |
| 가용시간 | 콘텐츠 제작에 쓸 수 있는 시간 | 명시적 또는 일과에서 유추 |
| 촬영환경 | 촬영 가능한 장소/환경 | 명시적 또는 직업에서 유추 |
| 카메라경험 | 촬영/편집 경험 수준 | 명시적 언급 |
| 예산 | 장비/촬영 투자 여유 | 간접 질문으로 파악 |

# [CONSTRAINTS]
- 한 턴에 3개 이상 질문 금지
- 사용자가 답변을 거부한 필드는 "정보 없음"으로 표시하고 더 이상 묻지 않음
- 면접 분위기 금지 — 자연스러운 대화 유지
- 사용자 답변을 임의로 해석하거나 과장 금지
- 한국어로만 대화
- 이전 턴에서 이미 수집한 정보를 중복 질문하지 않음

# [FORMAT]
반드시 아래 JSON 형식으로 응답합니다. JSON 외 텍스트 금지.

{
  "type": "question" 또는 "summary",
  "message": "사용자에게 보여줄 메시지 (자연스러운 한국어)",
  "extracted_so_far": {
    "이름": "추출된 값 또는 null",
    "직업": "추출된 값 또는 null",
    ...
  },
  "sufficient": false 또는 true
}

type="question": 아직 정보가 부족하여 추가 질문 필요
type="summary": 정보가 충분하여 수집 결과 요약 제시
sufficient=true: 필수 7개 필드 중 5개 이상 수집 완료 + 핵심 정보 확보
```

## 입출력 정의

### 입력 (대화)

InterviewEngine은 사용자와의 대화 턴을 입력으로 받는다.

```python
# start() 호출 시 입력 없음
ceo_greeting: str = engine.start()

# reply() 호출 시 사용자 메시지 입력
response: InterviewResponse = engine.reply("저는 김민수라고 합니다. 미용사예요.")
```

### 출력 (subject dict)

인터뷰 완료 후 반환되는 딕셔너리. 기존 collect_inputs()과 동일한 키 구조 + 확장 필드.

```python
{
    # 기존 7개 필드 (collect_inputs 호환)
    "이름": "김민수",
    "직업": "미용사 (헤어컬러 전문, 3년차)",
    "특기": "헤어컬러링, 특히 밝은 톤 블리치",
    "성격": "내향적, 말보다 손이 먼저 움직이는 타입",
    "타겟연령대": "20~30대 여성",
    "SNS경험": "개인 인스타그램 있으나 일상 위주, 팔로워 50명 수준",
    "목표": "6개월 내 인스타그램 팔로워 3000명 + 헤어 포트폴리오 구축",

    # 확장 4개 필드 (V2 신규)
    "가용시간": "평일 퇴근 후 1~2시간, 주말 오전",
    "촬영환경": "살롱 내부 (조명 양호, 거울 배경 활용 가능)",
    "카메라경험": "스마트폰 촬영만 해봄, 편집 경험 없음",
    "예산": "월 5만원 수준 소액 투자 가능"
}
```

미수집 필드는 "정보 없음"으로 채워진다.

### 호환성

| 기존 코드 | 동작 |
|----------|------|
| `CEO.run(context)` | context["대상자"]에 동일 키 구조 전달 → 변경 없음 |
| `init_context(subject)` | subject dict 키 확장 → 기존 키 유지, 신규 키 추가 |
| `OutputValidator` | "대상자.이름" 참조 → 변경 없음 |
| 에이전트 프롬프트 | 확장 필드를 참조하도록 프롬프트 업데이트 필요 |

## CLI 연동

### 명령줄 인자

```python
parser.add_argument(
    "--interview",
    action="store_true",
    default=True,
    help="CEO 대화형 인터뷰로 정보 수집 (기본값)",
)
parser.add_argument(
    "--legacy",
    action="store_true",
    help="기존 7필드 폼 입력 모드 사용",
)
```

### main.py 분기 로직

```python
def main():
    args = parser.parse_args()

    if args.legacy:
        subject = collect_inputs()          # 기존 7필드 폼
    else:
        engine = InterviewEngine(llm_client)
        subject = run_interview_cli(engine)  # 대화형 인터뷰

    context = init_context(subject)
    # ... 이하 기존 파이프라인 동일
```

(... 10+ 턴 대화 ...)

CEO: 감사합니다! 지금까지 말씀해주신 내용을 정리해볼게요.

[수집된 정보]
- 이름: 김민수
- 직업: 미용사 (헤어컬러 전문, 3년차)

## 웹 API 연동 (설계만)

### 엔드포인트

| 메서드 | 경로 | 설명 | 요청 | 응답 |
|--------|------|------|------|------|
| POST | /api/interview/start | 인터뷰 시작 | `{"session_id": "..."}` | `{"message": "CEO 첫 인사", "turn": 0}` |
| POST | /api/interview/reply | 사용자 답변 전송 | `{"session_id": "...", "message": "..."}` | `InterviewResponse JSON` |
| POST | /api/interview/confirm | 추출 결과 확인 | `{"session_id": "...", "approved": true, "corrections": {}}` | `{"subject": {...}, "confirmed": true}` |
| GET | /api/interview/status | 현재 인터뷰 상태 | `session_id` (query) | `{"state": "...", "turn": N, "extracted": {...}}` |

### 세션 관리

- SessionManager에 interview_engine 인스턴스 보관
- 세션별 독립 InterviewEngine 인스턴스
- 인터뷰 완료 시 세션의 subject에 결과 주입 후 기존 파이프라인 연결

### SSE 연동

- 인터뷰 턴마다 SSE 이벤트 발행: `event: interview_turn`
- 프론트엔드에서 채팅 UI로 실시간 표시
- 기존 에이전트 실행 SSE와 동일한 EventEmitter 활용

## 엣지케이스

| 상황 | 감지 조건 | 처리 방법 |
|------|----------|----------|
| 1단어 답변 반복 | 연속 3턴 이상 답변 길이 5자 미만 | CEO가 "좀 더 편하게 이야기해주셔도 돼요" + 구체적 예시 제시 |
| 답변 거부 | "그건 말하기 싫어요", "패스" 등 | 해당 필드 "정보 없음" 처리, 더 이상 해당 주제 질문 안 함 |
| 20턴 초과 | turn_count > max_turns | CEO가 현재까지 수집된 정보로 요약, 강제 추출 단계 진입 |
| "다 말했어" | 사용자 명시적 종료 의사 | LLM에 충분성 재확인 요청, 부족해도 현재 상태로 추출 |
| API 장애 | LLM 호출 실패 (429/503) | 대화 이력 파일 저장 + "잠시 후 다시 시도해주세요" 메시지 |
| JSON 파싱 실패 | LLM 응답이 유효한 JSON이 아님 | 응답 텍스트를 message로 래핑, type="question" 기본값 |
| 빈 입력 | 사용자가 Enter만 입력 | "혹시 질문이 어려우셨나요? 다른 방식으로 여쭤볼게요" 재질문 |
| 대화 중단 후 재개 | save_conversation() 파일 존재 | 파일에서 대화 이력 로드 후 이어서 진행 |
| 확인 단계에서 대폭 수정 | corrections 5개 이상 필드 수정 | 수정 반영 후 CEO가 변경사항 재확인 |
| 한글 인코딩 문제 | Windows cp949 환경 | 기존 main.py UTF-8 래핑 그대로 사용 |
| 사용자가 영어로 답변 | 영어 답변 감지 | CEO는 한국어로 계속 대화, 영어 답변 내용은 그대로 추출 |
