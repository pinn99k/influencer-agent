## 개요

현재 시스템의 에이전트는 문자열을 반환하고 지시적 프롬프트(약 24,600단어)를 따른다.
이 명세는 MVP 확장(V2)으로 세 가지 핵심 변경을 정의한다.

1. **구조화된 출력(AgentOutput)** — 에이전트가 산출물 외에 질문, 의견, 신뢰도를 함께 반환
2. **범위 제한 접근(AgentContext)** — 에이전트별로 읽을 수 있는 파일과 도구를 제한
3. **역할 가이드 프롬프트** — 지시적 프롬프트를 역할 중심 가이드로 전환 (24,600 -> 10,000단어)

목적: 에이전트가 "시킨 대로 채우기"에서 "역할 내에서 판단하기"로 전환한다.
CEO는 에이전트의 질문과 의견을 받아 전략적 판단을 내리고, 필요 시 회장에게 보고한다.

## AgentOutput 데이터 클래스

### 위치
`src/agents/agent_output.py`

### 필드 정의

```python
from dataclasses import dataclass, field

@dataclass
class AgentOutput:
    content: str                        # 기존 마크다운 산출물
    questions: list[str] = field(default_factory=list)   # CEO에게 보내는 질문
    comments: list[str] = field(default_factory=list)    # 에이전트 의견/제안
    confidence: float = 1.0             # 0.0~1.0 (1.0 = 확신, 0.0 = 불확실)
    metadata: dict = field(default_factory=dict)         # 부가 정보
```

### 필드 상세

| 필드 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| content | str | (필수) | 기존 마크다운 산출물. OutputValidator가 검증하는 대상 |
| questions | list[str] | [] | CEO에게 전달할 질문. 예: "타겟 연령대가 넓습니다. 20대 초반으로 좁힐까요?" |
| comments | list[str] | [] | 에이전트의 판단/의견. 예: "유튜브가 더 적합하다고 봅니다. 시각적 콘텐츠 강점 때문입니다" |
| confidence | float | 1.0 | 산출물 전체 신뢰도. 정보 부족 시 낮아짐 |
| metadata | dict | {} | 검색 쿼리, 참조 파일, 실행 시간 등 부가 정보 |

### LLM 출력 내 JSON 블록 형식

에이전트의 LLM 응답에서 AgentOutput JSON을 추출하는 구분자:

```
(마크다운 산출물 본문)

---AGENT_OUTPUT---
{
  "questions": ["타겟 연령대가 20~40대로 넓습니다. 좁힐까요?"],
  "comments": ["SNS 경험이 전무하므로 릴스보다 정적 이미지 시작을 권합니다"],
  "confidence": 0.75,
  "metadata": {"search_queries": ["미용사 유튜버 구독자"], "files_referenced": ["01_대상분석.md"]}
}
---END_OUTPUT---
```

### from_raw 파서 (관대한 파싱)

```python
@classmethod
def from_raw(cls, text: str) -> "AgentOutput":
    """LLM 원본 텍스트에서 AgentOutput 파싱.

    규칙:
    1. ---AGENT_OUTPUT--- ... ---END_OUTPUT--- 블록이 있으면 JSON 파싱
    2. JSON 파싱 실패 시 전체 텍스트를 content로 사용 (questions/comments 비움)
    3. 블록이 없으면 전체 텍스트를 content로 사용

    즉, 어떤 경우에도 에러를 발생시키지 않는다. 기존 문자열 출력도 그대로 동작한다.
    """
```

파싱 우선순위:
1. `---AGENT_OUTPUT---` 구분자 검색
2. 구분자 앞부분 = content, 구분자 내부 = JSON
3. JSON에서 questions, comments, confidence, metadata 추출
4. JSON 키가 없으면 기본값 사용
5. 구분자 자체가 없으면 text 전체 = content, 나머지 빈 값

### 사용 예시

```python
# 신규 형식 (JSON 블록 포함)
output = AgentOutput.from_raw(llm_response)
print(output.content)      # 마크다운 산출물
print(output.questions)    # ["타겟 연령대가 넓습니다. 좁힐까요?"]
print(output.confidence)   # 0.75

# 기존 형식 (순수 문자열) -- 하위 호환
output = AgentOutput.from_raw("## 대상 분석 결과\n...")
print(output.content)      # "## 대상 분석 결과\n..."
print(output.questions)    # []
print(output.confidence)   # 1.0
```

## AgentContext 접근 권한

### 위치
`src/agents/agent_context.py`

### 설계 목적

현재 모든 에이전트가 전체 context dict를 수신한다.
AgentContext는 에이전트별로 접근 가능한 범위를 제한하여:
- 불필요한 토큰 소비 방지
- 역할 경계 강제 (플랫폼추천이 컨셉기획 결과를 참조하는 사고 방지)
- CEO가 전략적 요약(ceo_summary)을 생성하여 하위 에이전트에 방향 제시

### AGENT_SCOPES 정의

```python
AGENT_SCOPES = {
    "CEO": {
        "read": ["*"],                          # 모든 컨텍스트 키 접근
        "knowledge": ["경영/"],              # 경영 도메인 지식
        "outputs": ["*"],                       # 모든 산출물 읽기
    },
    "대상분석": {
        "read": ["대상자"],                    # 대상자 기본 정보만
        "knowledge": ["부서/기획본부/대상분석/"],
        "outputs": [],                          # 이전 산출물 없음 (첫 에이전트)
    },
    "경쟁분석": {
        "read": ["대상자", "ceo_summary"],      # 대상자 + CEO 전략 요약
        "knowledge": ["부서/기획본부/경쟁분석/"],
        "outputs": ["01_대상분석.md"],          # 대상분석 산출물 참조
        "tools": ["serper"],                    # 웹 검색 도구 허용
    },
    "플랫폼추천": {
        "read": ["대상자", "ceo_summary"],
        "knowledge": ["부서/기획본부/플랫폼추천/"],
        "outputs": ["01_대상분석.md", "02_경쟁분석.md"],
    },
    "컨셉기획": {
        "read": ["대상자", "ceo_summary"],
        "knowledge": ["부서/기획본부/컨셉기획/"],
        "outputs": ["01_대상분석.md", "02_경쟁분석.md", "03_플랫폼추천.md"],
    },
}
```

### build_context 함수

```python
def build_context(
    agent_name: str,
    full_context: dict,
    file_manager: FileManager,
) -> dict:
    """에이전트별 허용 범위만 추출하여 필터링된 context를 반환.

    매개변수:
        agent_name: AGENT_SCOPES 키 (display_name)
        full_context: CEO가 관리하는 전체 컨텍스트
        file_manager: 산출물 파일 읽기용

    반환:
        필터링된 context dict (허용 키만 포함)

    동작:
        1. scope["read"]에 해당하는 키만 full_context에서 추출
        2. scope["outputs"]에 해당하는 산출물을 file_manager로 읽어 추가
        3. "*"는 모든 키/파일 접근 허용
    """
```

### CEO 전략 요약 (ceo_summary)

CEO가 에이전트 실행 전 생성하는 1단락 전략 방향 요약:

```
ceo_summary = "이 크리에이터는 내향적 미용사로, 시각적 강점(헤어컬러)을 살린
인스타그램 릴스 중심 전략이 유력하다. 경쟁분석 시 미용사 유튜버 포지셔닝
공백에 주목하라. 20대 여성 타겟 우선."
```

- 생성 시점: PlanningDepartment.run() 호출 직전, CEO가 대상자 정보 기반 생성
- 프롬프트: src/prompts/ceo/goal_interpretation.md 활용 (기존)
- 전달: full_context["ceo_summary"]에 저장 -> build_context가 각 에이전트에 전달

## 프롬프트 전환 전략

### 현재 구조 (약 24,600단어, 에이전트당 약 270줄)

```
[ROLE]          역할 정의 + 사고 방식
[GOAL]          목표 + 완료 조건
[CONSTRAINTS]   금지 사항 (15-20개)
[FORMAT]        출력 형식 (섹션 헤더 + 상세 템플릿)
[INPUT]         입력 변수 설명
[THINKING]      단계별 분석 순서 (5-8단계)
[EXAMPLES]      좋은/나쁜 예시 (각 3-5개)
[SELF-CHECK]    자기 검증 체크리스트 (10-15항목)
```

### 목표 구조 (약 10,000단어, 에이전트당 약 100줄)

```
[ROLE]            역할 + 사고 방식 (축소 유지)
[GOAL]            완료 조건 (유지)
[THINKING_GUIDE]  자연어 사고 흐름 (신규, [THINKING] 대체)
[ACCESS]          접근 가능 파일/도구 (신규)
[OUTPUT]          AgentOutput JSON + 콘텐츠 형식 (간소화)
[CONSTRAINTS]     핵심 금지만 (5-7개로 축소)
```

### 섹션별 전환 상세

| 현재 섹션 | 전환 방식 | 이동 위치 |
|----------|----------|----------|
| [ROLE] | 유지, 축소 | 프롬프트 내 [ROLE] |
| [GOAL] | 유지 | 프롬프트 내 [GOAL] |
| [THINKING] | [THINKING_GUIDE]로 교체 | 프롬프트 내 (자연어 가이드) |
| [ACCESS] | 신규 추가 | 프롬프트 내 |
| [FORMAT] 헤더 | 유지 (필수) | 프롬프트 내 [OUTPUT] |
| [FORMAT] 상세 | knowledge로 이동 | knowledge/부서/기획본부/{agent}/출력형식.md |
| [CONSTRAINTS] | 핵심만 유지 | 프롬프트 내 [CONSTRAINTS] |
| [INPUT] | 제거 | [ACCESS]로 통합 |
| [EXAMPLES] | knowledge로 이동 | knowledge/부서/기획본부/{agent}/예시.md |
| [SELF-CHECK] | 제거 | validator가 처리 (코드 레벨) |

### [FORMAT] 헤더 보존 규칙 (중요)

OutputValidator.RULES의 required_sections와 정확히 일치해야 한다.
프롬프트 전환 시 아래 헤더는 절대 변경 금지:

| 에이전트 | 보존 필수 헤더 |
|---------|--------------|
| 대상분석 | `## 대상 분석 결과`, `### 강점`, `### 약점`, `### 차별점` |
| 경쟁분석 | `## 경쟁 분석 결과`, `### 유사 포지션 크리에이터`, `### 시장 공백 기회`, `### 차별 전략` |
| 플랫폼추천 | `## 플랫폼 추천 결과`, `### 1순위 플랫폼`, `### 2순위 플랫폼`, `### 비추천 플랫폼` |
| 컨셉기획 | `## 컨셉 기획 결과`, `#### 컨셉 A:`, `#### 컨셉 B:`, `#### 컨셉 C:`, `### 영상 아이디어`, `### 4주 콘텐츠 캘린더`, `### 촬영 실행 가이드`, `### 업로드 최적화` |

### [THINKING_GUIDE] 예시 (대상분석)

기존 [THINKING] (단계별 명령):
```
1단계: 입력 정보에서 직업, 특기, 성격을 추출하라
2단계: 강점 후보 5개를 도출하고 상위 3개를 선택하라
3단계: 약점을 솔직하게 3개 도출하라
...
```

신규 [THINKING_GUIDE] (자연어 가이드):
```
이 사람이 카메라 앞에 서면 어떤 느낌일지 상상해 보라.
직업에서 자연스럽게 나오는 강점은 무엇인가?
성격이 콘텐츠 제작에 어떤 영향을 줄지 생각하라.
솔직하게 — 이 사람의 약점은 무엇인가? 회피하지 마라.
경쟁자와 다른 점이 정말 있는가? 없다면 만들어야 한다.
확신이 안 서면 confidence를 낮추고 질문을 남겨라.
```

### [ACCESS] 예시 (경쟁분석)

```
[ACCESS]
접근 가능:
- 대상자 기본 정보
- CEO 전략 요약

접근 불가:
- 다른 에이전트의 산출물 (02 이후)

```

### [OUTPUT] 예시

```

산출물을 마크다운으로 작성한 후, 맨 끝에 아래 블록을 추가하라:
```

## CEO 질문 처리

### 3단계 판단 체계

에이전트가 questions 필드로 보낸 질문을 CEO가 분류하고 처리한다.

| 단계 | 유형 | 설명 | 처리 |
|------|------|------|------|
| 1 | 전략적 질문 | 목표/방향 변경 관련 | 회장(핑구)에게 보고 |
| 2 | 전술적 질문 | 분석 범위/우선순위 | CEO 자체 판단 후 답변 |
| 3 | 데이터 질문 | 누락 정보 | 컨텍스트 확인 -> 없으면 회장에게 질문 |

### 질문 분류 예시

| 질문 | 유형 | CEO 처리 |
|------|------|---------|
| "타겟 연령대를 20대 초반으로 좁힐까요?" | 전략 | 회장 보고 (조건 3: 전략 방향 전환) |
| "경쟁분석에서 미용사만 볼까요, 뷰티 전체를 볼까요?" | 전술 | CEO 자체 판단: "미용사 중심, 뷰티 전체는 참고만" |
| "대상자의 경력 연수가 없습니다" | 데이터 | 컨텍스트 확인 -> 없으면 회장에게 질문 |

### _handle_agent_questions 메서드

```python
def _handle_agent_questions(
    self,
    agent_name: str,
    agent_output: AgentOutput,
    context: dict,
) -> dict:
    """에이전트 질문을 3단계로 분류하고 처리.

    매개변수:
        agent_name: 질문을 보낸 에이전트 이름
        agent_output: AgentOutput (questions 필드 사용)
        context: 현재 전체 컨텍스트

    반환:
        {
            "answers": dict[str, str],         # 질문 -> 답변 (CEO가 직접 답한 것)
            "escalated": list[str],            # 회장에게 보고할 질문
            "data_requests": list[str],        # 회장에게 물어볼 데이터 질문
        }

    동작:
        1. questions가 비어있으면 빈 결과 반환
        2. LLM 호출로 질문 분류 (question_handler.md 프롬프트)
        3. 전략적 질문 -> escalated 목록에 추가
        4. 전술적 질문 -> CEO가 LLM으로 답변 생성
        5. 데이터 질문 -> 컨텍스트에서 검색 -> 없으면 data_requests에 추가
    """
```

```
당신은 AI 기획사의 CEO이다.
하위 에이전트가 보낸 질문을 분류하고 처리한다.

[GOAL]

- DATA: 누락 정보 요청 -> 컨텍스트에서 찾거나 회장에게 질문

[OUTPUT]
JSON 형식으로 출력:
{
```

질문은 파이프라인 완료 후 일괄 처리한다 (실시간 아님):

```

    -> 전술적 질문 -> CEO 자체 답변 -> 재실행 여부 판단
    -> 데이터 질문 -> 회장에게 전달
```

### _run_agent 변경

```python
def _run_agent(self, agent_name, agent, dept_context, result):
    # ... 기존 retry 루프 ...
    raw = agent.run(dept_context)              # str 반환 (기존)

    # V2: AgentOutput 파싱
    agent_output = AgentOutput.from_raw(raw)
    content = agent_output.content             # OutputValidator에 전달

    # 질문/의견/신뢰도 저장
    if agent_output.questions:
        result.agent_questions[agent_name] = agent_output.questions
    if agent_output.comments:
        result.agent_comments[agent_name] = agent_output.comments
    result.agent_confidence[agent_name] = agent_output.confidence

    # OutputValidator는 content만 검증 (기존 로직 변경 없음)
    val_result = OutputValidator.validate(agent_name, content, ...)

    # context에도 content만 저장 (다음 에이전트 호환)
    dept_context[agent.context_key] = content
    result.agent_results[agent.context_key] = content
    # ...
```

## 하위 호환 전략

### 원칙
기존 테스트와 코드가 깨지지 않아야 한다. 단계적 전환.

### AgentOutput.from_raw 관대한 파싱

| 입력 | 동작 | 결과 |
|------|------|------|
| JSON 블록 포함 | 정상 파싱 | content + questions + comments + confidence |
| JSON 블록 없음 (기존 문자열) | 전체를 content로 | content=전체, questions=[], confidence=1.0 |
| JSON 블록 있으나 파싱 실패 | 구분자 제거 후 content로 | content=전체(구분자 제거), questions=[] |
| JSON에 일부 키만 있음 | 있는 키만 사용 | 없는 키는 기본값 |

### 테스트 마이그레이션

기존 테스트에서 agent.run() 반환값을 문자열로 사용하는 코드:

```python
# 기존
result = agent.run(context)
assert "## 대상 분석 결과" in result

# 전환 후
result = agent.run(context)  # 여전히 str 반환 (BaseAgent.run 내부에서 from_raw 안 함)
assert "## 대상 분석 결과" in result
```

BaseAgent.run()은 여전히 str을 반환한다.
AgentOutput.from_raw()는 PlanningDepartment._run_agent()에서만 호출한다.
이렇게 하면 BaseAgent의 인터페이스 변경 없이 Department 레벨에서만 V2 기능을 사용한다.

### OutputValidator 변경 없음

OutputValidator.validate()는 str을 받는다.
PlanningDepartment가 AgentOutput.content를 추출하여 전달하므로 validator 코드 변경 불필요.

## 엣지케이스

### JSON 파싱 실패

| 상황 | 처리 |
|------|------|
| JSON 블록 내 잘못된 JSON | 구분자 제거, 전체를 content로 사용. 로그 경고 출력 |
| 구분자 시작만 있고 끝이 없음 | 구분자 이전까지를 content로 사용 |
| JSON에 예상 외 키 포함 | 무시 (알려진 키만 추출) |
| JSON 값 타입 불일치 (confidence="high") | 기본값 사용 (1.0) |

### 빈 질문/의견

| 상황 | 처리 |
|------|------|
| questions: [] | 정상. CEO 질문 처리 스킵 |
| questions: [""] | 빈 문자열 제거 후 처리 |
| questions: null | 빈 리스트로 변환 |

### 신뢰도 이상

| 상황 | 처리 |
|------|------|
| confidence: 0.0 | 유효. CEO에게 "신뢰도 0" 경고 전달 |
| confidence > 1.0 | 1.0으로 클리핑 |
| confidence < 0.0 | 0.0으로 클리핑 |
| confidence 누락 | 기본값 1.0 |

### AgentContext 접근 위반

| 상황 | 처리 |
|------|------|
| 에이전트가 scope 외 컨텍스트 키 요청 | build_context가 키를 제외하고 반환 (에러 아님) |
| ceo_summary가 아직 없을 때 | 빈 문자열로 전달 |
| 산출물 파일이 아직 없을 때 (첫 실행) | 해당 outputs 항목 스킵 |

### 프롬프트 전환 중간 상태

| 상황 | 처리 |
|------|------|
| 일부 에이전트만 신규 프롬프트 적용 | from_raw가 양쪽 다 처리하므로 문제없음 |
| 신규 프롬프트 에이전트가 JSON 블록 미출력 | from_raw가 전체를 content로 처리 |
| knowledge 예시 파일 미생성 상태 | _load_knowledge가 빈 문자열 반환 (기존 동작) |
