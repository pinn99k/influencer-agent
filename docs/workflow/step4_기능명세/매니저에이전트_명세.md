# 기능명세 — 매니저 에이전트
updated: 2026-05-28

---

## 개요

매니저 에이전트는 AI 기획사 시스템과 사용자(핑구) 사이의 **전용 알림/보고 채널**이다.

### 왜 필요한가
현재 시스템은 CEO가 전략 판단 + 사용자 소통을 모두 담당한다.
CEO의 본질은 전략과 조율이다. 사용자 친화적 알림, 주간 실행 카드, 성과 입력 요청 등은 CEO의 역할이 아니다.
매니저를 분리하면:
- CEO는 전략/판단에만 집중 (SRP)
- 사용자 대면 메시지 품질이 올라감 (전용 포맷팅)
- 알림 이력이 독립 관리됨 (추적 가능)

### CEO와의 차이

| 구분 | CEO | 매니저 |
|------|-----|--------|
| 본질 | 전략 판단, 에이전트 조율 | 사용자 대면 알림/보고 |
| 의사결정 | O (어떤 에이전트를 호출할지) | X (결정 권한 없음) |
| LLM 호출 | 판단용 (품질 평가, 재분석 결정) | 포맷팅용 (구조화된 데이터를 사용자 언어로 변환) |
| 파이프라인 위치 | 오케스트레이터 | 파이프라인 외부 서비스 |
| 등록 | AGENT_CLASSES 미포함 | AGENT_CLASSES 미포함 |

### 핵심 원칙
- 매니저는 BaseAgent를 상속하지 않는다 (파이프라인 에이전트가 아님)
- 매니저는 전략적 판단을 하지 않는다 (데이터 포맷팅만)
- 매니저는 CEO가 호출한다 (독립 실행 불가)
- 매니저 산출물은 `.system/manager/`에 저장된다 (사용자 산출물과 분리)

## 역할 정의

### 3계층 역할 경계

```
CEO (전략 계층)
  - 목적/목표 보유
  - 에이전트 호출 순서 판단
  - 품질 판정 (OK/NG)
  - 회장 보고 조건 판단
  - 매니저에게 알림 생성 지시

매니저 (소통 계층)
  - CEO가 전달한 구조화 데이터를 사용자 친화적 포맷으로 변환
  - 주간 실행 카드 생성
  - 진행 보고서 생성
  - 성과 입력 요청 메시지 생성
  - EventEmitter를 통한 알림 발송

하위 에이전트 (실무 계층)
  - 대상분석, 경쟁분석, 플랫폼추천, 컨셉기획
  - 도메인 분석 작업만 수행
  - 매니저와 직접 소통 없음
```

### 매니저가 하는 것 / 하지 않는 것

| 하는 것 | 하지 않는 것 |
|---------|------------|
| 주간 실행 카드 생성 | 어떤 에이전트를 호출할지 결정 |
| 진행 보고서 포맷팅 | 산출물 품질 판정 |
| 성과 입력 요청 메시지 생성 | 재분석 여부 판단 |
| 완료 요약 생성 | 회장 보고 조건 판단 |
| EventEmitter로 알림 발송 | 에이전트 컨텍스트 관리 |
| 알림 이력 파일 저장 | 전략 방향 수정 |

## 클래스 설계

### ManagerAgent

```python
class ManagerAgent:
    """사용자 알림 및 보고 서비스.

    BaseAgent를 상속하지 않는다 — 파이프라인 에이전트가 아니라
    CEO가 호출하는 서비스 객체이다.
    """

    def __init__(self, file_manager, llm_client_fn=None, event_emitter=None):
        """
        Args:
            file_manager: FileManager 인스턴스 (파일 저장용)
            llm_client_fn: LLM 호출 함수 (기본: core.llm_client.call_llm)
                          주간 카드 등 포맷팅에 LLM을 사용할 수 있음
            event_emitter: EventEmitter 인스턴스 (알림 발송용, 선택)
        """
        self._fm = file_manager
        self._llm = llm_client_fn
        self._emitter = event_emitter

    # --- 산출물 생성 메서드 ---

    def generate_weekly_card(self, calendar_output: str, week_num: int) -> str:
        """4주 캘린더에서 해당 주차의 실행 카드를 추출/생성한다.

        Args:
            calendar_output: 컨셉기획 에이전트가 생성한 4주 캘린더 전문
            week_num: 주차 번호 (1~4)

        Returns:
            마크다운 형식의 주간 실행 카드

        동작:
            1. calendar_output에서 해당 week_num 섹션 추출
            2. LLM으로 실행 가능한 체크리스트 형태로 변환
            3. 촬영 목록 + 준비물 + 업로드 체크리스트 포함
            4. 파일 저장: .system/manager/weekly_card_week{N}.md
            5. EventEmitter로 알림 발송
        """

    def generate_progress_report(
        self, agent_results: dict, validation_results: dict
    ) -> str:
        """현재 진행 상황을 사용자 친화적으로 요약한다.

        Args:
            agent_results: {에이전트명: 산출물} 딕셔너리
            validation_results: {에이전트명: {"pass": bool, "score": int, ...}}

        Returns:
            마크다운 형식의 진행 보고서

        동작:
            1. 완료/미완료 에이전트 상태 정리
            2. 각 에이전트 품질 점수 포함
            3. 핵심 발견 1~2줄 요약
            4. 다음 단계 안내
            5. 파일 저장: .system/manager/progress_report_{date}.md
            6. EventEmitter로 알림 발송
        """

    def request_performance_input(self, week_num: int) -> str:
        """사용자에게 주간 성과 데이터 입력을 요청하는 메시지를 생성한다.

        Args:
            week_num: 주차 번호 (1~4)

        Returns:
            입력 양식이 포함된 마크다운 메시지

        동작:
            1. 해당 주차에 맞는 성과 입력 양식 생성
            2. 파일 저장: .system/manager/performance_request_week{N}.md
            3. EventEmitter로 알림 발송
        """

    def generate_completion_summary(self, final_report: str) -> str:
        """전체 파이프라인 완료 후 사용자 친화적 요약을 생성한다.

        Args:
            final_report: CEO가 생성한 최종리포트.md 전문

        Returns:
            사용자가 바로 행동할 수 있는 요약 메시지

        동작:
            1. 최종리포트에서 핵심 결정사항 추출
            2. 즉시 실행 가능한 첫 3가지 액션 정리
            3. 산출물 위치 안내
            4. EventEmitter로 알림 발송
        """

    # --- 알림 발송 ---

    def notify(self, notification_type: str, content: str) -> None:
        """EventEmitter를 통해 알림을 발송한다.

        Args:
            notification_type: "weekly_card" | "progress" | "performance_request" | "completion"
            content: 알림 본문 (마크다운)

        동작:
            1. event_emitter.emit("manager_notification", {...}) 호출
            2. emitter가 없으면 (CLI 모드) 무시 (로그만 남김)
        """
```

### 설계 결정 근거

| 결정 | 이유 |
|------|------|
| BaseAgent 미상속 | 파이프라인에 포함되지 않음. context_key, output_prefix 등 불필요 |
| AGENT_CLASSES 미등록 | CEO의 에이전트 순회 루프에 포함되면 안 됨 |
| llm_client_fn 선택 주입 | 단순 포맷팅은 템플릿으로 충분. LLM은 주간 카드 등 복잡한 변환에만 사용 |
| event_emitter 선택 주입 | CLI 모드에서는 emitter 없이도 동작해야 함 |

## CEO-매니저 인터페이스

### 호출 시점

CEO는 아래 4개 시점에서 매니저를 호출한다.

| # | 시점 | 호출 메서드 | CEO가 전달하는 데이터 |
|---|------|-----------|---------------------|
| 1 | 파이프라인 완료 후 | `generate_completion_summary()` | 최종리포트.md 전문 |
| 2 | 재분석 완료 후 | `generate_progress_report()` | agent_results + validation_results |
| 3 | 주간 트리거 | `generate_weekly_card()` | 컨셉기획 캘린더 + 주차 번호 |
| 4 | 성과 입력 시점 | `request_performance_input()` | 주차 번호 |

### CEO 코드 내 통합 위치

```python
# ceo.py 내부 (의사 코드)

class CEO:
    def __init__(self, ..., manager=None):
        self._manager = manager  # ManagerAgent 인스턴스 (선택)

    def run(self, subject_info, goal):
        # ... 기존 파이프라인 실행 ...

        # 파이프라인 완료 후
        if self._manager:
            summary = self._manager.generate_completion_summary(final_report)
            # summary는 이미 EventEmitter로 발송됨

    def reanalyze(self, performance_data, feedback):
        # ... 재분석 실행 ...

        if self._manager:
            self._manager.generate_progress_report(
                agent_results=self._results,
                validation_results=self._validations,
            )
```

### 데이터 흐름

```
하위 에이전트 산출물
    ↓ (컨텍스트 누적)
CEO: 판단 + 취합
    ↓ (구조화된 데이터 전달)
매니저: 사용자 친화적 포맷 변환
    ↓
EventEmitter → SSE → 웹 대시보드
    ↓
파일 저장 → .system/manager/
```

## 산출물 정의

### 1. 주간 실행 카드 (Weekly Action Card)

**용도:** 크리에이터가 이번 주에 무엇을 해야 하는지 한눈에 보여주는 카드

**출력 형식:**
```markdown
## 이번 주 할 일 (Week 2)

### 촬영 목록
1. [쇼츠] 헤어 컬러링 비포/애프터 (월요일)
2. [쇼츠] 고객 시술 타임랩스 (수요일)
3. [릴스] 이번 주 트렌드 컬러 소개 (금요일)

### 준비물
- 스마트폰 거치대, 자연광 위치 확인
- 고객 촬영 동의서 (해당 시)

### 업로드 체크리스트
- [ ] 해시태그 10개 포함
- [ ] 캡션 작성 (산출물 참고)
- [ ] 최적 시간대 (오후 6-8시) 업로드
- [ ] 썸네일 확인

### 지난주 성과 메모
- (자동 기입 — 성과기록이 있을 경우)
```

**입력:** 컨셉기획 에이전트의 4주 캘린더 + 주차 번호
**저장:** `outputs/{name}/.system/manager/weekly_card_week{N}.md`

### 2. 진행 보고서 (Progress Report)

**용도:** 현재 분석/기획 진행 상황을 사용자에게 알림

**출력 형식:**
```markdown
## 진행 보고

### 완료된 분석
- 대상분석: 완료 (품질 85/100)
- 경쟁분석: 완료 (품질 78/100)
- 플랫폼추천: 완료 (품질 82/100)
- 컨셉기획: 진행중...

### 핵심 발견
- 헤어 컬러 전문성이 강점. 20대 여성 타겟과 잘 맞음
- 인스타그램 릴스가 1순위 플랫폼으로 추천됨

### 다음 단계
- 컨셉기획 완료 후 4주 콘텐츠 캘린더가 생성됩니다
- 완료 시 알림을 보내드립니다
```

**입력:** agent_results 딕셔너리 + validation_results 딕셔너리
**저장:** `outputs/{name}/.system/manager/progress_report_{date}.md`

### 3. 성과 입력 요청 (Performance Input Request)

**용도:** 주간 성과 데이터를 사용자에게 요청

**출력 형식:**
```markdown
## 성과 기록 요청 (Week 2)

아래 항목을 입력해주세요:

- 팔로워 수: ___
- 이번 주 총 조회수: ___
- 가장 반응 좋았던 콘텐츠: ___
- 댓글/DM 특이사항: ___
- 특이사항/피드백: ___
```

**입력:** 주차 번호
**저장:** `outputs/{name}/.system/manager/performance_request_week{N}.md`

### 4. 완료 요약 (Completion Summary)

**용도:** 파이프라인 완료 후 핵심 결과와 즉시 행동 안내

**출력 형식:**
```markdown
## 분석 완료 요약

### 핵심 결과
- 추천 플랫폼: 인스타그램 릴스 (1순위), 유튜브 쇼츠 (2순위)
- 선택 컨셉: "동네 미용실의 마법사" — 비포/애프터 중심
- 4주 캘린더: 주 3회 업로드 (월/수/금)

### 지금 바로 할 3가지
1. 인스타그램 계정 프로필 세팅 (산출물 04 참고)
2. 이번 주 촬영 목록 확인 (주간 실행 카드 참고)
3. 첫 콘텐츠 촬영 시작

### 산출물 위치
- 전체 분석: outputs/{name}/산출물/
- 주간 실행 카드: 매주 자동 생성
```

**입력:** 최종리포트.md 전문
**저장:** 별도 저장 없음 (EventEmitter로만 발송)

## EventEmitter 연동

### 새로운 이벤트 타입

```python
# 매니저 알림 이벤트
event_emitter.emit("manager_notification", {
    "notification_type": "weekly_card",  # | "progress" | "performance_request" | "completion"
    "content": "## 이번 주 할 일 (Week 2)\n...",
    "week_num": 2,  # weekly_card, performance_request에만 포함
})
```

### 이벤트 타입 정의

| notification_type | 발생 시점 | 포함 필드 |
|-------------------|----------|----------|
| `weekly_card` | 주간 트리거 | content, week_num |
| `progress` | 파이프라인 중간/재분석 후 | content |
| `performance_request` | 성과 입력 시점 | content, week_num |
| `completion` | 파이프라인 완료 | content |

### 링 버퍼 연동

EventEmitter의 기존 `_history` deque(maxlen=50)에 매니저 알림도 저장된다.
GET /status 응답에서 `manager_notification` 타입 이벤트를 필터링하면
최근 매니저 알림을 복원할 수 있다.

### SSE 구독

프론트엔드에서 매니저 알림만 필터링:
```javascript
// app.js (기존 SSE 핸들러 확장)
eventSource.onmessage = (e) => {
    const event = JSON.parse(e.data);
    if (event.type === "manager_notification") {
        renderManagerNotification(event);
    }
};
```

### 저장 규칙

| 파일 | 저장 시점 | 덮어쓰기 | 버전 관리 |
|------|-----------|----------|----------|
| weekly_card_week{N}.md | 주간 카드 생성 시 | 덮어쓰기 (같은 주차) | 불필요 (최신만 유효) |
| progress_report_{date}.md | 진행 보고 생성 시 | 날짜별 신규 생성 | 불필요 (날짜로 구분) |
| performance_request_week{N}.md | 성과 요청 생성 시 | 덮어쓰기 (같은 주차) | 불필요 |

### FileManager 확장

file_manager.py에 매니저 전용 저장 메서드 추가:

```python
def save_manager_output(self, filename: str, content: str) -> Path:
    """매니저 산출물을 .system/manager/에 저장한다."""
    manager_dir = self._base / ".system" / "manager"
    manager_dir.mkdir(parents=True, exist_ok=True)
    path = manager_dir / filename
    path.write_text(content, encoding="utf-8")
    return path
```

## 엣지케이스

### 1. 캘린더 데이터 없음
- **상황:** 컨셉기획 에이전트가 캘린더를 생성하지 않았거나 파싱 실패
- **대응:** `generate_weekly_card()`가 빈 캘린더 감지 시 기본 템플릿 반환
  ```
  ## 이번 주 할 일 (Week {N})
  캘린더가 아직 생성되지 않았습니다.
  먼저 전체 분석을 실행해주세요.
  ```

### 2. 부분 결과 (파이프라인 중간 실패)
- **상황:** 4개 에이전트 중 일부만 완료된 상태
- **대응:** `generate_progress_report()`는 완료된 에이전트만 표시, 미완료는 "대기중"으로 표시
- CEO가 회장 보고(조건 5)로 넘어갔으면 해당 사실도 진행 보고서에 포함

### 3. 사용자가 성과 입력을 무시
- **상황:** `request_performance_input()` 발송 후 응답 없음
- **대응:** 매니저가 재촉하지 않음 (매니저는 요청만, 독촉은 CEO 판단 영역)
- CEO가 재분석 시 성과 데이터 없으면 기존 산출물 기반으로만 재분석

### 4. EventEmitter 없음 (CLI 모드)
- **상황:** 웹 UI 없이 CLI로 실행
- **대응:** `notify()` 메서드에서 emitter가 None이면 파일 저장만 수행
- 로그에 "[매니저] 알림: {type}" 기록

### 5. 재분석 후 주간 카드 갱신
- **상황:** 2주차에 재분석을 실행하여 후반 캘린더가 변경됨
- **대응:** CEO가 재분석 완료 후 매니저에게 변경된 캘린더 전달
- 매니저가 해당 주차 카드를 덮어쓰기로 갱신 + 알림 발송

### 6. LLM 호출 실패
- **상황:** 매니저의 포맷팅용 LLM 호출이 실패
- **대응:** LLM 없이 템플릿 기반 기본 포맷으로 폴백
- 매니저 LLM 실패가 파이프라인 전체를 멈추면 안 됨 (비핵심 경로)
