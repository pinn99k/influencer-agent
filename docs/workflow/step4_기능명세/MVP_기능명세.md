## 개발 방식

- 전체 설계: 폭포수 (Step 0→9 순서 고정, 명세 확정 후 구현)
- 구현 순서: 나선형 (Spiral 단위 개발 → 검증 → 다음 Spiral)
- UI: Spiral 3에서 Claude 디자인 활용, UX 심플 최우선

---

## 전체 개발 플로우

```
Spiral 0-A
CEO 뼈대 구현
프롬프트 파일 출력
핑구 직접 테스트
      ↓
Spiral 0-B
Groq API 실제 연결
end-to-end 1회 실행
마크다운 저장
      ↓
Spiral 1
직무 에이전트 4개 순서대로 구현
하나씩 Mock 교체 후 단독 테스트
경쟁 분석: Serper API 연결
      ↓
Spiral 2
검증기 + max_retry=1
회장 보고 게이트
실제 인플루언서 1명 적용
      ↓
Spiral 3
웹 UI (Claude 디자인)
입력 폼 + 실행 트리거 + 결과 뷰어
포트폴리오 완성
```

## Spiral 0-A: CEO 뼈대 + 프롬프트 출력

**목적:** API 비용 0으로 프롬프트 품질 먼저 검증

**완료 조건:**
- [ ] CEO가 인풋 7개 필드를 읽고 목표를 해석한다
- [ ] 직무 에이전트 4개 호출 대신 프롬프트를 파일로 출력한다
- [ ] **[핑구 작업]** 출력된 프롬프트를 Claude/GPT에 직접 붙여넣어 결과 확인
- [ ] 결과 품질 핑구 승인 → Spiral 0-B 진입

**기능 목록:**

| # | 기능 | 입력 | 출력 | 우선순위 |
|---|------|------|------|----------|
| 0-A-1 | 인풋 수집 (CLI) | 7개 필드 직접 입력 또는 파일 로드 | 검증된 대상자 dict | HIGH |
| 0-A-2 | 목표 해석 | 대상자 dict | 실행 계획 텍스트 | HIGH |
| 0-A-3 | 프롬프트 생성 + 파일 출력 | 대상자 dict + 목표 | `outputs/{인플루언서명}/.system/prompts/` 에 에이전트별 프롬프트 .md 저장 (ADR 006) | HIGH |
| 0-A-4 | 목표 불명확 예외 처리 | 목표 텍스트 | 재확인 질문 3개 출력 | MED |

**엣지케이스:**
- 필드 누락 시: "정보 없음"으로 표시 후 계속 진행 (중단 금지)
- 목표가 너무 짧을 시 (10자 미만): 불명확 판정 → 재질문

## Spiral 0-B: CEO + Groq API 실제 연결

**목적:** end-to-end 구조 검증, 마크다운 저장 확인

**완료 조건:**
- [ ] CEO가 Groq API로 직무 에이전트 4개를 순서대로 호출한다
- [ ] 각 결과가 context 딕셔너리에 누적된다
- [ ] `outputs/report_{이름}_{날짜}.md` 저장 성공
- [ ] CEO 완료 보고 출력 (리포트 경로 + 주요 결정)

**기능 목록:**

| # | 기능 | 입력 | 출력 | 우선순위 |
|---|------|------|------|----------|
| 0-B-1 | Groq API 호출 래퍼 | 시스템 프롬프트 + 유저 메시지 | LLM 응답 텍스트 | HIGH |
| 0-B-2 | 컨텍스트 누적 관리 | 에이전트 결과 | context dict 업데이트 | HIGH |
| 0-B-3 | 마크다운 리포트 통합 + 저장 | context dict | `outputs/{인플루언서명}/산출물/최종리포트.md` | HIGH |
| 0-B-4 | CEO 완료 보고 출력 | context dict | 완료 보고 텍스트 (CLI) | HIGH |
| 0-B-5 | .env 기반 API 키 로드 | `.env` | GROQ_API_KEY | HIGH |

**엣지케이스:**
- API 호출 실패 시: 에러 메시지 출력 + 중단 (재시도는 Spiral 2에서)
- outputs 폴더 없을 시: 자동 생성

## Spiral 1: 직무 에이전트 4개 실제 구현

**목적:** 아웃풋 4개 실사용 품질 달성 (하나씩 구현·테스트)

**완료 조건:**
- [ ] 대상 분석 에이전트 단독 테스트 통과
- [ ] 경쟁 분석 에이전트 단독 테스트 통과 (Serper 연결)
- [ ] **[핑구 작업]** 컨셉 기획 전 레퍼런스 영상/이미지 제공 (선택 — 없으면 생략)
- [ ] 플랫폼 추천 에이전트 단독 테스트 통과
- [ ] 컨셉 기획 에이전트 단독 테스트 통과
- [ ] 핑구가 각 아웃풋 읽고 "크리에이터에게 줄 수 있다" 판단

**구현 순서 (이 순서 지킴):**

| 순서 | 에이전트 | 입력 | 출력 | 비고 |
|------|----------|------|------|------|
| 1 | 대상 분석 | 대상자 7개 필드 | 강점·약점·차별점 | Mock 없음, 바로 구현 |
| 2 | 경쟁 분석 | 대상자 + 대상_분석 + Serper 결과 | 유사포지션·공백·차별전략 | Serper API 연결 필요 |
| 3 | 플랫폼 추천 | 대상자 + 대상_분석 + 경쟁_분석 | 1순위·2순위·비추천 | - |
| 4 | 컨셉 기획 | 전체 context | 컨셉 3개 + 아이디어 5개 | 누적 컨텍스트 최대 활용 |

**경쟁 분석 Serper 플로우:**
```
"{직업} {특기} 유튜버" Serper 검색
  → 결과 파싱 (채널명, 구독자, 콘텐츠 유형)
  → Groq 입력으로 전달
  → 결과 없을 시 fallback: LLM 추론 + "[데이터 부족 — 추론 기반]" 명시
```

**엣지케이스:**
- 품질 NG 에이전트 발생 시: 핑구 판단 → 해당 에이전트만 gpt-4o/Claude로 교체 (ADR 작성)
- Serper 결과 없을 시: fallback 자동 적용

## Spiral 2: 품질 레이어

**목적:** 안정성 확보 + 실제 인플루언서 1명 적용

**완료 조건:**
- [ ] 검증기 구현 (필수 섹션 존재 / 대상자 이름 / 최소 길이 / 금지 패턴)
- [ ] max_retry=1 루프 구현
- [ ] 회장 보고 조건 10개 감지기 구현
- [ ] 실행 전 승인 게이트 구현 (조건 9)
- [ ] **[핑구 작업]** 최종 리포트 확인 후 실제 인플루언서에게 결과 전달 + 적용
- [ ] 실제 인플루언서 1명 적용 완료 + 전 과정 기록

**기능 목록:**

| # | 기능 | 입력 | 출력 | 우선순위 |
|---|------|------|------|----------|
| 2-1 | 출력 검증기 | 에이전트 결과 | 통과/실패 + 실패 이유 | HIGH |
| 2-2 | max_retry=1 루프 | 검증 실패 결과 | 재시도 프롬프트 (실패 규칙 포함) | HIGH |
| 2-3 | 회장 보고 조건 감지기 | 전체 텍스트 | 보고 조건 번호 + 브리핑 | HIGH |
| 2-4 | 실행 전 승인 게이트 | 리포트 완성 시점 | 회장 확인 요청 출력 + 대기 | HIGH |
| 2-5 | 정보 부족 플래그 처리 | 에이전트 판단 | 추가 질문 3개 이내 출력 | MED |

**엣지케이스:**
- 재시도도 실패 시: 회장 보고 (A/B 선택지 포함)
- 보고 조건 감지 시: 재시도 금지, 즉시 중단 + 보고

## Spiral 3: 웹 UI

**목적:** 포트폴리오 완성, UX 심플 최우선

**완료 조건:**
- [ ] 웹에서 인풋 7개 필드 + 목표 입력 가능
- [ ] 실행 시작 → 진행 상태 확인 가능
- [ ] 에이전트별 결과 + 최종 리포트 웹에서 조회 가능
- [ ] 마크다운 다운로드 가능
- [ ] 핑구가 "포트폴리오로 보여줄 수 있다" 판단

**UX 원칙:**
