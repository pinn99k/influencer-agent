# CEO 에이전트 — 전략 수립
updated: 2026-05-12

---

## 한 줄 정의

> 목적을 수신한 직후, CEO가 스스로 상황을 분석하고 실행 계획을 수립하는 단계

---

## 전략 수립 시점 + 판단 루프와의 경계

전략 수립은 **최초 실행 시 1회** 수립한다. 이후 CEO 자율 루프 중 성과 데이터 기반으로 재수립 가능. 판단 루프(02_판단루프.md)와 혼동하지 않는다.

| 구분 | 전략 수립 (이 파일) | 판단 루프 (02) |
|------|---------------------|----------------|
| 시점 | 최초 실행 시 1회 (+ 루프 중 재수립 가능) | 에이전트 결과 수신마다 반복 |
| 역할 | 어떻게 달성할지 계획 | 결과가 OK인지 판단 |
| 출력 | plan.md | context 누적 + plan.md 업데이트 |

근거: 두 단계를 구분하지 않으면 CEO가 실행 중에도 계속 전략을 바꾸거나,
반대로 초기 계획을 바꾸지 못하고 고착되는 문제가 생긴다.

---

## 목적 분석 — CEO가 스스로 던지는 질문

핑구의 목적을 받으면 그대로 실행하지 않는다.
아래 질문을 순서대로 스스로 답하며 분석한다.

```
1. 핑구가 원하는 최종 상태는 무엇인가?
   → 아웃풋 4개 생성? 특정 인플루언서에게 적용? 구체적으로 무엇인가

2. 달성하려면 어떤 정보가 필요한가?
   → 인풋 7개 필드 중 부족한 것은? 추가로 수집해야 할 것은?

3. 어떤 에이전트가, 어떤 순서로 필요한가?
   → 이번 목적에 맞는 최적 순서 판단 (기본 순서 참고, 고착 금지)

4. 리스크 / 불확실성은 무엇인가?
   → 정보 부족? 목표 모호? 에이전트 결과 신뢰도 낮을 가능성?
```

이 분석 결과가 plan.md의 "전략 요약" 섹션이 된다.

## 실행 계획 수립 기준

### 에이전트 순서 판단 원칙

순서는 매번 CEO가 판단한다. 하드코딩 금지.

근거: 목적·인풋 상태·이전 결과에 따라 최적 순서가 달라진다.
순서가 고정되면 CEO는 파이프라인이 된다.

### 기본 순서 (MVP — 변경 사유 없을 때 기준)

```
1. 대상 분석     ← 인풋 7개 필드 기반, 항상 첫 번째
2. 경쟁 분석     ← 대상 분석 결과 + Serper 검색
3. 플랫폼 추천   ← 대상 분석 + 경쟁 분석 결과
4. 컨셉 기획     ← 전체 컨텍스트 누적 후 마지막
```

### 순서 변경 판단 기준

| 상황 | 대응 |
|------|------|
| 대상 분석 결과가 빈약 | 컨텍스트 보완 질문 선행 후 재실행 |
| 경쟁 분석 Serper 결과 없음 | fallback 적용 후 계속 진행 (순서 유지) |
| 특정 에이전트 2회 실패 | 회장 보고 (조건 5번) — 순서 임의 변경 금지 |

## plan.md 출력 형식

전략 수립 완료 후 즉시 `plan.md`를 생성한다.
세션 내내 이 파일이 CEO의 실행 지도다.

```markdown
# plan.md
생성: {datetime}
대상자: {이름}

## 목적
{핑구가 제시한 목적 — 그대로 기록}

## 전략 요약
{CEO의 목적 분석 결과 — 달성 방법, 핵심 판단}

## 에이전트 실행 계획
- [ ] 1. 대상 분석     ← {판단 근거}
- [ ] 2. 경쟁 분석     ← {판단 근거}
- [ ] 3. 플랫폼 추천   ← {판단 근거}
- [ ] 4. 컨셉 기획     ← {판단 근거}

## 현재 상태
{진행 중인 에이전트 / 완료 항목 / 다음 단계}

## 열린 리스크
{불확실성 / 정보 부족 항목}
```

**plan.md vs state.md 역할 구분:**

| 파일 | 역할 | 업데이트 주체 |
|------|------|--------------|
| plan.md | 전략 + 실행 계획 (목적 중심) | 전략 수립 시 생성, 판단 루프마다 업데이트 |
| state.md | 현재 실행 상태 (진행 추적용) | 에이전트 호출·완료 시마다 업데이트 |

근거: 두 파일을 분리하지 않으면 "왜 이 순서로 했는가"(전략)와
"지금 어디까지 왔는가"(상태)가 뒤섞여 재시작 시 파악이 어렵다.

## 재질문 트리거

목적 분석 중 아래 조건 해당 시 실행 전 핑구에게 재확인한다.
상세 처리 방식 → [02_판단루프.md](02_판단루프.md) 재질문 트리거 섹션 참조

| 조건 | 판단 기준 |
|------|-----------|
| 목표가 너무 짧음 | 10자 미만 |
| 목표가 모호함 | "잘 되게 해줘" 수준 — 성공 기준 판단 불가 |
| 핵심 필드 다수 누락 | 인풋 7개 중 3개 이상 "정보 없음" |

재질문은 최대 3개, 선택형 우선.
재질문 없이 진행 가능하다고 판단되면 그대로 전략 수립 진행.
