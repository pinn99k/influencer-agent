# CEO 에이전트 — 판단 루프
updated: 2026-05-12

---

## 한 줄 정의

> CEO가 하위 에이전트 결과를 수신한 뒤, OK/NG를 판단하고 다음 행동을 결정하는 반복 구조

---

## 검증 레이어 구분

판단은 두 단계로 이루어진다. 순서를 바꾸지 않는다.

### 1단계 — 검증기 (자동, 형식 체크)

자동으로 실행되는 규칙 기반 검사. CEO가 직접 읽기 전에 먼저 통과해야 한다.

| 검사 항목 | 판정 기준 |
|-----------|-----------|
| 필수 섹션 존재 | 에이전트별 출력 형식에 정의된 섹션 모두 포함 여부 |
| [추정] 태그 누락 | 입력에 없는 정보를 단정했는가 |
| 직무 범위 초과 | 해당 에이전트 역할 밖의 내용 포함 여부 |
| 실제 계정명·인물명 언급 | 실명 노출 여부 |
| 최소 길이 | 섹션별 기준 길이 충족 여부 |

검증기 실패 → NG 처리로 직행 (CEO 주관 판단 없이)

### 2단계 — CEO 판단 (주관, 목적 부합 여부)

검증기 통과 후 CEO가 직접 판단한다. 기준은 형식이 아니라 목적이다.

- 이번 에이전트의 결과가 **프로젝트 목적에 부합하는가?**
- 다음 에이전트가 이 결과를 바탕으로 **의미 있는 작업을 할 수 있는가?**
- 핵심 내용이 **구체적인가, 아니면 공허한 일반론인가?**

기준을 통과하지 못하면 NG.

## 판단 루프 흐름

```
하위 에이전트 결과 수신
        │
        ▼
[1단계] 검증기 실행
        │
   ┌────┴────┐
  PASS      FAIL
   │          │
   ▼          └──────────────────────┐
[2단계] CEO 판단                     │
        │                           │
   ┌────┴────┐                      │
   OK        NG                     │
   │          │                     │
   ▼          ▼                     ▼
컨텍스트    회장 보고 조건 확인   NG 처리
누적 +      (10개 기준)
plan.md     │
업데이트    ├── 해당 → 즉시 중단 + 회장 브리핑
   │        │
   │        └── 미해당 → 재작업 지시 (1회)
   │                  │
   │                  ▼
   │           재결과 → 판단 루프 재진입
   │                  │
   │             ┌────┴────┐
   │             OK        NG (2회 실패)
   │             │          │
   │             ▼          ▼
   │         컨텍스트    회장 보고
   │         누적        (조건 5번)
   │
   ▼
다음 에이전트 판단
(CEO가 매번 재판단 — 순서 하드코딩 금지)
```

---

## OK 처리

1. 에이전트 결과를 context 딕셔너리에 누적
2. `plan.md` 업데이트 — 완료 항목 체크 + 현재 상태 기록
3. **다음 에이전트 또는 완료 판단** (CEO가 매번 재판단)
   - 남은 작업이 있는가?
   - 지금 결과로 다음 에이전트를 돌리기에 충분한 컨텍스트가 쌓였는가?
   - 순서 변경이 필요한 상황인가? (예: 경쟁 분석 전에 대상 분석이 빈약하면 보완 선요청)

근거: 에이전트 순서가 하드코딩되면 상황 변화에 대응할 수 없다.
다음 단계 판단 자체가 CEO 오케스트레이터의 핵심 역할이다.

---

## NG 처리 — 회장 보고 조건 우선

NG 판정 즉시, 재작업 전에 먼저 확인한다.

**회장 보고 조건 해당 여부 확인 순서:**
1. 이 실패가 10개 보고 조건 중 하나라도 해당하는가?
2. 해당 → **즉시 작업 중단** + 회장 브리핑 생성 (재작업 금지)
3. 미해당 → 재작업 지시로 진행

근거: 보고 조건 해당 항목을 재작업으로 해결하려 하면 회장이 알아야 할 사항을 우회하게 된다.
재작업은 형식·품질 문제에만 적용한다.

---

## 재작업 지시 형식

단순 "다시 해줘"는 금지다. 같은 결과가 반복될 뿐이다.

재작업 지시에는 반드시 아래 3가지가 포함되어야 한다:

```
[재작업 지시]
에이전트: {에이전트명}
문제: {무엇이 문제인지 — 구체적으로}
이유: {왜 문제인지 — 목적 기준으로}
수정 방향: {어떻게 고쳐야 하는지 — 구체적 지침}
재작업 횟수: {1회차}
```

재작업은 최대 1회. 이후 동일 실패 → 회장 보고 (조건 5번).

## plan.md 연동

판단 루프는 항상 `plan.md`와 연동된다.

| 시점 | plan.md 업데이트 내용 |
|------|----------------------|
| OK 판정 | 완료 항목 체크 + 다음 에이전트 기록 |
| NG → 재작업 지시 | 재작업 대상 + 이유 기록 |
| 회장 보고 발생 | 중단 지점 + 보고 사유 기록 |
| 2회 실패 | 실패 항목 + 회장 보고 기록 |

근거: 세션은 휘발성이다. 판단 결과가 파일에 남지 않으면 재시작 시 컨텍스트를 잃는다.
