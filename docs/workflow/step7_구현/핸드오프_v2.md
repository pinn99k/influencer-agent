## 0. 핵심 — 당신은 views.js + style.css 만 건드립니다

프론트가 MVC로 분리됐습니다. 6개 파일 중 당신(Claude Code, 디자인 담당)이
재작성할 곳은 **views.js 와 style.css 둘뿐**입니다.

| 파일 | 레이어 | 당신이 수정? | 설명 |
|------|--------|:---:|------|
| config.js | 상수 | 거의 X | FIELDS, AGENT_META, h() DOM헬퍼, renderMd |
| api.js | Controller(통신) | X | REST + SSE. 절대 안 건드림 |
| model.js | Model | X | State 데이터 + pub/sub. 절대 안 건드림 |
| actions.js | Controller(로직) | X | API호출+State갱신. 절대 안 건드림 |
| **views.js** | **View** | **O** | 렌더 함수. 여기를 새 디자인으로 재작성 |
| main.js | 부트 | 거의 X | render 구독 + 초기화 |

이유: View는 데이터를 State에서 읽기만, 동작은 Actions를 호출만 합니다.
그래서 화면(View)을 통째로 새로 그려도 로직(api/model/actions)이 안 깨집니다.

## 1. 반드시 지킬 5대 제약

### C-1. 화면 100vh 고정 + 내부 패널만 스크롤
- 최상위 컨테이너 height: 100vh (현재 버그: min-height라 페이지 전체가 늘어남)
- 페이지 전체 스크롤 금지. 스크롤은 각 패널(채팅/활동/산출물) 내부에서만.

### C-2. CEO 상시 채팅 (가장 중요)
- 사용자는 입장 때뿐 아니라 분석 완료 후에도 CEO와 계속 대화 가능해야 함.
- 대화 패널이 운영 화면(대시보드)에도 상시 노출. "대화 중"과 "완료"는 배타 아님.

### C-3. 양방향 네비게이션
- 운영(대시보드/산출물) / 재분석 / 새 대상자 자유 왕복.
- 현재 버그: 재분석 들어가면 결과 화면으로 못 돌아옴. 단방향 금지.

### C-4. View 순수성 (MVC 계약 — 어기면 분리가 깨짐)
- View는 State에서 읽기만 (State.persona, State.reports 등).
- 동작은 Actions.xxx() 호출만. fetch / EventSource / localStorage 직접 호출 금지.
- 재렌더가 필요하면 State.notify() 호출 (pub/sub 트리거, 데이터 변경 아님 — 허용).
- 즉 View 안에 비즈니스 로직 넣지 말 것. 새 동작이 필요하면 나에게 요청
  (actions.js에 내가 추가). 당신은 그 Actions를 호출만.

### C-5. 동작 훅 (data-action) 권장
- 버튼 등에 onclick으로 Actions.xxx() 직접 연결해도 되고(현재 방식),
  data-action 속성으로 표시해도 됨. 단 어느 Actions를 부르는지 명확히.

## 2. View가 쓸 수 있는 것 (계약 — 이게 인터페이스다)

### 2-1. State 읽기 필드 (model.js, 읽기 전용)
appointed, persona(11필드 dict), currentJobId, influencerName, ceoState,
activity[], approvals[], reports[], view, viewParam, activityTab, agentDetail,
entryTab('interview'|'reanalyze'), interview(대화세션), managerNotes[],
reanalyzeMode, subjects[], selectedSubject, feedbackContent, performanceContent

### 2-2. Actions 호출 (actions.js, 동작)
- Actions.startInterview()          새 대화 시작
- Actions.sendInterview(text)       대화 한 턴 전송
- Actions.confirmInterview()        제출(분석 시작). 미충족이면 CEO가 더 물음
- Actions.appoint(persona)          (구) 폼 제출 — 현재 미사용
- Actions.loadSubjects()            재분석용 기존 대상자 로드
- Actions.selectSubject(name)       재분석 대상 선택 + 피드백/성과 로드
- Actions.startReanalyze()          재분석 시작
- Actions.reset()                   초기화 (DEV)
- State.notify()                    강제 재렌더

### 2-3. State.interview 구조 (대화 세션)
{ id, messages:[{role:'ceo'|'user', text}], extracted:{11필드},
  pending(로딩중), turn(턴수), canSubmit(제출가능?), missing:[부족필드],
  summary, sufficient }
- 제출 버튼은 interview.canSubmit===true 일 때만 노출 (대화/제출 분리 규칙).

### 2-4. config.js 헬퍼 (그대로 사용)
- h(tag, props, ...children)  : DOM 빌더 (props.onclick 등 지원)
- renderMd(markdown)          : 마크다운 -> 안전한 HTML (산출물 표시용)
- avatar(agentKey), pill(text, kind), $(sel), clear(el)
- AGENT_META: {ceo, target, comp, platform, concept} 색/이름/이니셜

## 3. 백엔드 계약 (actions/api가 쓰는 것 — 당신은 참고만)

### REST (모두 /api)
인터뷰: POST /interview/start|reply|confirm
실행/상태: POST /start, GET /status/{job_id}, GET /stream/{job_id}(SSE),
  POST /decision/{job_id}, DELETE /job/{job_id}
산출물: GET /reports/{name}, GET /reports/{name}/{file}
재분석: GET /subjects, POST /reanalyze, GET|PUT /feedback/{name},
  GET|PUT /performance/{name}

### SSE 이벤트 15종 (actions.pushActivity가 수신 -> State 갱신)
job_started, plan_created, pipeline_started, agent_start, agent_done,
validation_fail, agent_questions_pending, pipeline_completed,
pipeline_completed_with_issues, briefing_pending(회장결재), decision_received,
finalize_started, manager_notification(주간카드/성과요청/완료/진행),
job_completed, job_failed
- 이 이벤트들은 actions.js가 받아 State.activity / State.managerNotes /
  State.approvals 에 넣음. View는 그 배열을 그리기만 하면 됨.

## 4. 현재 화면 함수 (views.js — 재작성 대상)

현재 14개 view 함수가 있음. 이름은 바꿔도 되나 render() 진입점 계약은 유지:
viewTopBar, viewActivityFeed, viewAgentDetailModal, viewGoalCard, viewMetrics,
viewDeliverables, viewApprovals, viewDashboard, viewBriefing, viewReport,
viewEntry, viewReanalyzeEntry, viewInterview, viewManagerPanel

### render() 진입 계약 (main.js가 호출 — 유지 필요)
- main.js의 render()가 매 State.notify()마다 호출됨.
- render()는 State.appointed 로 분기: false -> 입장(viewEntry), true -> 운영.
- 이 진입 구조(render가 State 보고 전체 다시 그림)는 유지. 내부 구성은 자유.

## 5. 화면 상태 인벤토리 (무엇이 언제)

| 상태 | 조건 | 보일 것 |
|------|------|---------|
| 입장(대화) | !appointed | CEO 채팅 + 수집정보. 제출버튼 canSubmit일 때만 |
| 분석 중 | appointed, ceoState=running | 실시간 활동(activity[]) + CEO 채팅 유지 |
| 완료 | ceoState=completed | 산출물(reports[]) + 매니저알림(managerNotes[]) + 재분석. 채팅 유지 |
| 결재 대기 | approvals[] 있음 | 결재 UI (A/B -> Actions 없음, 직접 API.decision 필요 -> 나에게 요청) |
| 재분석 | entryTab=reanalyze | subjects[] 선택 + 피드백/성과 입력 + 시작 |

주의(C-2): 분석이 끝나도 CEO 채팅은 살아있어야 함. interview 세션을
완료 후에도 재open 하려면 Actions에 메서드 추가 필요 -> 나에게 요청.
