## 1. 유저 플로우 (핑구 관점)

> CEO는 일회성 실행이 아니라 지속적 경영 루프다.
> 핑구는 최초 설정 후 개입 최소화. 보고 조건 해당 시만 결재.

### 1-1. 최초 설정 (1회만)

```
핑구                                   시스템
 │
 ├─ [입력] 인풋 7개 + 목표
 │    이름 / 직업 / 특기 / 성격
 │    타겟연령대 / SNS경험 / 목표
 │                                      ↓
 │                               CEO: 목표 해석
 │                               CEO: plan.md 생성 (전략)
 │                               CEO: state.md 초기화
 │                                      ↓
 │                          ┌── CEO 자율 경영 루프 시작 ──┐
 │                          │  (이후 핑구 개입 최소화)    │
 │                          └─────────────────────────────┘
```

### 1-2. CEO 자율 경영 루프 (반복)

```
                               CEO
                                │
                     현재 상태 파악
                     (구독자 수 / 최근 성과 / 완료 작업)
                                │
                     다음 액션 스스로 판단 (ADR 011)
                     예: 3개월 경과 → 경쟁 재분석
                     예: 컨셉 반응 저조 → 컨셉 업데이트
                                │
                    ┌── 에이전트 실행 ──┐
                    │  필요한 에이전트  │
                    │  (Serper 포함)    │
                    └──────────────────┘
                                │
                    결과 평가 / 피드백 / 승인
                                │
              ┌─────────────────┴──────────────────┐
         보고 조건 해당                        보고 조건 없음
              │                                    │
핑구◄── 회장 보고 (즉시 중단)            state.md / plan.md 업데이트
핑구 ──► A/B 결정                              루프 계속
              │
         CEO 재개
```

### 1-3. 핑구가 개입하는 시점

| 시점 | 트리거 | 핑구 행동 |
|------|--------|-----------|
| 최초 1회 | 목표 + 대상자 설정 | 7개 필드 + 목표 입력 |
| 에이전트 2회 실패 | 회장 보고 조건 5 (자동) | A/B 선택 |
| 실행 단계 진입 | 회장 보고 조건 9 (자동) | 승인 또는 중단 |
| 비용 발생 결정 | 회장 보고 조건 1 (CEO 판단) | 승인 |
| 목표 변경 필요 | 회장 보고 조건 3 (CEO 판단) | 방향 결정 |
| 외부 콘텐츠 실행 | 회장 보고 조건 9 | 촬영·편집·업로드 직접 수행 |

## 3. CEO 다음 에이전트 판단 — LLM 방식

결정: **CEO LLM이 매 결과 후 다음 에이전트를 재판단한다.**
순서 하드코딩 금지. 상황에 따라 CEO가 조정 가능.

### 판단 조건 (언제 재판단하는가)

- 에이전트 결과 OK 판정 직후 (매번)
- 회장 보고 결정 수신 후 재개 시

### 판단 범위 (CEO가 바꿀 수 있는 것)

| 허용 | 금지 |
|------|------|
| 에이전트 실행 순서 변경 | 4개 에이전트 중 건너뛰기 (MVP 전부 필수) |
| 대상 분석 보완 재요청 (결과 빈약 시) | 5번째 에이전트 임의 추가 |
| 컨텍스트 추가 보완 후 재실행 | 회장 보고 조건 우회 |

### 참고 도메인 (판단 시 CEO가 참조하는 것)

```
1. 현재 context 상태 (누적된 결과물)
2. 대상자 목표 (context["대상자"]["목표"])
3. 판단 유형별 경영 지식 선택 주입 (ADR 014)
   → 품질 판단: src/knowledge/경영/05_품질관리/, 08_의사결정/
   → 전략 수립: src/knowledge/경영/01_전략경영/, 06_운영관리/
   → 보고 판단: src/knowledge/경영/07_리스크관리/, 09_커뮤니케이션/
   → ROI 판단:  src/knowledge/경영/03_재무관리/, 04_조직인사관리/
   (전체 상시 로드 금지 — 토큰 한계 초과 방지)
4. 하위 에이전트 도메인 지식 (에이전트 init 시 1회 로드)
   → src/knowledge/부서/기획본부/{역할}/
5. plan.md (현재 전략 + 실행 계획)
```

### 판단 프롬프트 구조 (ceo 시스템 프롬프트에 포함)

```
현재 완료된 에이전트: {완료 목록}
현재 context 요약: {각 에이전트 결과 첫 줄}
대상자 목표: {목표}
남은 에이전트: {미실행 목록}

판단 질문:
- 지금 결과로 다음 에이전트를 돌리기에 충분한가?
- 순서를 바꿔야 할 이유가 있는가?
- 보완이 필요한 에이전트가 있는가?

출력 형식:
next_action: run | rework | complete | chairman_report
target_agent: {에이전트명}
reason: {이유 한 줄}
```

## 4. Serper 호출 위치 결정

결정: **경쟁 분석 에이전트가 직접 `core/serper_client.py`를 호출한다.**

```python
# 02_경쟁분석.py 내부
serper_result = serper_client.search(f"{직업} {특기} 유튜버")
# 결과를 프롬프트에 주입 후 LLM 호출
```

### 결정 근거

- 에이전트가 "외부 정보가 필요한지" 스스로 판단하는 구조
- CEO가 Serper 결과를 사전 주입하면 CEO가 검색 필요 여부를 대신 판단하게 됨 → 역할 침범
- 경쟁 분석 에이전트만 Serper 필요 (다른 에이전트 무관) → 의존성 최소화

### ⚠️ 미래 이슈 명시

**Serper 제거 가능 시점:** 사용 모델이 네이티브 웹 검색을 지원하게 되면 (예: GPT-4o with search, Gemini with grounding) `serper_client.search()` 호출 제거 → 모델의 네이티브 검색으로 대체 가능.

이 교체는 `02_경쟁분석.py` 단일 파일만 수정하면 됨. CEO나 다른 에이전트 무관.

교체 시 ADR 작성 필수.

## 6. context dict 표준

CEO/05_컨텍스트관리.md 확정 기준. 이 구조 재정의 금지.

```python
context = {
    "대상자": {
        "이름": str,
        "직업": str,
        "특기": str,
        "성격": str,
        "타겟연령대": str,
        "SNS경험": str,
        "목표": str,
    },
    "대상_분석":   str | None,
    "경쟁_분석":   str | None,
    "플랫폼_추천": str | None,
    "컨셉_기획":   str | None,
    "검증_결과": {
        "단계": str,
        "통과여부": bool,
        "실패규칙": str | None,
        "재시도여부": bool,
    },
    "보고_조건": {
        "조건번호": int,
        "근거": str,
        "조치": str,   # "즉시중단" | "대기중" | "처리완료"
    } | None,
}
```

에이전트별 전달 키 (필요한 것만):

| 에이전트 | 전달 키 |
|---------|---------|
| 대상 분석 | `대상자` |
| 경쟁 분석 | `대상자`, `대상_분석` |
| 플랫폼 추천 | `대상자`, `대상_분석`, `경쟁_분석` |
| 컨셉 기획 | `대상자`, `대상_분석`, `경쟁_분석`, `플랫폼_추천` |

```

   │
   ▼

   │
   └── NG (보고조건 해당) → 회장보고대기 ────────┘
   │
   ▼
완료
```

state.md 업데이트 타이밍: **에이전트 호출 직전 + 결과 수신 직후** 양쪽 다.
이유: 호출 직전 미기록 시 크래시 후 해당 에이전트 실행 여부 판단 불가.

## 8. 기술 결정 요약

| 항목 | 결정 | 근거 |
|------|------|------|
| 모듈 구조 | core/ 기능별 분리 | 독립 테스트 + Claude Code task 단위 분리 |
| AI 호출 | requests (OpenAI 호환) | provider 교체 = base_url만 변경 |
| AI 라우팅 | core/llm_client.py 단일 진입점 | 멀티 AI + 교체 대응 |
| Serper 호출 | 에이전트 직접 (core/serper_client.py 경유) | 역할 독립 + 미래 네이티브 검색 교체 가능 |
| 프롬프트 관리 | .md 파일 분리 (prompts/templates/) | 코드 수정 없이 프롬프트 튜닝 가능 |
| CEO 다음 판단 | LLM 판단 (조건·범위·도메인 정의) | 순서 유연성 + 상황 대응 |
| 웹 프레임워크 | FastAPI + SSE | 비동기 필수, 실시간 진행 표시 |
| 에러 처리 | requests.raise_for_status() + HTTPError | MVP 범위 충분, SDK 불필요 |
