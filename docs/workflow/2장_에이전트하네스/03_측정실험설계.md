# 2장 — 측정·실험 설계 (최우선 레이어)
# "데이터가 생기면 검증된다"는 틀렸다. 측정 가능한 실험 설계가 검증을 만든다.
date: 2026-06-04 (세션 31, cross-verify 핵심 보강)
status: 초안 — Step 4에서 스키마 확정

---

## 왜 이게 1순위인가

성과 데이터를 쌓아도, "무엇이 무슨 결과를 만들었는지" 귀속할 수 없으면 노이즈다.
자율 에이전트가 "컨셉 A 망함 -> 바꾸자"고 판단해도, A가 망한 게 컨셉 탓인지
업로드 시간 탓인지 모르면 그 판단은 근거가 없다. 따라서 측정 레이어가 자율 루프보다 먼저다.

---

## 1. KPI (측정 대상)

| KPI | 정의 | 수집 |
|-----|------|------|
| 팔로워 | 누적 팔로워 수 | 주간 수기 입력 |
| 콘텐츠 수 | 누적 게시물 | 주간 수기 입력 |
| 조회수/콘텐츠 | 게시물별 조회 | 게시물별 입력 |
| 참여(저장/좋아요/댓글) | 게시물별 | 게시물별 입력 |
| 전환(문의/협찬) | 누적 | 발생 시 입력 |

---

## 2. 콘텐츠 변수 태깅 (성과 귀속의 핵심)

게시물마다 아래 변수를 태깅한다. 이것이 있어야 "무엇이 효과였는지" 분석 가능.

| 변수 | 예시 값 |
|------|---------|
| 주제 | before/after, 컬러 변신, 시술 팁, 일상 |
| 형식 | 릴스, 쇼츠, 캐러셀, 스토리 |
| 길이 | 15초 / 30초 / 60초 |
| 업로드 시간대 | 오전 / 점심 / 저녁 / 밤 |
| 업로드 빈도(주) | 주 2 / 3 / 4 |

-> 게시물별 {KPI + 변수 태그}를 한 줄로 기록. 주차 말에 변수별 성과 비교.

---

## 3. 의사결정 로그 + provenance (포폴 진정성)

모든 전략 판단/조정을 기록한다. **누가 했는지(AI/사람)를 반드시 구분**한다.

```
[의사결정 로그 항목]
- 일시:
- 주체: AI(최소 자율 루프) | 사람(핑구 수동)
- 입력 근거: (어떤 성과/변수를 보고)
- 결정: (무엇을 바꿨나 — 컨셉/시간대/형식 등)
- 결과(후속 주차): (그 결정이 만든 변화)
```

이유: 실제로는 핑구가 수동 운영했는데 AI 성과처럼 보이면 포폴 설득력이 붕괴한다.
provenance가 있으면 "AI가 N건, 사람이 M건 판단" 을 정직하게 보여줄 수 있다.

---

## 4. 4주 실험 설계 (변수 1개씩 통제)

| 주차 | 실험(바꾸는 변수) | 측정(고정 비교) |
|------|------------------|----------------|
| 1 | 기준선(baseline) 콘텐츠 — 컨셉 3종 골고루 | 주제별 초기 반응 |
| 2 | 반응 좋은 주제 집중 + 업로드 시간대 A/B | 시간대 효과 |
| 3 | 형식 변경(릴스 vs 캐러셀 등) 1개 실험 | 형식 효과 |
| 4 | 최적 조합 집중 | 누적 성장률 |

원칙: 한 번에 변수 하나만 바꿔 귀속 가능하게. 여러 개 동시 변경 금지(노이즈).

---

## 5. 저장 위치 (초안 — Step 4 확정)

```
outputs/{이름}/.system/measure/
├── kpi.md            (주차별 KPI)
├── content_log.md    (게시물별 {KPI + 변수 태그})
└── decision_log.md   (의사결정 + provenance)
```

핵심 구현: core/measure.py(신규) — 기록/로드/주차별 변수 비교.
자율 루프의 read_performance / log_decision 도구가 이 레이어를 읽고 쓴다.
