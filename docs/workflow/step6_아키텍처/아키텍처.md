## 1. 계층 구조 (전체 그림)

```

│  Layer 0 — Entry              main.py                │  입력 수집만. 로직 없음.
├──────────────────────────────────────────────────────┤
│  Layer 1 — Orchestration      agents/ceo.py          │  판단·지휘·흐름 제어만.

├──────────────────────────────────────────────────────┤
│  Layer 2 — Domain Agents      agents/*_analyst.py 등 │  각자 하나의 직무만. CEO 지시에만 응답.
│                               agents/__init__.py     │  Agent Registry SSoT.
├──────────────────────────────────────────────────────┤
│  Layer 3 — Infrastructure     core/                  │  LLM·검색·파일·설정. 도메인 무관 유틸.
├──────────────────────────────────────────────────────┤

└──────────────────────────────────────────────────────┘
```

**의존 방향 (단방향):**
```
main.py → CEO → [agents, validators, core]
agents/ → core/
validators/ → (의존 없음 — 순수 규칙 모듈)
core/ → (의존 없음 — 순수 유틸 모듈)
```

레이어는 바로 아래만 참조. `core/`가 `agents/`를 역참조하거나 `validators/`가 `agents/`를 아는 구조 금지.

### 3-0b. `agents/__init__.py` — Agent Registry SSoT (NEW — 세션 16)

**책임:** 에이전트 등록 단일 진입점. 에이전트 추가 시 이 파일 + OutputValidator.RULES 2곳만 수정.

```python
from agents.subject_analyst import SubjectAnalystAgent
from agents.competition_analyst import CompetitionAnalystAgent
from agents.platform_recommender import PlatformRecommenderAgent
from agents.concept_planner import ConceptPlannerAgent

AGENT_CLASSES = [
    SubjectAnalystAgent,
    CompetitionAnalystAgent,
    PlatformRecommenderAgent,
    ConceptPlannerAgent,
]

def get_agent_order() -> list[str]:
    return [cls.display_name for cls in AGENT_CLASSES]

def get_context_keys() -> list[str]:
    return [cls.context_key for cls in AGENT_CLASSES]

def get_key_labels() -> list[tuple[str, str]]:
    return [(cls.context_key, cls.output_label) for cls in AGENT_CLASSES]
```

**알아야 할 점:**
- 이전: ceo.py에 AGENT_KEY_MAP, AGENT_ORDER 하드코딩 + main.py에도 context 키 하드코딩 → 7곳 수동 동기화 필요.
- 이후: AGENT_CLASSES 리스트 하나에서 모든 메타데이터 파생 (display_name, context_key, output_prefix, output_label).
- CEO, main.py, conftest.py 모두 이 레지스트리에서 import.

### 3-1. `main.py` — 함수형, 클래스 없음

**책임:** 인풋 7개 수집 → context 초기화 → CEO 호출. 로직 없음.

```python
from agents import get_context_keys  # Registry에서 키 파생

def collect_inputs() -> dict:
    """CLI에서 7개 필드 입력받아 대상자 dict 반환"""

def init_context(subject: dict) -> dict:
    """context dict 초기화 — 에이전트 키는 레지스트리에서 파생."""
    ctx = {"대상자": subject}
    for key in get_context_keys():      # Registry SSoT
        ctx[key] = None
    ctx["검증_결과"] = None
    ctx["보고_조건"] = None
    return ctx

def main():
    subject = collect_inputs()
    context = init_context(subject)
    fm = FileManager(subject["이름"])
    ceo = CEO(fm)
    ceo.run(context)
```

**알아야 할 점:**
- context 키 목록이 하드코딩이 아닌 `get_context_keys()`에서 동적 파생.
- 에이전트 추가 시 main.py 수정 불필요 — Registry에 등록만 하면 자동 반영.
- `FileManager`와 `CEO`를 여기서 조립 (DI 진입점).

### 3-2. `core/llm_client.py` — 모듈 함수, 클래스 없음

**책임:** provider별 LLM API 호출 단일 진입점. provider·model 상수의 SSoT.

```python

DEFAULT_PROVIDER = "groq"
DEFAULT_MODEL    = "llama-3.3-70b-versatile"   # 세션 11: llama3-70b-8192 deprecated

PROVIDER_CONFIG = {
    "groq":      {"base_url": "https://api.groq.com/openai/v1",  "api_key_env": "GROQ_API_KEY"},
    "openai":    {"base_url": "https://api.openai.com/v1",       "api_key_env": "OPENAI_API_KEY"},
    "anthropic": {"base_url": "https://api.anthropic.com/v1",    "api_key_env": "ANTHROPIC_API_KEY"},
}

def call_llm(provider: str, model: str, system: str, user: str) -> str:
    """
    OpenAI 호환 엔드포인트로 통일 호출.
    429 Too Many Requests 시 Retry-After 헤더 기반 재시도 (최대 LLM_MAX_RETRIES회).
```

- `requests` 직접 사용 (SDK 없음 — ADR 009).

### 3-5. `agents/base_agent.py` — 추상 클래스 (ABC)

**책임:** 4개 하위 에이전트 공통 계약 정의. 중복 제거.

```python
from abc import ABC, abstractmethod
from core.llm_client import call_llm, DEFAULT_PROVIDER, DEFAULT_MODEL
from core.config import PROMPTS_DIR, KNOWLEDGE_DIR

class BaseAgent(ABC):
    display_name:  str = ""    # 한글 표시명 (state.md 출력용)
    prompt_file:   str = ""    # prompts/ 하위 경로 (확장자 제외)
    knowledge_dir: str = ""    # knowledge/ 하위 경로 — "" 이면 지식 없음
    context_key:   str = ""    # context dict 저장 키 — "대상_분석" 등 (NEW — 세션 14)
    output_prefix: str = "00"  # 산출물 파일 번호 — "01", "02" 등 (NEW — 세션 16)
    output_label:  str = ""    # 최종 리포트 표시명 — "대상 분석" 등 (NEW — 세션 16)

    def __init__(self):
        self.system_prompt = self._load_template() + self._load_knowledge()

    def run(self, context: dict) -> str:
        user_prompt = self.build_prompt(context)
        return call_llm(DEFAULT_PROVIDER, DEFAULT_MODEL, self.system_prompt, user_prompt)

    @abstractmethod
    def build_prompt(self, context: dict) -> str: ...

    @abstractmethod
    def get_context_keys(self) -> list[str]: ...

    def _load_template(self) -> str:
        path = PROMPTS_DIR / f"{self.prompt_file}.md"
        if not path.exists():
            raise FileNotFoundError(f"프롬프트 파일 없음: {path}")
        return path.read_text(encoding="utf-8")

    def _load_knowledge(self) -> str: ...
```

**변경 사항 (세션 14+16):**
- `context_key` 클래스 변수 추가 → ceo.py의 AGENT_KEY_MAP 제거 가능.
- `output_prefix`, `output_label` 클래스 변수 추가 → 레지스트리 패턴 지원.
- 경로 상수를 `config.py`에서 import (PROMPTS_DIR, KNOWLEDGE_DIR).

- `CompetitionAnalystAgent`는 더 이상 `run()`을 오버라이드하지 않음. Serper 호출을 `build_prompt()` 안에서 처리 (LSP fix — 세션 14).
- `build_prompt()` 서명이 모든 에이전트에서 `build_prompt(self, context: dict) -> str`로 통일 (LSP 준수).
- `context_key`, `output_prefix`, `output_label` 클래스 변수 추가.

### 3-7. `validators/output_validator.py` — 클래스 + dataclass

**책임:** 1단계 자동 검증 (형식·구조 체크). CEO의 2단계 LLM 판단 이전에 실행.

```python
@dataclass
class ValidationResult:
    passed: bool
    agent_name: str
    failed_rules: list[str] = field(default_factory=list)
    retry_required: bool = False

class OutputValidator:
    RULES = { ... }  # 에이전트별 FORMAT 규칙 SSoT

    @classmethod
    def validate(cls, agent_name: str, output: str, subject_name: str) -> ValidationResult:
        """
        순서대로 체크:
        1. 필수 섹션 헤더 존재
        2. 대상자 이름 포함
        3. 필수 키워드 포함
        4. 개수 체크 (섹션 범위 내 — _extract_section 사용)
        5. 최소 길이
        6. CJK 한자 혼입 감지
        """
```

**변경 사항 (세션 14+16):**
- 미등록 에이전트(rules 없음) → early return (pass). 알려진 에이전트만 검증.
- `count_check`가 전체 출력이 아닌 해당 섹션 범위 내에서만 카운트 (`_extract_section()` 추가 — false positive 제거).
- CJK 한자(U+4E00~U+9FFF) 혼입 감지 규칙 추가 (Groq llama3-70b 출력 오염 방지).

### 3-8. `agents/ceo.py` — 클래스 (SRP 분해 완료)

**책임:** 전략 수립 + 에이전트 루프 흐름 제어만. 산출물 직접 생성 금지.

**세션 14 SRP 분해:** CEO가 원래 담당하던 상태 관리·문서 생성·프롬프트 로딩을 3개 전담 클래스로 분리.

```python
from agents import AGENT_CLASSES, get_agent_order, get_key_labels  # Registry
from core.config import MAX_RETRY
from core.state_manager import CEOStateManager
from core.report_builder import ReportBuilder
from core.prompt_loader import PromptLoader

AGENT_ORDER = get_agent_order()  # Registry에서 파생

class CEO:
    def __init__(self, file_manager: FileManager, dry_run: bool = False):
        self.fm = file_manager
        self.dry_run = dry_run
        self._state_mgr = CEOStateManager(file_manager)  # 상태 전담
        self._reports = ReportBuilder(file_manager)       # 문서 전담
        self._prompts = PromptLoader()                     # 로딩 전담
        self.agents = {cls.display_name: cls() for cls in AGENT_CLASSES}  # Registry

    def run(self, context: dict) -> None:
        self._interpret_goal(context)    # Phase 1: 전략 수립
        self._agent_loop(context)        # Phase 2: 에이전트 루프
        if not self.dry_run:
            self._finalize(context)      # Phase 3: 완료 처리
```

**변경 사항 (세션 14+16):**
- **SRP 분해:** `_update_state()` → `CEOStateManager`, `_build_final_report()` → `ReportBuilder`, `_load_prompt()` → `PromptLoader`.
- **Registry 패턴:** `AGENT_KEY_MAP` 하드코딩 제거. `AGENT_CLASSES`에서 모든 메타데이터 파생.
- `save_output(agent.output_prefix, agent_name, result)` — prefix를 에이전트 클래스에서 직접 참조.
- `_finalize()`에 `get_key_labels()` 전달 → 최종 리포트가 레지스트리 기반.

### 3-9. `core/state_manager.py` — CEOStateManager (NEW — 세션 14)

**책임:** CEO 상태 초기화·갱신·저장 전담. SRP: 상태 관리만.

### 9-2. AgentOutput (Spiral 5-A 구현 완료)

에이전트의 산출물 외에 질문·의견·신뢰도를 함께 전달하는 구조화 반환 타입.

```python
@dataclass
class AgentOutput:
    content: str                  # 마크다운 산출물 (OutputValidator 검증 대상)
    questions: list = []          # CEO에게 보내는 질문
    comments: list = []           # 에이전트 의견/제안
    confidence: float = 1.0       # 0.0~1.0
    metadata: dict = {}

    @classmethod
    def from_raw(cls, text) -> "AgentOutput":
        # ---AGENT_OUTPUT--- ... ---END_OUTPUT--- JSON 블록 파싱.
        # 블록 없거나 JSON 깨지면 전체 텍스트를 content로 (관대한 파싱, 예외 없음).
```

핵심 하위 호환 규칙:
- `BaseAgent.run()`은 여전히 `str` 반환 — 기존 코드/테스트 불변.
- `AgentOutput.from_raw()` 파싱은 `PlanningDepartment._run_agent()` 한 곳에서만.
- `OutputValidator`는 `.content`만 수신 — 검증 로직 불변, `RULES` 헤더 불변.
- 구분자 없는 평문 출력 → 전체가 content, questions/comments 빈 값.

### 9-3. AgentContext (Spiral 5-A 구현 완료)

에이전트별 컨텍스트 접근 범위를 `AGENT_SCOPES`로 강제한다.

```python

    "CEO":       {"read": ["*"],                   "knowledge": ["경영/"], "outputs": ["*"]},

    "경쟁분석":   {"read": ["대상자", "ceo_summary"], "knowledge": ["부서/기획본부/경쟁분석/"], "outputs": ["01_대상분석.md"], "tools": ["serper"]},

    "컨셉기획":   {"read": ["대상자", "ceo_summary"], "knowledge": ["부서/기획본부/컨셉기획/"], "outputs": ["01_대상분석.md", "02_경쟁분석.md", "03_플랫폼추천.md"]},
}

def build_context(agent_name, full_context, file_manager=None) -> dict:
    # read 키만 필터링하여 반환. "*" 또는 미등록 에이전트는 전체 반환(폴백).
```

5-A 범위 한정: `read` 키 필터링만 구현. `outputs` 파일 읽기와 `tools` 게이팅은

### 9-4. 데이터 흐름 변경 (PlanningDepartment._run_agent)

```
agent.run(context) → str (raw, ---AGENT_OUTPUT--- 블록 포함 가능)
  → fm.save_raw(raw)                         # 원본 보존 (디버그용)
  → AgentOutput.from_raw(raw)                # 단일 파싱 지점
  → content = output.content                 # 블록 제거된 깔끔한 산출물
  → OutputValidator.validate(content, ...)   # content만 검증
  → _judge_quality(content, ...)             # content만 품질 판정
  → dept_context[key] = content / save_output(content)
  → DepartmentResult.agent_questions/comments/confidence 저장
```

CEO는 `_agent_loop()`에서 `dept_result`의 questions/comments/confidence를
context에 저장한다 (`에이전트_질문`, `에이전트_의견`, `에이전트_신뢰도`).
실제 질문 처리(`_handle_agent_questions` 3단계 판단)는 Spiral 5-B.
