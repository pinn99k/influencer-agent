## 정직한 현 위치 — "V2 완료"가 아니다

이전 표현은 과장이었다. 정확히는:
- **완료(검증됨)**: 하위 워커 에이전트 4개 (대상/경쟁/플랫폼/컨셉)
  - gpt-4o 실 LLM E2E 품질 90/85/90/85, OutputValidator PASS, 외래문자 0
- **구현됨(미검증)**: 그 위의 감독층 — CEO, 매니저
  - CEO의 LLM 판단 메서드 6개가 테스트에서 전부 mock 처리 (배선만 검증)
  - 기존 품질 측정 스크립트는 CEO.run()을 우회하고 워커만 직접 실행
  - 매니저 산출물은 템플릿 조립 단위테스트만 있음 (유용성 미평가)

즉 이 프로젝트의 핵심(자율 경영 CEO + 매니저)은 **품질이 측정된 적 없다.**
워커 위에 더 쌓기(확장) 전에 감독층을 먼저 검증해야 한다.

### 개발자 관점 명시
이 시스템을 만드는 주체는 Claude(개발자)다. CEO/매니저/워커는 모두
**개발·검증 대상**이다. 회장 보고·자율 루프·매니저 알림은 개발자가 따르는
규칙이 아니라, 만들고 검증하는 시스템의 명세다.

## 트랙 0 (선결) — 감독층 검증: CEO + 매니저

확장이 아니라 "이미 만든 것의 미검증 부분 메우기". 모든 확장보다 앞선다.

### 검증 대상 (코드 근거)
- ceo.py _interpret_goal (L159) — 목표 -> 실행계획 생성
- ceo.py _decide_next (L388) — 다음 에이전트 판단 (LLM 복원됨)
- ceo.py _handle_agent_questions (L205) — 질문 3단계 분류(STRATEGIC/TACTICAL/DATA)
- ceo.py _build_ceo_summary (L258) — 워커용 전략 요약
- ceo.py _judge_quality (L446) — 산출물 품질 채점
- ceo.py _decide_rerun (L640) — 재분석 시 재실행 에이전트 판단
- manager.py generate_weekly_card / progress_report / completion_summary
  (주의: 매니저는 LLM 미사용. 정규식+템플릿 조립 — manager.py L18 _llm 주입은 있으나
   메서드는 결정론적. 따라서 '품질 채점' 대상 아님. '추출·포맷 정확성'으로 검증)

### 핵심 발견 — 조용한 폴백 (self-review, 코드 근거)
CEO의 LLM 판단 6개 메서드가 전부 try/except로 감싸여 실패 시 조용히 폴백한다:
- _decide_next 실패 -> 순차 폴백 (ceo.py L443)
- _decide_rerun 실패 -> 전체 재실행 (ceo.py L677)
- _judge_quality 실패 -> passed:True 무조건 통과 (ceo.py 약 L477)
- _handle_agent_questions 실패 -> 전체 에스컬레이션 (ceo.py L247)
- _build_ceo_summary 실패 -> 빈 요약 (ceo.py L274)

함의: LLM이 쓰레기를 뱉어도 CEO는 에러 없이 완주한다. 즉 "CEO.run()이 통과했다"는
사실은 아무것도 보장하지 않는다 (폴백으로 굴러간 건지 실제 판단인지 구분 불가).
따라서 트랙 0의 핵심 질문은 "점수가 80이냐"가 아니라
"CEO 판단이 폴백이 아닌 실제 발화이며, 그 판단이 타당한가"다.

### 단계 (정정)
| 단계 | 내용 | 완료 기준 |
|------|------|----------|
| V-1 | e2e_ceo_quality.py 신규 — CEO.run() 전체 실 LLM 완주(워커 우회 금지) + 각 판단의 폴백 발화 여부 계측 | 6개 판단 중 폴백 0건 (전부 실제 LLM 발화) |
| V-2 | 폴백 0 확인 후 발화된 판단의 타당성 채점 — 계획 순서·재실행 선택·질문분류가 맞는가 | 타당성 PASS 또는 오판 사례 기록 |
| V-3 | 매니저 결정론적 정확성 — 캘린더 Week 추출·검증결과 반영이 입력과 일치하는가 (LLM 아님) | 추출/포맷 정확성 단위테스트 PASS |
| V-4 | 회장 보고 조건 발화 검증 — 조건 9(실행진입)는 발화됨(ceo.py L538). 나머지 9개 조건의 미구현 갭을 명시 기록 | 갭 목록 작성 (구현 범위 결정은 회장) |

### 신규/변경 파일
- 신규: src/tests/e2e_ceo_quality.py (폴백 계측 포함), src/tests/test_ceo_triggers.py
- 변경 가능성: ceo.py (폴백 발생 시 계측 훅 추가 — 판단 자체는 보존), prompts/ceo/*.md
- 주의: 폴백 try/except는 운영 안전장치이므로 제거하지 않는다. 계측만 추가한다.

### 리스크
- CEO.run() 전체 = LLM 호출 다수(워커4 + CEO판단 5~6 + 매니저). 비용/시간 큼 -> 1회 측정 기준
- 폴백이 많이 발화될 수 있음 -> 그게 바로 미검증이던 진짜 위험. 측정의 핵심 목적
- 조건 발화 갭은 버그가 아니라 미구현 -> 어디까지 구현할지는 회장 결정 사항

## 트랙 C — 컨셉기획 심화 (신규 에이전트 아님, 흡수)

### 정정 — 어제 로드맵의 오류
어제는 "스크립트/해시태그/캡션 에이전트 신설"이라 했으나 **코드 확인 결과
컨셉기획이 이미 생성**한다 (concept_planner.py L51-79):
- 4주 캘린더, 촬영 가이드, 해시태그 10개, 캡션 템플릿, 영상 아이디어 5개

따라서 C는 독립 트랙이 아니라 컨셉기획 강화로 흡수:
1. 스크립트(30초 대본) 단계 추가 — 유일한 진짜 갭
2. multi-step 재활성화 — concept_planner.py L99~100에서 get_steps()가 []
   반환(비활성). 사유 주석은 "Groq RPM 초과"인데 현재 provider는 gpt-4o라
   **stale**. 5단계 메서드(_prompt_concepts/ideas/calendar/guide/assemble)는
   이미 구현돼 있고 단일호출에 욱여넣는 중 -> 숨은 품질 누수

### 워크플로우 / 파일
- Loop: Loop-003 (신규). Step 4(스크립트 단계 명세) -> 7(Spiral 7-A)
- 변경: agents/concept_planner.py (get_steps 5~6단계 복원, 스크립트 단계, 주석 정정),
  validators/output_validator.py (컨셉기획 RULES에 스크립트 섹션),
  tests/ (단일호출 가정 어셋션 갱신)

### 완료 기준
- multi-step on/off E2E 품질 비교 -> 오르면 유지
- 컨셉별 30초 대본 생성, 외래문자 0, 품질 80+ 유지

## 권장 순서

```
트랙 0 (CEO/매니저 검증)  <- 선결, 확장 전 필수
   |
트랙 B (Cross-review)     <- 워커 품질 천장
   |
트랙 C (컨셉기획 심화)    <- 실행 산출물 강화 (가벼움)
   |
트랙 A (마케팅 본부)      <- 가장 무거운 가로 확장
```

근거:
1. 0번 먼저 — 감독층이 미검증인데 그 위에 부서를 늘리면 "검증 안 된 지휘관에게
   부대를 더 주는" 격. 측정이 선결.
2. B — 워커 단위 품질을 먼저 올린다 (부서 늘려도 단위 질이 그대로면 양만 증가).
3. C — 가볍고(기존 에이전트 강화) 실적용 효과 직접적.
4. A — 가장 무겁고 투기적. 0/B/C로 단위·감독 품질 검증 후가 안전.

## 실적용과의 병행

트랙 0/B/C는 실제 미용사 없이 가상 E2E로 검증 가능. 미용사 섭외(사람 병목)를
백그라운드로 두고 0 -> B -> C 진행. A는 실적용 1차 데이터 후 착수 권장.
