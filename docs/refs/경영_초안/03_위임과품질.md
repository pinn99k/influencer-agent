# 경영 도메인 — 03 위임과 품질관리
# 목적: CEO 에이전트가 하위 에이전트에게 위임하고 결과를 판단할 때 참조
updated: 2026-05-14

---

## 위임 원칙

CEO는 실무를 직접 하지 않는다.
"무엇을 할지 (목적)"와 "결과 기준 (완료 조건)"만 전달한다.
"어떻게 할지 (방법)"는 하위 에이전트의 영역이다.

---

## 위임 체크리스트

하위 에이전트를 호출하기 전 아래 4가지가 준비됐는지 확인한다.

| 항목 | 확인 |
|------|------|
| 이 에이전트가 받아야 할 컨텍스트가 context에 존재하는가? | ✓ |
| 이전 에이전트 결과가 검증(pass)됐는가? | ✓ |
| 재작업이 발생한 경우 재작업 지시 내용이 rework.md에 저장됐는가? | ✓ |
| state.md가 호출 직전 상태로 업데이트됐는가? | ✓ |

---

## 품질 판단 기준 (2단계)

### 1단계 — 자동 검증 (OutputValidator)
형식·구조 체크. 코드가 처리. CEO가 개입하지 않는다.

### 2단계 — CEO LLM 품질 판단
1단계를 통과한 결과에 대해 CEO가 아래 3가지 기준으로 판단한다.

**기준 1: 다음 에이전트가 이 결과로 작업할 수 있는가?**
- 다음 에이전트가 필요한 정보가 모두 포함됐는가?
- 모호하거나 해석이 필요한 내용이 있는가?

**기준 2: 프로젝트 목적에 부합하는가?**
- 초기 목표(plan.md)와 방향이 일치하는가?
- 대상자의 특성이 실제로 반영됐는가?

**기준 3: Generic 여부**
- 대상자 이름을 지워도 다른 사람에게 그대로 쓸 수 있는가?
- → Yes면 Generic → NG

## 재작업 지시 작성법

재작업 지시는 "다시 해줘"가 아니다.
아래 3가지를 반드시 포함한다.

```
## 재작업 지시
문제: {구체적으로 무엇이 부족한가}
이유: {왜 이것이 문제인가 — 다음 에이전트에 미치는 영향}
수정 방향:
  1. {구체적 수정 지시}
  2. {구체적 수정 지시}
  3. {구체적 수정 지시}
```

**나쁜 예:** "분석이 부족합니다. 다시 해주세요."
**좋은 예:** "강점 3개 중 2개가 대상자 정보(직업: 미용사)와 무관한 일반적 내용입니다. 직업과 특기에서 직접 도출된 강점으로 교체해주세요."

| 상황 | 처리 |
|------|------|
| 1단계 자동검증 실패 | 재작업 지시 (1회) |
| 2단계 CEO 판단 NG | 재작업 지시 (1회) |
