# 목적: CEO 에이전트가 목표를 실행 가능한 계획으로 분해할 때 참조
updated: 2026-05-14

---

## 핵심 원칙

계획은 "무엇을 할 것인가"가 아니라 "어떤 순서로, 무엇을 먼저 검증할 것인가"다.
가장 불확실한 것을 가장 먼저 검증한다.

---

## 목표 분해 절차

### 1단계 — 목표 명확화
목표를 받으면 즉시 아래 3가지를 확인한다.

- **완료 조건**: 언제 끝난 것인가? (측정 가능한 기준)
- **범위 밖**: 이번에 하지 않는 것은 무엇인가?
- **전제 조건**: 이 목표가 성립하려면 무엇이 참이어야 하는가?

전제 조건이 흔들리면 계획 전체가 무너진다. 먼저 확인하라.

### 2단계 — 작업 분해 (WBS)
목표를 독립 실행 가능한 단위로 쪼갠다.

기준:
- 하나의 작업 = 하나의 담당자 (에이전트)
- 하나의 작업 = 검증 가능한 산출물 1개
- 다음 작업이 이전 산출물에 의존하는 경우 → 순서 고정
- 의존성 없으면 → 병렬 가능 (MVP에서는 순서 고정)

### 3단계 — 우선순위 결정
분해된 작업을 아래 기준으로 정렬한다.

| 기준 | 설명 |
|------|------|
| 의존성 | 다른 작업의 인풋이 되는 것 먼저 |
| 불확실성 | 결과를 모르는 것 먼저 (검증 우선) |
| 위험도 | 실패 시 전체에 영향 큰 것 먼저 |
| 비용 | 저비용 검증 먼저, 고비용은 나중 |

## 계획 문서 형식

계획 수립 후 아래 형식으로 plan.md에 기록한다.

```
## 목표
{한 문장}

## 완료 조건
- {측정 가능한 기준}

## 전제 조건
- {참이어야 하는 것}

## 실행 순서
1. {작업명} — 담당: {에이전트} — 산출물: {파일}
2. ...

## 범위 밖
- {하지 않는 것}
```

---

## 계획 수정 조건

계획은 고정이 아니다. 아래 상황에서 즉시 수정한다.

- 전제 조건이 틀렸음이 확인됨
- 에이전트 결과가 다음 작업의 인풋으로 쓸 수 없는 품질
- 목표 자체가 변경됨 (→ 회장 보고 조건 3)

계획 수정 시 이전 버전 snapshots/에 보관 후 덮어쓴다.
