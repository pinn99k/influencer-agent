# 기능 의존관계

## 정의/배경

- **"A 기능이 없으면 B 기능을 만들 수는 있어도 확인할 수 없다"**는 관계
- 클래스 의존관계(코드 구조)와는 다른 개념 — 개발 순서를 결정하는 도구
- 핵심 원칙: **"확인할 수 없는 코드는 만들지 않는다"**

## 핵심

### 클래스 의존관계 vs 기능 의존관계

```
클래스 의존관계 (코드 구조):
  "이 클래스가 어떤 클래스를 참조하는가?"
  GameHub → RoomService → GameRoom
  → 코드를 어떻게 짤지 판단하는 도구

기능 의존관계 (개발 순서):
  "이 기능을 확인하려면 뭐가 먼저 돼야 하는가?"
  방 생성 → 방 참가 (방이 있어야 참가를 테스트할 수 있음)
  → 뭘 먼저 만들지 판단하는 도구
```

||클래스 의존관계|기능 의존관계|
|---|---|---|
|관점|코드 구조|개발 순서|
|질문|"이 클래스가 어떤 클래스를 쓰는가?"|"이 기능을 확인하려면 뭐가 먼저 돼야 하는가?"|
|결과물|클래스 다이어그램, 코드 설계|개발 로드맵, 스프린트 계획|
|위반 시|컴파일 에러, 순환 참조|테스트 불가, "아마 맞겠지" 코드|

둘 다 맞는 개념이고, 다른 레벨의 이야기.

---

### 기능 의존을 찾는 법

모든 기능에 이 질문을 던진다:

> **"이 기능이 동작하는 걸 눈으로 확인하려면, 뭐가 먼저 돼 있어야 하지?"**

```
예시: 웹 게임 (Capital Wars)

맵 로드 → "JSON에서 노드 12개 읽히나?" → 아무것도 필요 없음 → 시작점
방 생성 → "방 만들고 코드 나오나?" → 맵 데이터 필요 → 맵 로드에 의존
유닛 생산 → "매초 +1 되나?" → 노드가 있어야 함 → 맵 로드에 의존
전송 시작 → "서울→인천 전송 되나?" → 유닛이 있어야 함 → 유닛 생산에 의존
유닛 이동 → "좌표가 움직이나?" → 이동 유닛이 생겨야 함 → 전송에 의존
도착 상쇄 → "도착해서 싸우나?" → 유닛이 도착해야 함 → 이동에 의존
승리 판정 → "상대 노드 0개면 끝나나?" → 점령이 일어나야 함 → 도착 상쇄에 의존
```

### 의존 그래프 그리는 법

**1단계: 기능을 전부 나열한다**

```
F-01 맵 로드
F-02 방 생성
F-03 방 참가
F-04 대기/카운트다운
F-05 유닛 생산
F-06 전송 시작
F-07 자동 전송
F-08 유닛 이동
F-09 도착 상쇄/점령
F-10 이동 중 상쇄
F-11 승리 판정
```

**2단계: 각 기능에 "확인하려면 뭐가 필요하지?" 질문**

```
F-01 맵 로드       → 의존 없음 (시작점)
F-02 방 생성       → F-01 (맵 데이터가 있어야 방에 맵 넣음)
F-03 방 참가       → F-02 (방이 있어야 참가)
F-04 대기/카운트다운 → F-03 (참가해야 대기)
F-05 유닛 생산     → F-01 (노드가 있어야 생산)
F-06 전송 시작     → F-05 (유닛이 있어야 전송)
F-07 자동 전송     → F-06 (전송 명령이 있어야 자동 발사)
F-08 유닛 이동     → F-07 (이동 유닛이 생겨야 이동)
F-09 도착 상쇄     → F-08 (도착해야 상쇄)
F-10 이동 중 상쇄  → F-08 (양쪽에서 이동 중이어야 만남)
F-11 승리 판정     → F-09 (점령이 일어나야 노드 수 변동)
```

**3단계: 그래프로 그린다**

```
F-01 맵 로드 (시작점)
  │
  ├──→ F-02 방 생성
  │      └──→ F-03 방 참가
  │             └──→ F-04 대기/카운트다운
  │
  └──→ F-05 유닛 생산
         └──→ F-06 전송 시작
               └──→ F-07 자동 전송
                      └──→ F-08 유닛 이동
                             ├──→ F-09 도착 상쇄/점령
                             │      └──→ F-11 승리 판정
                             └──→ F-10 이동 중 상쇄
```

**4단계: 시작점(의존 없음)을 찾는다 → 거기서부터 만든다**

```
의존 없음 = 시작점 = 가장 먼저 만들 수 있는 기능
여러 기능이 의존하는 기능 = 병목 = 일찍 안정화해야 함
의존 체인이 긴 기능 = 리스크 = 일찍 시작해야 함
```

### 왜 순서를 안 지키면 문제인가

```
나쁜 예시: "이동 중 상쇄를 먼저 만들자!"

→ 코드를 짰다
→ 테스트하려면? 양쪽에서 유닛이 이동해야 한다
→ 이동하려면? 이동 처리가 있어야 한다
→ 이동 유닛이 생기려면? 자동 전송이 있어야 한다
→ 전송하려면? 전송 시작이 있어야 한다
→ 전송할 유닛이 있으려면? 유닛 생산이 있어야 한다

결과: 코드 짰는데 5개 기능이 없어서 테스트 불가.
"아마 맞을 거야" → 나중에 전부 연결하면 버그 폭탄.
```

```
좋은 예시: 의존 순서대로 만들기

F-01 맵 로드 완성 → "노드 12개 로드 됐나?" 로그 확인 ✅
F-05 유닛 생산 완성 → "매초 +1 되나?" 로그 확인 ✅
F-06 전송 시작 완성 → "전송 명령 등록됐나?" 로그 확인 ✅
F-07 자동 전송 완성 → "유닛 빠져나가나?" 로그 확인 ✅
F-08 유닛 이동 완성 → "좌표 이동하나?" 로그 확인 ✅
F-10 이동 중 상쇄 완성 → "만나면 사라지나?" 로그 확인 ✅

매 단계마다 "여기까진 확실히 동작한다"를 쌓아가는 것.
```

### 병렬 작업이 가능한 지점 찾기

그래프에서 **서로 의존하지 않는 가지**는 동시에 작업할 수 있다:

```
F-01 맵 로드
  │
  ├──→ F-02~F-04 (방 시스템)     ← 개발자 B
  │
  └──→ F-05~F-11 (게임 로직)    ← 개발자 A

방 시스템과 게임 로직은 서로 의존하지 않음
→ F-01만 끝나면 둘이 나눠서 동시에 작업 가능
→ 이게 "Week 1에서 인프라 잡고, Week 2부터 분담"의 근거
```

### 다른 프로젝트에도 적용

```
예시: 쇼핑몰

회원가입 (시작점)
  └──→ 로그인
         ├──→ 상품 목록 (로그인 없이도 가능? → 의존 없음으로 분리 가능)
         ├──→ 장바구니 담기
         │      └──→ 주문하기
         │             └──→ 결제
         │                    └──→ 주문 내역 조회
         └──→ 리뷰 작성 → 상품을 구매해야 리뷰 가능? → 결제에 의존

이렇게 그리면 "뭘 먼저 만들지"가 자동으로 나옴.
```

```
예시: Library 프로젝트 (네가 만든 것)

서버 소켓 열기 (시작점)
  └──→ HTTP 파싱
         └──→ 라우팅
                ├──→ 도서 목록 조회
                ├──→ 도서 등록 → 목록에 표시되는지 확인 → 목록 조회에 의존
                └──→ 대출 → 도서가 있어야 → 등록에 의존
                       └──→ 반납 → 대출 상태여야 → 대출에 의존
                              └──→ 연체 확인 → 반납 기한이 지나야 → 대출에 의존
```

### 실전 체크리스트

프로젝트 시작할 때 이 순서로:

```
1. 기능을 전부 나열한다
2. 각 기능에 "확인하려면 뭐가 필요?" 질문한다
3. 의존 그래프를 그린다
4. 시작점(의존 없음)을 찾는다 → 여기서부터 만든다
5. 서로 의존 안 하는 가지를 찾는다 → 팀원 간 분담 가능 지점
6. 의존 체인이 가장 긴 경로를 찾는다 → 일정 리스크 (여기가 밀리면 전체가 밀림)
7. 매 기능 완성 시 "확인" 단계를 거친다 → "여기까진 동작함" 확인
```

> 6번의 "가장 긴 경로"를 프로젝트 관리에서는 **크리티컬 패스(Critical Path)**라고 부른다. PM이 일정을 관리할 때 핵심적으로 보는 개념.

## 면접 포인트

- **"프로젝트에서 개발 순서를 어떻게 정했나요?"** → 기능 간 의존관계를 파악하고, 의존 없는 기능부터 시작. 매 단계마다 동작 확인 후 다음으로. 서로 독립적인 가지는 팀원 간 분담.
- **"팀 프로젝트에서 분담은 어떻게?"** → 기능 의존 그래프에서 서로 의존하지 않는 가지를 찾아서 분담. 공통 의존(인프라)을 먼저 한 명이 잡고, 이후 분기.
- **"기능 의존관계와 클래스 의존관계 차이?"** → 클래스 의존은 코드 구조(컴파일 시점), 기능 의존은 개발/테스트 순서(프로젝트 관리 시점). 다른 레벨의 분석.

## 실무적용

- 스프린트 계획: 의존 그래프로 "이번 스프린트에 뭘 먼저 끝내야 다음 스프린트가 막히지 않나" 판단
- 팀 분담: 독립적인 가지를 팀원에게 배분
- 리스크 관리: 크리티컬 패스(가장 긴 의존 체인)가 밀리면 전체 일정이 밀림
- PM 관점: "이 기능이 늦어지면 어디까지 영향 가나요?" → 의존 그래프 보면 바로 답 나옴

## 개발연결

-  — 관심사 분리가 잘 되어 있으면 기능 간 의존도 줄어듦
-  — API 명세를 먼저 합의하면 프론트/백이 독립적으로 개발 가능 (의존 끊기)
-  — Server/Client/Shared 분리도 의존 관리의 일종
