# 동시성 기초 (Concurrency)

## 정의/배경

- 여러 작업이 **동시에 진행되는 상황**을 다루는 프로그래밍 영역
- "가끔 터지는 버그"의 대부분이 동시성 문제
- 웹 서버는 기본적으로 동시성 환경 (여러 요청이 동시에 들어옴)

## 핵심

### 스레드풀 (Thread Pool)

> ❌ 요청마다 스레드 생성
>    → 스레드 1개 = 메모리 ~1MB, 생성/소멸 비용 큼
>    → 1000명이면 1000개? → 서버 터짐
>
> ✅ 스레드풀 = 미리 스레드 N개 만들어두고 재사용
>    → 요청 들어오면 빌려줌, 끝나면 반환
>    → ASP.NET Core, Spring(Tomcat) 다 이 방식

```java
// Java에서 스레드풀 직접 사용
ExecutorService pool = Executors.newFixedThreadPool(10);
pool.submit(() -> handleRequest(request));  // 풀에서 스레드 빌려서 실행

// Spring Boot는 Tomcat이 내부적으로 스레드풀 관리
// 기본 200개 스레드 → server.tomcat.threads.max=200
```

Capital Wars에서:
- Hub 메서드 → ASP.NET Core **스레드 풀**에서 스레드 빌려서 실행 → 반환
- GameLoopService → BackgroundService 전용 스레드에서 루프
- 개발자가 직접 `new Thread()` 쓸 일 거의 없음

### 레이스 컨디션 (Race Condition)

> 스레드A: units = node.Units       // 10을 읽음
> 스레드B: units = node.Units       // 10을 읽음 (A가 아직 저장 전)
> 스레드A: node.Units = units + 1   // 11 저장
> 스레드B: node.Units = units + 1   // 11 저장 (12여야 하는데!)
>
> → +2 해야 하는데 +1만 됨. "가끔" 발생해서 재현 어려움
> → 테스트에서는 안 터지다가 실서비스 터지는 전형적인 패턴

**왜 "가끔"만 발생하는가?**

> 두 스레드가 정확히 같은 타이밍에 읽어야 문제 발생
> 대부분은 한 스레드가 먼저 끝나서 괜찮아 보임
> → 부하가 높아지면 확률 증가 → 실제 환경에서 터짐
> → "내 로컬에서는 됐는데요" 의 대표적 원인

### 해결 방법

| 방법 | 설명 | C# | Java |
|------|------|-----|------|
| **Lock (Mutex)** | 한 번에 하나만 접근 | `lock(obj) { }` | `synchronized` |
| **스레드 안전 컬렉션** | 내부적으로 lock 처리 | `ConcurrentDictionary` | `ConcurrentHashMap` |
| **Atomic 연산** | 하드웨어 수준 원자성 | `Interlocked.Increment` | `AtomicInteger` |
| **불변 객체** | 수정 자체를 불가능하게 | `readonly`, `record` | `final`, `record` |
| **단일 스레드 처리** | 공유 자원을 한 스레드에서만 접근 | Command Queue | Actor 모델 |

```csharp
// C# lock 예시
private readonly object _lock = new();

public void AddUnit(Node node)
{
    lock (_lock)  // 이 블록은 한 스레드만 들어감
    {
        node.Units += 1;  // 안전
    }
}
```

```java
// Java synchronized 예시
public synchronized void addUnit(Node node) {
    node.setUnits(node.getUnits() + 1);  // 안전
}

// 또는 AtomicInteger (lock 없이 원자적 연산)
AtomicInteger units = new AtomicInteger(10);
units.incrementAndGet();  // 11, 원자적으로 실행
```

### 교착상태 (Deadlock)

> 스레드A: lockA 확보 → lockB 대기 중...
> 스레드B: lockB 확보 → lockA 대기 중...
> → 서로가 서로를 기다림 → 영원히 멈춤
>
> 발생 조건 4가지 (전부 만족해야 발생):
> 1. 상호 배제: 자원을 한 스레드만 사용
> 2. 점유 대기: 자원을 가진 채 다른 자원 대기
> 3. 비선점: 다른 스레드의 자원을 강제로 뺏을 수 없음
> 4. 순환 대기: A→B→A 순환 구조
>
> 해결: 락 획득 순서 통일 (항상 lockA → lockB 순서로 하면)

```java
// ❌ 교착상태 발생 가능
// Thread A: synchronized(lockA) { synchronized(lockB) { ... } }
// Thread B: synchronized(lockB) { synchronized(lockA) { ... } }

// ✅ 락 순서 통일
// Thread A: synchronized(lockA) { synchronized(lockB) { ... } }
// Thread B: synchronized(lockA) { synchronized(lockB) { ... } }
```

### Capital Wars에서 동시성 주의점

> GameLoopService (스레드 A): 매 150ms마다 GameState 읽기/쓰기
> Hub 메서드 (스레드 B): StartTransfer → ActiveTransfers에 추가
>
> 두 스레드가 같은 List<TransferOrder>에 동시 접근 → 위험!
>
> 해결 선택지:
> 1. ConcurrentDictionary 사용 (스레드 안전 컬렉션)
> 2. lock으로 보호
> 3. Command Queue 패턴 (Hub는 큐에 넣기만, 루프에서만 처리)
>    → 가장 깔끔. 동시성 문제를 구조적으로 제거

### 동시성 ≠ 병렬성

> 동시성 (Concurrency) = 여러 작업을 "번갈아가며" 진행 (1코어도 가능)
>   → 구조적 문제: "동시에 진행 중인 작업을 어떻게 관리할까"
>
> 병렬성 (Parallelism) = 여러 작업을 "물리적으로 동시에" 진행 (멀티코어 필요)
>   → 실행의 문제: "일을 쪼개서 동시에 처리하자"
>
> 비유:
>   동시성 = 요리사 1명이 한식/스테이크 번갈아 조리 (컨텍스트 스위칭)
>   병렬성 = 요리사 2명이 각각 하나씩 동시에 조리
>
> 관계:
>   동시성 없이 병렬성 = 가능 (독립된 작업을 병렬 실행)
>   병렬성 없이 동시성 = 가능 (싱글코어에서 번갈아 실행)
>   실제로는 둘 다 함께 = 멀티코어에서 동시성 위에 병렬 처리

### volatile 키워드

```java
// Java
private volatile boolean running = true;

// volatile이 없으면?
// 스레드가 변수를 CPU 캐시에 복사해서 사용
// 다른 스레드가 바꿔도 캐시된 값을 계속 읽음 → 변경 감지 못함

// volatile = "항상 메인 메모리에서 읽어라"
// 가시성(Visibility) 보장
// 주의: 원자성은 보장하지 않음 (읽기/쓰기는 원자적, 읽고+쓰기는 아님)
```

```csharp
// C#에서는 volatile 키워드도 있지만
// Interlocked 클래스가 더 안전하고 자주 사용됨
Interlocked.Increment(ref counter);  // 원자적 증가
Interlocked.CompareExchange(ref value, newValue, expected);  // CAS
```

## 면접 포인트

- **"레이스 컨디션이 뭔가요?"** → 여러 스레드가 공유 자원에 동시 접근해서 실행 순서에 따라 결과가 달라지는 버그. 재현이 어렵고 부하 높을 때 더 발생.
- **"교착상태 발생 조건과 해결?"** → 상호배제+점유대기+비선점+순환대기 4가지 동시 충족 시 발생. 락 획득 순서 통일, 타임아웃 설정, lock-free 알고리즘으로 해결.
- **"동시성과 병렬성 차이?"** → 동시성은 구조(여러 작업 관리), 병렬성은 실행(물리적 동시 처리). 1코어도 동시성 가능, 병렬성은 멀티코어 필요.
- **"ConcurrentHashMap이 HashMap보다 느린 이유?"** → 내부적으로 락/CAS 연산 오버헤드. 동시성이 필요 없으면 HashMap이 더 빠름.

## 실무연결

- 웹 서버는 기본적으로 동시성 환경 (여러 요청 동시 처리)
- DB 트랜잭션은 동시성 문제 해결 수단
- **"이 코드가 동시에 실행되면 안전한가?"** 항상 질문해야 함
- PM 관점: 동시성 버그는 재현이 어렵고 수정 비용이 높다. "동시에 같은 자원을 수정하는 경우가 없는가?"라는 질문으로 사전 방지.

## 개발연결

-  → 비동기는 동시성을 효율적으로 다루는 방법 중 하나
-  → 루프 스레드와 Hub 스레드의 공유 자원 접근
-  → 동시성 해결 비용도 성능에 영향
-  → 불변 객체는 동시성 문제를 구조적으로 방지
