# 핵심 설계 원칙

## 1. 관심사 분리 (Separation of Concerns)

### 정의

- 하나의 모듈/클래스는 **하나의 책임**만 가진다
- "한 클래스가 두 가지 일을 하면 설계가 틀린 것"
- SOLID 원칙 중 **SRP(Single Responsibility Principle)**과 같은 맥락

### Capital Wars에서

> GameHub        → 라우팅만 (요청 받아서 서비스에 넘김)
> GameService    → 게임 로직만 (점수, 점령, 승리)
> GameLoopService → 타이밍만 (매 틱 실행, 브로드캐스트)
> RoomService    → 방 관리만 (생성, 입장, 나가기)

```java
// Java/Spring 대응
@RestController  → 라우팅만 (요청 받아서 서비스에 넘김)
@Service         → 비즈니스 로직만
@Repository      → 데이터 접근만
```

### 나쁜 신호

> 🚨 Hub/Controller에 10줄 넘는 로직이 있다
>    → Service로 분리해야 함
>
> 🚨 한 서비스가 5개 이상의 다른 서비스를 의존한다
>    → 책임이 너무 많음, 쪼개야 함
>
> 🚨 클래스 이름에 "And"가 들어간다
>    → UserAndOrderService → UserService + OrderService
>
> 🚨 메서드가 "여러 이유"로 변경된다
>    → 비즈니스 로직 변경 때도, UI 변경 때도 수정해야 한다면
>    → 두 관심사가 섞여 있다는 신호

### 계층별 관심사 분리 (Layered Architecture)

> ┌─────────────────────────┐
> │   Presentation          │ ← Controller/Hub: 요청 받기, 응답 보내기
> ├─────────────────────────┤
> │   Business Logic        │ ← Service: 핵심 로직, 규칙
> ├─────────────────────────┤
> │   Data Access           │ ← Repository: DB 접근
> ├─────────────────────────┤
> │   Database              │ ← 저장소
> └─────────────────────────┘
>
> 각 계층은 바로 아래 계층만 알아야 함
> Controller가 DB를 직접 호출? → 위반
> Service가 HTTP 응답 형식을 알아야 함? → 위반

## 2. 단일 집중 원천 (Single Source of Truth, SSoT)

### 정의

- 특정 데이터의 **원본(정본)은 딱 한 곳**에만 존재
- 다른 곳에서는 복사본을 받아서 쓸 뿐, 수정 권한 없음
- "이 데이터의 진짜 주인이 누구냐"가 항상 명확해야 함

### Capital Wars에서

> 게임 상태의 집중 = 서버의 GameState 단 하나의 필드
> 클라이언트가 가진 건 = GameStateDto (읽기 전용 복사본)
>
> 클라이언트가 "유닛 +1" 하면? → 무시됨
> 서버만 GameState를 수정할 수 있음

### SSoT 위반 시 생기는 일

> ❌ 서버 DB에도 유저 이름, 캐시에도 유저 이름, 클라이언트에도 유저 이름
>    → 유저가 이름 변경
>    → DB는 업데이트됨, 캐시는 안 됨, 클라이언트는 안 됨
>    → 어디를 보느냐에 따라 다른 이름이 보임
>
> ✅ DB가 유일한 집중, 캐시는 DB에서 파생, 클라이언트는 API로 조회
>    → 이름 변경 시 DB만 수정 → 캐시 무효화 → 클라이언트 재조회

### 어디서든 적용

| 분야 | 진정한 원천 | 복사본 |
|------|------------|--------|
| 프론트엔드 (Redux/Zustand) | Store | 컴포넌트 로컬 상태 |
| DB 설계 | 정규화된 테이블 | 캐시, 뷰, 비정규화 테이블 |
| 멀티플레이어 게임 | 서버 GameState | 클라이언트 화면 |
| 설정 관리 | appsettings.json | 환경변수 오버라이드 |
| 팀 문서 | Confluence 원본 | 개인 메모, 슬랙 공유 |

### 코드 수준의 SSoT

```java
// ❌ SSoT 위반 → 같은 정보가 두 곳에
public class User {
    String firstName;
    String lastName;
    String fullName;  // firstName + lastName과 항상 같아야 함
    // → fullName을 별도 수정하면 불일치 발생
}

// ✅ SSoT 준수 → 파생값은 계산으로
public class User {
    String firstName;
    String lastName;
    public String getFullName() { return firstName + " " + lastName; }
    // → fullName은 항상 firstName + lastName임을 코드가 보장
}
```

## 3. 의존성 주입 (Dependency Injection, DI)

### 정의

- 객체가 필요한 의존성을 **직접 생성하지 않고, 외부에서 받는** 패턴
- "내가 직접 new 하지 않고, 누군가 넣어준다"
- SOLID의 **DIP(Dependency Inversion Principle)**를 구현하는 방법

### 왜 직접 new 하면 안 되나

```csharp
// ❌ 직접 생성 → 강한 결합
public class GameHub
{
    private RoomService _room = new RoomService();  // 하드코딩
}
// 문제:
// 1. 테스트 시 가짜(Mock) 객체로 교체 불가
// 2. RoomService 생성자가 바뀌면 GameHub도 수정 필요
// 3. RoomService 대신 다른 구현체로 교체 불가

// ✅ 주입 → 느슨한 결합
public class GameHub
{
    private readonly IRoomService _room;
    public GameHub(IRoomService room)  // 인터페이스로 주입
    {
        _room = room;
    }
}
// 장점:
// 1. 테스트 시 MockRoomService 주입 가능
// 2. RoomService 내부 변경이 GameHub에 영향 없음
// 3. 인터페이스만 맞으면 다른 구현체로 교체 가능
```

```java
// Java/Spring 대응

// ❌ 직접 생성
public class GameController {
    private RoomService room = new RoomService();
}

// ✅ 생성자 주입 (Spring 권장 방식)
@RestController
public class GameController {
    private final RoomService room;

    // Spring 4.3+: 생성자 1개면 @Autowired 생략 가능
    public GameController(RoomService room) {
        this.room = room;
    }
}
```

### DI 등록과 생명주기

```csharp
// C# → Program.cs에서 등록
builder.Services.AddSingleton<IRoomService, RoomService>();  // 앱 전체 1개
builder.Services.AddScoped<IDbContext, AppDbContext>();       // 요청마다 1개
builder.Services.AddTransient<IValidator, Validator>();       // 주입마다 새로
```

```java
// Java/Spring → 어노테이션으로 등록
@Component          // 기본 = Singleton
@Scope("request")   // 요청마다 1개
@Scope("prototype") // 주입마다 새로
```

### 의존성 역전 원칙 (DIP) → DI의 이론적 기반

> DIP: "상위 모듈이 하위 모듈에 의존하지 않는다. 둘 다 추상화에 의존한다."
>
> ❌ DIP 위반
>   GameHub → RoomService (구체 클래스에 직접 의존)
>   → RoomService 변경 시 GameHub 영향
>
> ✅ DIP 준수
>   GameHub → IRoomService (인터페이스에 의존)
>   RoomService → IRoomService (인터페이스를 구현)
>   → 둘 다 인터페이스에 의존, 서로를 모름
>   → RoomService를 MockRoomService로 바꿔도 GameHub 수정 불필요

### Singleton 생명주기 — Lazy vs Eager 초기화

DI 컨테이너의 Singleton은 기본적으로 **지연 초기화(Lazy)** — 누군가 처음 요청할 때 생성됨 (서버 시작 시가 아님). .NET과 Spring은 기본값이 반대다.

| | .NET Singleton | Spring @Component |
|---|---|---|
| 기본 초기화 | Lazy (첫 요청 시) | Eager (서버 시작 시) |
| 반대로 하려면 | `GetRequiredService` | `@Lazy` 어노테이션 |

> Eager가 필요한 경우:
> - 서버 시작 시 데이터 로딩이 필수 (맵 데이터, 설정 파일) — 첫 요청에서 로딩하면 사용자 대기
> - 초기화 실패를 서버 시작 시 즉시 감지하고 싶을 때
> - 초기화가 무거운 작업 (파일 I/O, 네트워크) → 런타임 중 갑자기 느려지는 것 방지

#### Eager 초기화 패턴 — GetRequiredService

```csharp
// Program.cs
builder.Services.AddSingleton<MapLoaderService>();  // 등록만 됨 (아직 안 만들어짐)

var app = builder.Build();

// 강제로 생성 → Eager 초기화
var mapLoader = app.Services.GetRequiredService<MapLoaderService>();
// → MapLoaderService 생성자 실행 → 맵 파일 로드 완료
// → 이후 모든 요청에서 이미 로드된 데이터 사용

app.Run();
```

| 상황 | 선택 | 이유 |
|------|------|------|
| 맵 데이터, 설정 파일 로드 | Eager | 서버 시작 시 준비 완료되어야 함 |
| DB 연결 풀 | Eager | 첫 요청 지연 방지 |
| 일반 Service (로직만) | Lazy (기본) | 생성 비용 낮음 |
| 거의 안 쓰이는 Service | Lazy | 불필요한 메모리 사용 방지 |

## 4. SOLID 원칙 요약 (추가)

| 원칙 | 이름 | 한줄 요약 |
|------|------|-----------|
| **S** | Single Responsibility | 클래스는 하나의 이유로만 변경되어야 함 |
| **O** | Open/Closed | 확장에는 열려있고, 수정에는 닫혀있어야 함 |
| **L** | Liskov Substitution | 부모 타입 자리에 자식 타입을 넣어도 동작해야 함 |
| **I** | Interface Segregation | 인터페이스는 작고 구체적으로 분리 |
| **D** | Dependency Inversion | 구체 클래스가 아닌 추상화에 의존 |

> S = 관심사 분리와 연결
> O = 새 기능을 기존 코드 수정 없이 추가 (이벤트 기반과 연결)
> L = 다형성의 올바른 사용
> I = "뚱뚱한 인터페이스"보다 작은 인터페이스 여러 개
> D = DI의 이론적 기반
>
> 면접에서 SOLID를 전부 설명하라고 하면 S, O, D 위주로 설명하고 나머지는 간단히. 실무에서 가장 체감되는 것 이 순서이기 때문.

## 면접 포인트

- **"관심사 분리란?"** → 하나의 모듈이 하나의 책임만 갖는 원칙. Controller는 라우팅, Service는 로직, Repository는 데이터 접근. 변경 이유가 하나여야 함.
- **"SSoT란?"** → 특정 데이터의 원본이 딱 한 곳에만 존재. 다른 곳은 파생/복사본. DB가 집중, 캐시는 파생, 프론트는 조회.
- **"DI의 장점?"** → 테스트 가능성(Mock 주입), 느슨한 결합(구현 교체 용이), 관심사 분리(생성과 사용 분리).
- **"DI 없이 테스트하기 어려운 이유?"** → 직접 new하면 실제 DB/네트워크에 의존 → 단위 테스트 불가. DI로 Mock을 주입하면 외부 의존 없이 로직만 테스트 가능.

## 실무연결

- 이 3가지(+SOLID)는 현대 프레임워크, 언어 무관하게 어디서나 적용
- 면접에서는 단골 질문 → 특히 DI, SRP, SSoT
- PM 관점:
  - SoC → "이 변경이 다른 기능에 영향 주나요?" 판단 기준
  - SSoT → "이 데이터의 주인이 누구냐" 이해해야 의사결정 가능
  - DI → "이 모듈을 다른 팀이 교체할 수 있나요?" 아키텍처 유연성 판단

## 개발연결

-  → SSoT의 게임에 적용 사례
-  → DI가 내장된 프레임워크
-  → SoC를 통신 레벨로 확장한 패턴
-  → 불변 객체(SSoT 방식)가 동시성 문제를 구조적으로 방지
