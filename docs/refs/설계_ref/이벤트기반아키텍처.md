# 이벤트 기반 아키텍처

## 정의/배경

- 컴포넌트 간 통신을 **직접 호출이 아니라 이벤트 발행/구독**으로 하는 아키텍처
- "A가 B를 직접 부른다" → "A가 이벤트를 던지고, B가 자신이 받는다"
- 디자인 패턴의 **Observer 패턴**이 아키텍처 수준으로 확장된 것

## 핵심

### 직접 호출 vs 이벤트

> 직접 호출 (결합도 높은 방식):
>   GameLoopService → GameService.ProcessArrival()
>                   → GameHub.SendState()
>   → 루프가 "누가 있는지", "누구한테 보내야 하는지" 다 알아야 함
>   → 새 기능 추가 시 루프 코드 수정 필요
>
> 이벤트 기반 (결합도 낮은 방식):
>   GameLoopService → "TickCompleted" 이벤트 발행
>   GameService가 구독 → 도착 처리
>   GameHub가 구독 → 상태 전송
>   LogService가 구독 → 로깅 (새로 추가해도 루프 수정 없음!)
>   → 루프는 구독자가 누군지 몰라도 됨

### Observer 패턴 (이벤트 기반의 근본)

```java
// Java로 본 Observer 패턴 핵심 구조

// 발행자 (Subject/Publisher)
public class EventBus {
    private Map<String, List<Consumer<Object>>> listeners = new HashMap<>();

    public void subscribe(String event, Consumer<Object> listener) {
        listeners.computeIfAbsent(event, k -> new ArrayList<>()).add(listener);
    }

    public void publish(String event, Object data) {
        listeners.getOrDefault(event, List.of())
                 .forEach(listener -> listener.accept(data));
    }
}

// 사용
eventBus.subscribe("OrderCreated", order -> sendEmail(order));
eventBus.subscribe("OrderCreated", order -> updateInventory(order));
eventBus.publish("OrderCreated", newOrder);
// → 발행자는 구독자를 모름. 구독자 추가/제거가 자유로움.
```

```csharp
// C#에서는 event 키워드가 언어 수준에서 지원
public class GameLoopService
{
    public event Action<GameState>? OnTickCompleted;

    private void Tick()
    {
        // 상태 업데이트 후
        OnTickCompleted?.Invoke(gameState);  // 구독자 전원에게 알림
    }
}

// 구독
gameLoop.OnTickCompleted += state => hub.BroadcastState(state);
gameLoop.OnTickCompleted += state => logger.LogTick(state);
```

### SignalR에서의 이벤트 패턴

> 서버 → 클라이언트 (발행/구독):
>   서버: Clients.Group(roomId).SendAsync("GameStateUpdated", state)
>   클라이: hubConnection.On<GameStateDto>("GameStateUpdated", state => { ... })
>   → 서버가 "GameStateUpdated"를 발행, 클라이언트가 구독
>
> 클라이언트 → 서버 (RPC 호출):
>   클라이: hubConnection.InvokeAsync("StartTransfer", from, to, count)
>   서버: Hub의 StartTransfer 메서드가 실행
>   → 이건 이벤트보다는 직접 호출에 가까움
>
> 하이브리드:
>   클라 → 서버 = 직접 호출 (명확한 요청)
>   서버 → 클라 = 이벤트 (브로드캐스트, 구독 기반)

### 이벤트 기반의 3가지 레벨

| 레벨 | 범위 | 예시 | 기술 |
|------|------|------|------|
| **언어 수준** | 클래스 간 | C# event, Java Observer | 직접 구현 |
| **프로세스 수준** | 모듈 간 | SignalR, EventEmitter | 프레임워크 내장 |
| **시스템 수준** | 서비스 간 | Kafka, RabbitMQ | 메시지 브로커 |

> 소규모 (Capital Wars):
>   SignalR로 서버↔클라이언트 이벤트 통신
>   프로세스 내에서 직접 호출 + 이벤트 혼합
>
> 대규모 (MSA):
>   서비스 A → Kafka → 서비스 B, C, D
>   서비스 간 직접 호출 최소화
>   이벤트 = 비동기 메시지
>
> 원리는 동일: 발행자는 구독자를 모르고, 새 구독자 추가가 쉬움

### 이벤트 소싱 (Event Sourcing) → 심화

> 일반적인 방식: 현재 상태만 저장
>   → DB에 "잔고: 10,000원" 저장
>
> 이벤트 소싱: 상태 변화 이벤트를 전부 저장
>   → 입금 5,000원 → 출금 2,000원 → 입금 7,000원
>   → 이벤트를 순서대로 재실행하면 현재 상태(10,000원) 복원 가능
>
> 장점: 모든 변경 이력 추적 가능, 특정 시점으로 되돌리기 가능
> 단점: 구현 복잡도 높음, 이벤트 수가 많아지면 재실행 느려짐
>
> 적용: 금융(거래 이력), 게임(리플레이), Git(커밋 히스토리)
> → 지금 당장 쓸 일은 없지만, 이벤트 기반의 극단적인 형태로 알아두면 좋음

### 장단점

> 장점:
>   → 느슨한 결합 → 발행자/구독자가 서로를 모름
>   → 확장 쉬움 → 새 구독자 추가 시 기존 코드 수정 없음
>   → 비동기 처리 자연스러움 → 이벤트를 큐에 넣고 나중에 처리 가능
>   → 팀 간 독립 개발 → 인터페이스(이벤트 명세)만 합의하면 됨
>
> 단점:
>   → 흐름 추적 어려움 → "이 이벤트 누가 받고 있지?" 파악 힘든 흐름
>   → 디버깅 복잡 → 콜 스택 대신 이벤트 체인 추적해야 함
>   → 이벤트 순서 보장 어려움 → 여러 구독자의 실행 순서가 불확실
>   → 이벤트 폭풍 → 이벤트가 이벤트를 발생시키는 연쇄 → 예측 불가

### 언제 직접 호출, 언제 이벤트?

> 직접 호출이 나은 경우:
>   → 호출 대상이 명확하고 1개
>   → 호출 결과가 즉시 필요 (동기적)
>   → 흐름이 단순하고 추적이 중요
>
> 이벤트가 나은 경우:
>   → 하나의 사건에 여러 곳이 반응해야 할 때
>   → 발행자가 구독자를 알면 안 될 때
>   → 향후 구독자가 추가될 가능성이 높을 때
>   → 느슨한 결합이 중요할 때 (MSA)

## 면접 포인트

- **"이벤트 기반 아키텍처란?"** → 컴포넌트 간 통신을 직접 호출 대신 이벤트 발행/구독으로 하는 구조. 느슨한 결합과 확장성이 장점.
- **"Observer 패턴과 Pub/Sub 차이?"** → Observer는 발행자가 구독자 목록을 직접 관리, Pub/Sub는 중간에 브로커(이벤트 버스)가 있어서 발행자/구독자가 완전히 분리.
- **"Kafka가 왜 필요한가요?"** → 프로세스 내 이벤트는 서버 재시작 시 유실. Kafka는 이벤트를 영속적으로 저장하고, 서비스 간 비동기 통신을 보장.
- **"이벤트 순서가 중요할 때는?"** → 파티션 키로 같은 키의 이벤트를 같은 파티션에 보내면 순서 보장 (Kafka 패턴).

## 실무연결

- Capital Wars: SignalR 전체가 이벤트 기반
- 프론트엔드: DOM 이벤트(click, change), React의 onChange
- 백엔드: Spring ApplicationEvent, C# event 키워드
- MSA: Kafka, RabbitMQ로 서비스 간 메시지 통신
- PM 관점: 이벤트 기반 = 팀 간 독립 개발 가능. 서비스 간 계약(이벤트 스키마)만 정하면 각 팀이 독립적으로 개발/배포 가능.

## 개발연결

-  → 이 프로젝트의 이벤트 기반 통신 구현체
-  → 관심사 분리를 통신 레벨로 확장한 패턴
-  → 이벤트 처리가 동기/비동기 모두 가능
-  → 서버의 이벤트를 클라이언트가 구독하는 구조
