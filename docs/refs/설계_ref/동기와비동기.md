# 동기와 비동기

## 정의/배경

- **동기(Synchronous)**: 작업이 끝날 때까지 기다린 뒤 다음 작업 진행
- **비동기(Asynchronous)**: 작업을 시작만 해두고, 끝나길 기다리지 않고 다음 작업 진행
- 현대 서버/앱 개발에서 가장 기본적이면서 중요한 개념

## 핵심

### 비유로 이해하기

```
동기  = 카운터 앞에서 음식 나올 때까지 서서 기다림 (아무것도 못 함)
비동기 = 진동벨 받고 자리에서 다른 일 하다가 벨 울리면 받으러 감

핵심: "기다리는 동안 아무것도 못 하냐 vs 다른 일 할 수 있냐"
```

---

### 블로킹 (Blocking) — 먼저 이해해야 하는 개념

**"이 줄에서 결과 올 때까지 스레드가 멈추는 것."**

```java
// Library 프로젝트에서 전부 블로킹 호출
serverSocket.accept();   // 누가 접속할 때까지 멈춤
reader.readLine();        // 파일에서 한 줄 읽을 때까지 멈춤
inputStream.read();       // 소켓에서 데이터 올 때까지 멈춤
```

이 줄에서 스레드가 아무것도 안 하고 서 있음. CPU는 놀고 있는데 스레드가 잡혀있는 상태.

```
블로킹 vs 논블로킹:

블로킹:    "결과 줄 때까지 여기서 기다릴게" → 스레드 멈춤
논블로킹:  "요청 보낼게, 결과 나오면 알려줘" → 스레드 안 멈춤

블로킹이 문제되는 시점:
  스레드 10개, 전부 블로킹 중 (파일 대기, DB 대기 등)
  → 11번째 요청: "스레드 없어, 기다려"
  → CPU는 놀고 있는데 요청을 못 받는 상황
```

### 스레드 풀 (Thread Pool)

**스레드 풀 = 미리 스레드를 만들어두고 재사용하는 것.**

```java
// Library 프로젝트에서 직접 사용한 코드
ExecutorService pool = Executors.newFixedThreadPool(10);
```

```
스레드 풀 없이:
  요청 올 때마다 new Thread() → 끝나면 스레드 죽음
  → 생성/소멸 비용이 비쌈 (메모리 할당, OS 등록)
  → 1000명이면 1000개 생성 → 서버 터짐

스레드 풀:
  미리 10개 만들어둠
  → 요청 오면: 놀고 있는 스레드 빌려줌
  → 요청 끝나면: 스레드를 죽이지 않고 풀에 반납
  → 다음 요청이 같은 스레드를 재사용

  놀고 있는 스레드가 없으면?
  → 대기열(Queue)에서 기다림
  → 앞 작업 끝나서 스레드 반납되면 그때 실행
```

```java
// Library 프로젝트에서 일어나는 일
while (running) {
    Socket client = serverSocket.accept();           // 연결 대기 (블로킹)
    pool.submit(new HttpRequestHandlerV4(client));   // 풀에서 스레드 빌려서 실행
}
// HttpRequestHandlerV4.run()이 끝나면 → 스레드가 풀에 반납됨
// → Spring Boot의 Tomcat도 이 구조. 스레드가 10개가 아니라 기본 200개.
```

### 왜 비동기가 중요한가 — I/O 대기 시간의 낭비

```
서버가 100명의 요청을 동시에 처리해야 할 때:

동기 방식:
  요청 1 → 스레드 1 → DB 요청 보냄 → 멈춤... 기다림... → 응답 옴 → 처리
  요청 2 → 스레드 2 → DB 요청 보냄 → 멈춤... 기다림... → 응답 옴 → 처리
  ...
  요청 100 → 스레드 100개 필요
  스레드 1개 = 메모리 ~1MB → 대기 중인 스레드만 100MB 낭비

비동기 방식:
  요청 1 → 스레드 1 → DB 요청 보냄 → 스레드 풀에 반납 (안 기다림!)
  요청 2 → 스레드 1 재사용 → DB 요청 보냄 → 스레드 반납
  ...
  DB 응답 오면 → 풀에서 아무 스레드나 받아서 이어서 처리
  → 스레드 몇 개로 100명 처리 가능
```

**핵심: 비동기는 I/O 대기 시간에 스레드를 놓아줘서 다른 요청을 처리할 수 있게 한다.** CPU가 바쁜 게 아니라 네트워크/디스크를 기다리는 것이므로, 스레드가 붙잡고 있을 이유가 없다.

---

### ExecutorService(동기 스레드풀) vs 비동기 스레드풀 — 핵심 차이

이 둘은 비슷해 보이지만 다르다.

```
Library 프로젝트의 ExecutorService (동기 + 스레드풀):
  pool.submit(task)
  → 스레드가 task를 실행
  → task 안에서 readLine() → 블로킹 → 스레드 멈춤
  → 파일 읽기 끝남 → 같은 스레드가 이어서 실행
  → task 끝남 → 스레드 풀에 반납

  핵심: 스레드 1개가 task 1개를 처음부터 끝까지 전담
  중간에 블로킹되면 그 스레드는 잡혀있음
```

```
비동기 스레드풀:
  스레드 1이 task 시작 → DB 요청 보냄 → 스레드 1 풀에 반납 (블로킹 안 함!)
  (스레드 1은 다른 task를 처리하러 감)
  DB 응답 옴 → 풀에서 아무 스레드(3이든 7이든) → 나머지 이어서 처리

  핵심: 스레드가 task 중간에 놓아줌
  놓아준 동안 다른 task를 처리하다가, 응답 오면 이어서 처리
```

그림으로 비교:

```
동기 + 스레드풀 (Library 프로젝트):
  스레드1: [A시작]──[파일대기.........]──[처리]──[응답]──[반납]
  스레드2: [B시작]──[파일대기.........]──[처리]──[응답]──[반납]
  → 대기 중에도 스레드가 잡혀있음 → 10개 스레드 = 동시 10명만

비동기 + 스레드풀:
  스레드1: [A시작]─[DB요청]─[반납]─[C시작]─[반납]─[A이어서]─[반납]
  스레드2: [B시작]─[DB요청]─[반납]─[D시작]─[반납]─[B이어서]─...
  → 대기 시간에 다른 일을 함 → 3개 스레드로 수백 명 처리 가능
```

**핵심 차이: "블로킹 구간에서 스레드를 놓아주냐 안 놓아주냐."**
ExecutorService 자체는 그냥 스레드 풀. 안에서 블로킹 코드를 쓰면 동기, 논블로킹 코드를 쓰면 비동기.

### 비동기 ≠ 멀티스레드 (다른 개념)

```
멀티스레드 = "일하는 사람 수"의 문제
  → CPU 작업을 여러 스레드가 나눠서 처리
  → 사람을 더 고용하는 것

비동기 = "기다리는 시간 활용"의 문제
  → I/O 대기 중 스레드를 놓아줘서 효율화
  → 한 사람이 대기 시간에 다른 일 하는 것

둘은 별개 개념이라 조합이 가능:
  동기  + 싱글스레드:  초기 프로그래밍 (순차 실행)
  동기  + 멀티스레드:  Library 프로젝트, Spring Boot 기본 ⭐ 대부분 이거
  비동기 + 싱글스레드:  JavaScript (Node.js)
  비동기 + 멀티스레드:  Spring WebFlux, Netty, 게임 서버
```

---

### 비동기가 빛나는 곳 vs 의미 없는 곳

```
✅ I/O 바운드 (기다리는 시간이 긴 작업) → 비동기 효과적
  → 동시 접속 수만 명인 채팅 서버
  → 실시간 알림 서버
  → 여러 외부 API를 동시에 호출해야 할 때
  → 게임 서버 (네트워크 I/O가 많으니까)
  → DB 조회가 많은 API 서버

❌ CPU 바운드 (CPU가 실제로 계산하는 작업) → 비동기 의미 없음
  → 이미지 변환, 영상 인코딩
  → 복잡한 수학 계산, 머신러닝 학습
  → 기다리는 시간이 아니라 CPU가 바쁜 거라 놓아줄 게 없음
  → 이런 건 멀티스레드(병렬 처리)가 답

판단 기준:
  "이 작업이 느린 이유가 뭐지?"
  → CPU가 계산하느라 느림 → 멀티스레드(병렬)
  → DB/네트워크/파일을 기다려서 느림 → 비동기
```

### 실무에서 어떤 조합을 쓰냐

```
대부분의 웹 서버 (Spring Boot 기본):
  → "동기 + 멀티스레드" (Tomcat 스레드풀 200개)
  → 대부분의 서비스는 이걸로 충분
  → Library 프로젝트와 같은 구조, 스레드만 더 많은 것

동시 접속이 아주 많을 때 (수만~수십만):
  → "비동기 + 멀티스레드" (Spring WebFlux, Netty)
  → 또는 Java 21+ Virtual Thread
  → 채팅, 실시간 알림, 게임 서버 등

JavaScript (Node.js):
  → "비동기 + 싱글스레드"
  → I/O 위주 서버에 적합
  → CPU 무거운 작업에는 부적합
```

### Java에서 비동기가 어떻게 생겼냐

```java
// ── 동기 (Library 프로젝트 방식) ──
String result = database.getData();   // 이 줄에서 멈춤
System.out.println(result);           // 위가 끝나야 실행됨
```

```java
// ── 비동기: CompletableFuture (Java 8+) ──
CompletableFuture.supplyAsync(() -> {
    return database.getData();             // 별도 스레드에서 실행
}).thenAccept(result -> {
    System.out.println(result);            // 데이터 오면 이때 실행
});
System.out.println("DB 안 기다리고 바로 여기 옴");
```

```java
// ── 비동기: Virtual Thread (Java 21+) ──
Thread.startVirtualThread(() -> {
    String result = database.getData();    // 동기 코드처럼 생겼는데
    System.out.println(result);            // 내부적으로 비동기처럼 동작
});
// 겉은 동기, 속은 비동기
// JVM이 대기 중 자동으로 스레드 놓아줌
// CompletableFuture의 콜백 지옥 없이 동기 코드 그대로 쓸 수 있음
```

```java
// ── Spring에서 비동기 ──
@Async
public void sendEmail(String to) {
    emailClient.send(to, "환영합니다");  // 별도 스레드에서 실행
}
// 호출하는 쪽은 이메일 발송 안 기다리고 바로 다음 로직으로
```

### Library 프로젝트에서의 스레드 경험 정리

```
직접 해본 것:
  ✅ 스레드 풀 생성 (ExecutorService, newFixedThreadPool)
  ✅ Runnable 구현 (HttpRequestHandlerV4)
  ✅ pool.submit()으로 작업 제출
  ✅ 멀티스레드 환경에서 동시 요청 처리
  ✅ 블로킹 I/O 사용 (accept, readLine, read)

해보지 않은 것:
  ❌ 동시성 보호 (synchronized, ConcurrentHashMap)
     → 스레드 10개가 LibraryMap을 동시에 읽고 쓰는데 보호 없음
  ❌ 비동기 I/O (CompletableFuture, 논블로킹)
  ❌ 스레드 간 통신 (BlockingQueue, CountDownLatch 등)
  ❌ Virtual Thread (Java 21+)
  ❌ @Async (Spring)
```

## 면접 포인트

- **"동기와 비동기 차이?"** → 동기는 결과 올 때까지 스레드가 멈춤(블로킹), 비동기는 요청 후 스레드를 놓아주고 결과 오면 이어서 처리. I/O 바운드 작업에서 비동기가 스레드 효율을 극대화.
- **"비동기와 멀티스레드 차이?"** → 비동기는 기다리는 방식(I/O 효율), 멀티스레드는 일하는 주체 수(CPU 병렬). JavaScript는 싱글스레드인데 비동기.
- **"스레드 풀이 왜 필요?"** → 스레드 생성/소멸 비용이 비쌈. 미리 만들어두고 재사용하면 오버헤드 감소. 동시 스레드 수도 제한 가능.
- **"블로킹 I/O의 문제점?"** → 스레드가 대기 중 점유됨 → 동시 요청 수만큼 스레드 필요 → 메모리 낭비 + 컨텍스트 스위칭 비용.
- **"CPU 바운드에 비동기가 의미 없는 이유?"** → 비동기는 I/O 대기 시간에 스레드를 놓아주는 것. CPU가 연산 중이면 놓아줄 대기 시간이 없음. 병렬 처리가 답.

## 실무적용

- Spring Boot 기본 = 동기 + 멀티스레드 (Tomcat 200개) → 대부분 충분
- "이 작업이 느린 이유가 I/O 대기냐 CPU 연산이냐"가 비동기/병렬 선택 기준
- PM 관점: 비동기 도입은 서버 처리량(throughput)을 높이지만 코드 복잡도도 올림. "동시 접속 몇 명까지 버텨야 하냐"가 기술 결정의 기준.
