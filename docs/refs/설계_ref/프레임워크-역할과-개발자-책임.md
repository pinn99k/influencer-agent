# 프레임워크 역할과 개발자 책임

## 정의/배경

프레임워크는 인프라(스레드, 소켓, HTTP)를 맡고, 개발자는 "무엇을 만들 것인가"에 집중한다.

## 핵심

### 계층 구조

> 내가 짜는 코드 (비즈니스 로직) ← 여기만 집중
> 프레임워크 (Spring Boot / ASP.NET Core) ← 라우팅, DI, 트랜잭션
> 웹 서버 (Tomcat / Kestrel) ← 스레드풀, HTTP 파싱, 소켓
> JVM / CLR / OS ← 스레드 스케줄링

Library 프로젝트에서는 아래 3개 계층을 전부 직접 만들었다. 보통은 맨 위 한 칸만 작업한다.

### 프레임워크가 자동으로 해주는 것

| 영역 | 직접 구현 (Library 프로젝트) | Spring Boot (자동) |
|------|---|---|
| 소켓/연결 | `ServerSocket.accept()` | Tomcat 내장 |
| 스레드풀 | `Executors.newFixedThreadPool(10)` | Tomcat 기본 200개 |
| HTTP 파싱 | `HttpRequest` 직접 구현 | `HttpServletRequest` 제공 |
| 라우팅 | `ServletManager` 직접 구현 | `@GetMapping` 등 |
| 세션 | `SessionManager` + UUID | `HttpSession` (쿠키 기반) |
| DI | 없음 (직접 new) | `@Autowired` / 생성자 주입 |
| 트랜잭션 | 없음 | `@Transactional` |
| 보안 | `PageUtil.escape()` 수동 | Spring Security + 자동 이스케이프 |

### 그래도 내가 직접 판단해야 하는 것

1. **비즈니스 로직 자체** — "대출은 3권까지" 같은 규칙은 프레임워크가 못 만들어줌
2. **"이건 비동기로 해야 하나?"** — 이메일 발송 같은 오래 걸리는 건 @Async, 바로 응답해야 하는 건 동기로
3. **Singleton에 공유 상태가 있는가?** — @Service는 기본 Singleton, mutable 필드 두면 동시성 문제는 내 책임
4. **게임 루프 같은 주기적 작업** — @Scheduled로 선언은 가능하지만 설계는 직접
5. **실시간 통신 설계** — WebSocket 양방향은 "어떤 이벤트를 누구에게 보내냐" 직접 설계
6. **API 설계, 도메인 모델, 데이터 정합성**

### 사고방식의 전환

> 이전 (Library 프로젝트):
>   "소켓 어떻게 열지?" / "스레드 몇 개?" / "HTTP 헤더 파싱?" / "세션 ID 저장?"
>
> 앞으로 (Spring Boot / ASP.NET Core):
>   "이 API가 뭘 받고 뭘 돌려주지?"
>   "동시 접근하면 문제 되는 부분 있나?"
>   "동기로 충분한가, 비동기 필요한가?"
>   "이 데이터의 SSoT(Single Source of Truth)는 어디인가?"

### 핵심 깨달음

안 해본 사람은 프레임워크를 **"마법"**으로 쓰고, 해본 사람은 **"자동화"**로 쓴다. Library 프로젝트에서 아래 계층을 직접 만들어본 경험이 있기 때문에, @Transactional이 왜 필요한지, Singleton에 mutable 상태가 왜 위험한지를 감으로 알 수 있다.

## 면접 포인트

- **"프레임워크를 왜 쓰나요?"** → 인프라(소켓, 스레드, HTTP) 걱정 없이 비즈니스 로직에 집중. 검증된 코드 재사용으로 보안/안정성 확보.
- **"프레임워크 없이 개발해본 경험?"** → Library 프로젝트에서 ServerSocket, 스레드풀, HTTP 파싱, 세션 관리 직접 구현. 이 경험으로 프레임워크가 뭘 해주는지 이해하면서 사용.

## 개발연결

-  → 비동기 판단은 개발자 몫
-  → Singleton 공유 상태 보호는 개발자 책임
-  → API/도메인 설계 원칙
