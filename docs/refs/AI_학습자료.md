## 1. LLM이란 (Large Language Model)

### 한 줄 정의
텍스트를 입력받아 텍스트를 출력하는 확률 기계. "다음에 올 가장 그럴듯한 단어"를 계속 예측한다.

### 토큰
LLM은 글자가 아니라 **토큰** 단위로 읽는다. 대략 한국어 1글자 ≈ 1~2토큰, 영어 단어 ≈ 0.75토큰.
- "안녕하세요" = 약 5토큰
- 토큰이 중요한 이유: API 비용 = 토큰 수 × 단가

### 컨텍스트 윈도우
LLM이 한 번에 볼 수 있는 최대 토큰 수. 이 범위 밖은 **존재하지 않는 것**과 같다.

```
[시스템 프롬프트] + [대화 기록] + [참조 문서] + [내 질문] + [AI 답변]
= 전부 컨텍스트 윈도우 안에 들어가야 함
```

2026년 기준 주요 모델 컨텍스트:
- Groq llama3-70b: 8K~32K 토큰
- Claude: 200K 토큰
- GPT-4o: 128K 토큰

**핵심:** 컨텍스트 윈도우가 꽉 차면 앞 내용을 잊는다. 문서를 잘 정리해야 하는 이유.

## 2. API 호출 구조

LLM API는 단순하다. 보내고 받는 것.

```
입력 (Request)
├── system: "너는 인플루언서 분석 전문가야..."   ← 역할·규칙 정의
├── messages: [{"role": "user", "content": "..."}]  ← 대화 내용
└── model, temperature, max_tokens 등 파라미터

출력 (Response)
└── content: "분석 결과: ..."
```

**무상태(Stateless):** LLM API는 호출할 때마다 기억이 없다. 이전 대화를 기억시키려면 매번 대화 기록을 통째로 넣어줘야 한다. → 이게 컨텍스트가 쌓이는 이유.

## 3. 에이전트란

### 일반 LLM 호출 vs 에이전트

| 일반 LLM 호출 | 에이전트 |
|-------------|--------|
| 질문 1개 → 답변 1개 | 목표를 주면 스스로 계획·실행·검증 반복 |
| 사람이 다음 단계 결정 | AI가 다음 단계 스스로 결정 |
| 단발성 | 루프 구조 (Goal → Plan → Execute → Verify → Loop) |

### 에이전트 루프
```
목표 수신
→ 계획 수립 (어떤 순서로 할까)
→ 실행 (도구 사용 or 하위 에이전트 호출)
→ 결과 검증 (원하는 결과가 나왔나)
→ 완료면 종료 / 아니면 다시 루프
```

**중요:** 루프가 무한히 돌면 안 됨 → max_iteration 필수 설정.

## 4. 멀티 에이전트

### 왜 하나의 에이전트로 다 하지 않나?

하나의 에이전트에 모든 걸 시키면:
- 컨텍스트가 너무 길어져 품질 저하
- 역할이 섞여 아웃풋이 흐릿해짐
- 어디서 잘못됐는지 디버깅 불가

### 분리하면
- 각 에이전트가 **하나의 역할만** → 집중도 높음
- 독립 테스트 가능
- 교체·개선이 쉬움

### 우리 프로젝트 구조 (Orchestrator-Worker 패턴)
```
경영 에이전트 (Orchestrator)
— 목표 보유, 순서 결정, 결과 취합, 보고 판단

    ↓ 순서대로 호출

대상 분석 에이전트  →  경쟁 분석 에이전트  →  플랫폼 추천 에이전트  →  컨셉 기획 에이전트
(Worker)              (Worker)                (Worker)                  (Worker)
```

Workers는 서로를 모른다. 경영 에이전트만 전체를 안다.

## 5. 컨텍스트 전달 방식 3가지

에이전트 간 결과를 어떻게 넘길 것인가.

| 방식 | 설명 | 토큰 비용 | 정보 손실 |
|------|------|----------|---------|
| Full | 이전 결과 전체를 다음 에이전트에 통째로 넘김 | 높음 (기하급수적 증가) | 없음 |
| Structured | 경영 에이전트가 구조화된 객체로 필요한 것만 넘김 | 낮음 | 거의 없음 |
| Summarized | LLM이 요약해서 넘김 | 낮음 | 있음 (요약 손실) |

**우리 프로젝트 선택: Structured**
- 경영 에이전트가 각 에이전트 결과를 섹션별로 정리
- 다음 에이전트는 필요한 섹션만 참조
- 토큰 효율 + 정보 손실 없음

## 6. RAG (Retrieval-Augmented Generation)

### 한 줄 정의
LLM이 모르는 정보를 **외부 문서에서 찾아서** 컨텍스트에 넣어주는 방식.

```
질문 → 관련 문서 검색 → 문서를 컨텍스트에 추가 → LLM 답변
```

### 우리 프로젝트에 RAG 필요한가?
현재 MVP에서는 **불필요**.
- 입력: 대상자 기본 정보 (직접 입력)
- 참조: 마크다운 파일 (직접 Read)
- 벡터 DB, 임베딩 모델 셋업 비용 대비 효과 없음

Phase 2 이후 크리에이터 데이터가 쌓이면 그때 고려.

## 7. 컨텍스트 엔지니어링

### 왜 가장 중요한가

> `output_quality = context_quality × prompt_quality`

프롬프트를 아무리 잘 써도 컨텍스트가 부실하면 쓰레기 아웃풋이 나온다.
에이전트는 자기가 볼 수 있는 것만 안다. **문서 품질 = 에이전트 품질.**

### 좋은 컨텍스트 vs 나쁜 컨텍스트

| 나쁜 컨텍스트 | 좋은 컨텍스트 |
|------------|------------|
| "대상자: 김민수, 미용사" | "이름: 김민수 / 직업: 미용사 수습생(1년차) / 특기: 헤어컬러 / 성격: 내향적·섬세 / 타겟: 20대 여성" |
| 정보가 흩어져 있음 | 섹션이 명확히 구분됨 |
| 레이블 없음 | 레이블 + 값 구조 |
| 업데이트 시점 불명확 | 항상 최신 상태 유지 (Write-Through) |

### 우리 프로젝트에서 컨텍스트 엔지니어링 = 마크다운 문서 관리
- step 파일들이 에이전트의 컨텍스트 소스
- 파일이 부실하면 → 에이전트 아웃풋도 부실
- 문서를 먼저 잘 만드는 게 코드보다 중요

## 8. 잘 설계된 에이전트의 조건

### 왜 작은 모델 + 좋은 에이전트가 큰 모델을 이기는가

모델 크기보다 **에이전트에게 주는 정보의 질**이 더 중요하기 때문.
- 360도 완전한 컨텍스트를 가진 작은 모델 > 부실한 컨텍스트의 큰 모델
- 역할이 명확한 전문 에이전트 > 뭐든 다 하는 범용 에이전트

### 잘 설계된 에이전트의 5가지 조건

**① 역할이 단 하나**
"분석도 하고 기획도 하고 추천도 해" → 실패
"너는 오직 대상자 강점·약점·차별점만 분석한다" → 성공

**② 완료 조건이 명확**
에이전트가 언제 멈춰야 하는지 알아야 한다.
"분석해줘" X → "강점 3개, 약점 3개, 차별점 1개를 섹션으로 나눠 출력하면 완료" O

**③ 하면 안 되는 것이 명시됨**
대부분 빠뜨리는 부분. 없으면 에이전트가 범위를 벗어난다.
"generic한 조언 금지 / 대상자 이름 반드시 포함 / 추측 금지"

**④ 출력 형식이 고정됨**
다음 에이전트가 파싱할 수 있어야 한다.
형식이 매번 다르면 경영 에이전트가 결과를 읽지 못함.

**⑤ 참조할 문서가 잘 정리되어 있음**
에이전트는 자기가 볼 수 있는 것만 안다.
컨텍스트 문서가 부실하면 아무리 좋은 에이전트도 실패한다.
