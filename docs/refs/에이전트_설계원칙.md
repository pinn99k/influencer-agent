## 1. 시스템 프롬프트 필수 4섹션

모든 에이전트 시스템 프롬프트는 아래 4섹션을 반드시 포함한다. 하나라도 빠지면 작성 불가.

### [ROLE] 역할
- 에이전트가 누구인지 정의
- 단 하나의 역할만 부여 (복수 역할 금지)
- "어떻게 생각하는가"까지 포함

```
예시:
당신은 10년 경력의 인플루언서 마케팅 전문가입니다.
오직 크리에이터의 강점·약점·차별점 분석만 담당합니다.
칭찬보다 냉정한 분석을 우선합니다.
```

### [GOAL] 목표 + 완료 조건
- 무엇을 해야 하는지
- 언제 완료인지 (완료 조건 명시 필수)
- 완료 조건 없으면 에이전트가 멈추는 시점을 모름

```
예시:
목표: 대상자 기본 정보를 바탕으로 강점 3개, 약점 3개, 차별점 1개를 분석한다.
완료 조건: 위 항목이 모두 채워진 마크다운 섹션 출력 시 완료.
```

### [CONSTRAINTS] 하면 안 되는 것
- 대부분 빠뜨리는 섹션. 반드시 포함.
- 범위 이탈, generic 출력, 추측, 역할 침범 방지

```
예시:
- 대상자 정보에 없는 내용 추측 금지 → "정보 부족"으로 표시
- generic한 조언 금지 (예: "꾸준히 업로드하세요" 수준 금지)
- 플랫폼 추천 금지 (플랫폼 추천 에이전트 역할)
- 대상자 이름 반드시 포함
```

### [FORMAT] 출력 형식
- 다음 에이전트 또는 경영 에이전트가 파싱 가능한 구조로 고정
- 형식이 매번 달라지면 경영 에이전트가 결과를 읽지 못함
- 마크다운 섹션 고정 또는 JSON

```
예시 (마크다운 섹션 고정):
## 대상 분석 결과
**대상자:** {이름}

### 강점
1. ...
2. ...
3. ...

### 약점
1. ...
2. ...
3. ...

### 차별점
- ...
```

## 2. 컨텍스트 문서 작성 기준

에이전트가 참조하는 모든 파일은 아래 기준을 충족해야 한다.

### 구조 규칙
- 섹션 레이블 명확히 (`##`, `###` 사용)
- 레이블: 값 구조 사용 (`**직업:** 미용사`)
- 한 섹션 = 한 주제 (혼합 금지)
- 업데이트 날짜 상단 명시

### 내용 규칙
- 구체적 수치·사실 우선 (추상적 표현 금지)
- 빈 필드는 `미정` 또는 `정보 없음` 명시 (공백 금지)
- 에이전트가 판단해야 할 내용과 참조만 할 내용 구분

### 업데이트 규칙 (Write-Through)
- 단계 완료 즉시 관련 문서 업데이트. 세션 종료까지 기다리지 않는다.
- 이유: 세션은 휘발성. 문서가 유일한 상태 저장소.

## 3. 컨텍스트 전달 방식 — Structured 선택

### 우리 프로젝트 결정: Structured Context

경영 에이전트가 각 에이전트 결과를 구조화된 섹션으로 정리 → 다음 에이전트에 필요한 섹션만 전달.

```python
# 컨텍스트 객체 예시
context = {
    "대상자": {"이름": "김민수", "직업": "미용사", ...},
    "대상_분석": "## 대상 분석 결과\n...",      # 대상 분석 에이전트 결과
    "경쟁_분석": "## 경쟁 분석 결과\n...",      # 경쟁 분석 에이전트 결과
    "플랫폼_추천": "## 플랫폼 추천 결과\n...",  # 플랫폼 추천 에이전트 결과
}

# 컨셉 기획 에이전트 호출 시 전체 컨텍스트 전달
# 대상 분석 에이전트 호출 시 대상자 정보만 전달
```

### 선택 근거
- Full 방식 대비 토큰 80~90% 절감
- Summarized 방식과 달리 정보 손실 없음
- 경영 에이전트가 컨텍스트 크기를 통제 가능

## 4. 에이전트 루프 설계

### 기본 구조 (03_AGENT 기반)
```
Goal → Plan → Execute → Verify → Loop
```

### 필수 제약
- `max_iteration` 반드시 설정 (무한 루프 방지)
- 상태는 반드시 파일에 저장 (메모리 의존 금지 — 세션 휘발성)
- 각 iteration 결과 로깅

### 우리 프로젝트 경영 에이전트 루프
```python
MAX_RETRY = 1  # 에이전트당 최대 재시도 횟수

for agent in [대상분석, 경쟁분석, 플랫폼추천, 컨셉기획]:
    result = agent.run(context)
    
    if not validate(result):           # 출력 검증
        result = agent.run(context)    # 1회 재시도
        if not validate(result):
            report_to_chairman()       # 회장 보고 (보고 조건 5번)
            break
    
    check_chairman_report(result)      # 보고 조건 10개 체크
    context.update(agent.name, result) # 컨텍스트 누적
```

## 5. 출력 검증 레이어 (04_HARNESS 기반)

에이전트 결과는 반드시 검증 후 다음 단계로 넘긴다.

```
LLM 호출 → 출력 검증 → 통과 시 컨텍스트 누적
                      → 실패 시 재시도 (1회)
                               → 재실패 시 회장 보고
```

### 검증 항목
- 필수 섹션 존재 여부 (FORMAT 기준)
- 대상자 이름 포함 여부
- 최소 길이 충족 여부 (generic 방지)
- 금지 패턴 포함 여부 (CONSTRAINTS 기준)

## 6. 회장 보고 강제 레이어 (07_GUARDRAIL 기반)

보고 조건 10개는 경영 에이전트 시스템 프롬프트에 명시 + 코드 레벨에서도 체크.

### 코드 레벨 체크 대상 (자동 감지 가능)
- 조건 5: 에이전트 2회 실패 → `MAX_RETRY` 초과 시 자동 트리거
- 조건 9: 실행 단계 진입 → 리포트 저장 전 반드시 핑구 확인 요청

### 프롬프트 레벨 체크 대상 (경영 에이전트 판단)
- 조건 3: 전략 방향 전환 감지
- 조건 6: 에이전트 간 결과 충돌
- 조건 10: 기대치와 결과 괴리

### 보고 형식
```
## 회장 보고
보고 조건: {조건 번호 + 내용}
현재 상황: {무슨 일이 발생했는가}
선택지:
  A. {선택지 A}
  B. {선택지 B}
결정 요청: A 또는 B 선택 후 진행
```

## 7. 에이전트별 시스템 프롬프트 작성 순서

Step 4 기능명세 단계에서 에이전트별 프롬프트 작성 시 아래 순서 준수:

1. ROLE 정의 → 단 하나의 역할
2. GOAL + 완료 조건 정의
3. CONSTRAINTS 목록 작성 (빠뜨리기 쉬움 — 주의)
4. FORMAT 확정 → 다음 에이전트 입력 형식과 맞춤
5. 검증 항목 도출 (FORMAT 기반)
6. 핑구 검사 후 `prompts/{agent_name}.md` 저장
