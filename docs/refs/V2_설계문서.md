## 한 줄 정의

**V1**: 7개 필드 입력 -> 4개 에이전트 순차 실행 -> 마크다운 5개 출력 -> 끝 (1회성 리포트 생성기)
**V2**: CEO 대화형 인터뷰 -> 자율 판단 에이전트 -> 양방향 소통 -> 매니저 보고 -> 재분석 루프 (AI 기획사)

## 핑구 확정 결정사항 (세션 26 Q&A)

아래는 핑구가 직접 선택/답변한 내용이다. 추측이 아님. 변경 시 핑구 재확인 필수.

### 1. 대화형 인터뷰
| 항목 | 결정 | 핑구 원문 |
|------|------|----------|
| UI | CLI 먼저, 웹 채팅 나중에 | "둘 다 (CLI 먼저)" |
| 깊이 | 10턴+ 깊은 인터뷰 | "깊은 인터뷰 (10턴+)" |
| 판단 주체 | CEO가 "충분하다" 판단할 때까지 | 깊은 인터뷰 선택 시 암묵적 |

### 2. 에이전트 자율성
| 항목 | 결정 | 핑구 원문 |
|------|------|----------|
| 프롬프트 방식 | 역할 정의 + 사고 가이드 (세부 지시서 X) | "프롬프트가 아니라 역할 정의 및 가이드" |
| 에이전트 반환 | 결과물 + 질문 + 첨언 | "둘 다 + 추가 제안" |
| 소통 방식 | 결과와 함께 제출 (실시간 아님) | "결과와 함께 제출" |
| 파일 접근 | CEO가 전반적 파악, 요약 전달. 에이전트별 접근 범위 강제 | "CEO가 전반적인 걸 알고 산출물로 요약한걸 하위 에이전트가 읽고 필요 도메인 지식을 추가로 찾아서 읽고" |

### 3. 매니저 에이전트
| 항목 | 결정 | 핑구 원문 |
|------|------|----------|
| 역할 | 알림/보고 전담 (단방향) | "매니저 = 알림/보고 전담" |
| 전략 대화 | CEO가 직접 (매니저 아님) | 매니저 역할에서 제외 |
| 보고 채널 | MVP는 웹 대시보드, 나중에 봇 | "MVP는 웹, 나중에 봇" |

### 4. 구현 방식
| 항목 | 결정 | 핑구 원문 |
|------|------|----------|
| 코드 전략 | 핵심 인프라 유지 + 에이전트 레이어 재설계 | "너가 제안해봐 다 갈아엎으면 아쉽긴 한데, 그게 맞으면 그렇게 해" |
| API | OpenAI gpt-4o 확보 완료 | "OpenAI API 키 있음" |
| 테스터 | 핑구 본인 | "본인이 테스트" |

## 소통 구조

```
사용자 <---> CEO (전략 대화 + 초기 인터뷰)
사용자 <---  매니저 (알림/보고, 단방향)
CEO <------> 하위 에이전트 (명령 + 결과/질문/첨언)
CEO -------> 매니저 (보고 지시)
```

### 핵심 원칙
- 사용자는 CEO와 전략 대화, 매니저에게 알림 수신
- 하위 에이전트는 결과 + 질문 + 첨언을 CEO에게 제출 (실시간 아님)
- CEO가 질문을 3단계로 처리: 전략적 -> 사용자 에스컬레이션, 전술적 -> 자체 판단, 데이터 -> 컨텍스트 확인
- 에이전트별 파일 접근 범위 강제 (경쟁분석이 목표 파일 건드리면 안 됨)
- CEO가 전체 파악 -> 요약본 생성 -> 하위 에이전트에게 전달

## 아키텍처

### 레이어 구조

```
Layer 0 — Entry         main.py (인터뷰 모드 추가)
Layer 1 — Orchestration CEO (대화+판단+양방향소통)
                        InterviewEngine (신규 — multi-turn 대화)
                        ManagerAgent (신규 — 알림/보고 전담)
Layer 2 — Domain        Agent x 4 (AgentOutput 반환, 접근권한 적용)
                        AgentContext (신규 — 파일 접근 범위 관리)
Layer 3 — Infra         core/ (기존 유지 — llm_client, file_manager, config 등)
Layer 4 — Cross-cut     validators/ (기존 유지 — OutputValidator)
```

### 실행 흐름 (V2)

```
1. main.py --interview
2. InterviewEngine: CEO 대화형 인터뷰 (10턴+)
   -> 매 턴: LLM이 질문 생성 -> 사용자 답변 -> 정보 추출 -> 충분성 판단
   -> 충분 시: 추출된 정보 확인 요청 -> subject dict 반환
3. CEO._interpret_goal(context): 목표 해석 + plan.md 생성
4. CEO._agent_loop(context): PlanningDepartment에 위임
   -> 각 에이전트: AgentContext로 접근범위 필터링 -> 실행 -> AgentOutput 반환
   -> AgentOutput 파싱: content(검증) + questions(CEO 전달) + comments(참고)
   -> CEO._handle_agent_questions(): 3단계 판단
5. CEO._finalize(context): 최종 리포트 + 회장보고 조건 체크
6. CEO -> ManagerAgent: 완료 요약 + 주간카드 생성 지시
7. ManagerAgent -> EventEmitter -> 웹 대시보드 (or CLI 출력)
```

### 재분석 흐름

```
1. main.py --reanalyze --name {name}
2. CEO: 기존 산출물 + 성과기록 + 피드백 로드
3. CEO._decide_rerun(): 어떤 에이전트를 재실행할지 LLM 판단
4. PlanningDepartment.run_partial(): 필요 에이전트만 재실행
5. 수정된 산출물 + 후반 캘린더 생성
6. ManagerAgent: 변경사항 보고
```

### 1. AgentOutput — 결과물 + 메타데이터

```python
@dataclass
class AgentOutput:
    content: str           # 기존 마크다운 결과물
    questions: list[str]   # CEO에게 질문
    comments: list[str]    # 첨언/제안
    confidence: float      # 0.0~1.0
    metadata: dict         # 추가 정보

    @classmethod
    def from_raw(cls, text: str) -> "AgentOutput":
        # Lenient: JSON 블록 없으면 전체를 content로 처리
```

### 2. AgentContext — 접근 범위 강제

```python
AGENT_SCOPES = {
    "CEO":       {"read": ["*"], "knowledge": ["경영/"], "outputs": ["*"]},
    "대상분석":   {"read": ["대상자"], "knowledge": ["부서/기획본부/대상분석/"], "outputs": []},
    "경쟁분석":   {"read": ["대상자", "ceo_summary"], "knowledge": ["부서/기획본부/경쟁분석/"],
                  "outputs": ["01_대상분석.md"], "tools": ["serper"]},
    "플랫폼추천": {"read": ["대상자", "ceo_summary"], "knowledge": ["부서/기획본부/플랫폼추천/"],
                  "outputs": ["01_대상분석.md", "02_경쟁분석.md"]},
    "컨셉기획":   {"read": ["대상자", "ceo_summary"], "knowledge": ["부서/기획본부/컨셉기획/"],
                  "outputs": ["01_대상분석.md", "02_경쟁분석.md", "03_플랫폼추천.md"]},
}
```

### 3. 프롬프트 전환

```
현재 (~24,600 단어): [ROLE]+[GOAL]+[CONSTRAINTS]+[FORMAT]+[INPUT]+[THINKING]+[EXAMPLES]+[SELF-CHECK]

변경 후 (~10,000 단어):
  [ROLE]           — 역할 + 사고 방식 (축소)
  [GOAL]           — 완료 조건 (유지)
  [THINKING_GUIDE] — 사고 플로우 (신규)
  [ACCESS]         — 접근 가능 파일/도구 (신규)
  [OUTPUT]         — AgentOutput JSON + content 포맷
  [CONSTRAINTS]    — 핵심만 (금지어, 언어 규칙)

이동:
  [EXAMPLES]    -> knowledge/부서/기획본부/{agent}/예시.md
  [SELF-CHECK]  -> validator가 대체
  [FORMAT] 상세 -> knowledge/부서/기획본부/{agent}/출력형식.md

불변: [FORMAT] 섹션의 마크다운 헤더 (OutputValidator.RULES 매칭)
```

### 4. 매니저 에이전트

- AGENT_CLASSES에 등록하지 않음 (파이프라인 외부 서비스)
- CEO가 직접 호출
- EventEmitter를 통해 웹 대시보드에 알림 전달
- 산출물: 주간 실행 카드, 진행보고, 성과입력 요청, 완료 요약

### 5. Validator 불변

프롬프트를 역할 가이드로 전환하되, FORMAT 섹션의 마크다운 헤더는 그대로 유지.
OutputValidator.RULES가 깨지지 않도록 보장.
