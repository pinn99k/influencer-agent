## 0. 한 줄 선언

프레임워크 없이 맨손으로 LLM 워크플로우 시스템을 끝까지 구현했다.
이것을 1장 졸업으로 선언한다. 다음은 같은 손으로 자율 에이전트 하네스를 짓는다.

---

## 1. 목표 재정의 (이번에 명확해진 것)

지난 30세션은 하나의 목표를 좇는 것처럼 보였지만, 사실 두 목표가 섞여 있었다.

| 구분 | 목표 | 성격 |
|------|------|------|
| (A) | 자율적으로 판단·계획·실행하는 AI 기획사를 직접 만들어본다 | 학습·엔지니어링 (에이전트의 작동 원리 체득) |
| (B) | 실제 미용사 1명의 팔로워를 성장시킨다 | 제품·검증 |

이 둘은 최적화 방향이 다르다. (A)는 복잡도를 키울수록, (B)는 단순할수록 좋다.
지금까지는 (A)의 야망을 가지고 (B)에는 못 닿는 워크플로우를 지어왔다.

### 새 북극성 (확정)

> 특정 도메인(인플루언서 기획)에서 AI가 자율적으로 판단·계획·실행하는 시스템을,
> 프레임워크 없이 맨 API로 직접 구현하며 "에이전트가 어떻게 작동하는가"를 체득한다.

- **1장 (완료):** LLM 워크플로우 시스템 — 고정 파이프라인 + 품질 레이어 + 웹 UI
- **2장 (시작):** 자율 에이전트 하네스 — 도구 호출 + 루프 + 작업기억 + 가드레일
- **(B) 실사용 검증:** 워크플로우 버전으로 선행 가능하나, 본 여행의 1순위는 (A).
  비교 대상은 Claude Code가 아니라 "3주 전의 나(call_llm 한 번 부르고 끝)"다.

### 핵심 깨달음 — 워크플로우 vs 에이전트

- 업계 정의: **워크플로우** = 미리 정한 경로로 LLM을 엮음 / **에이전트** = LLM이 스스로
  경로를 결정(도구 사용 + 환경 피드백 + 루프).
- 지금까지 만든 것은 "잘 만든 워크플로우"였다. 결함이 아니라 정답에 가까웠다.
  문제는 그것을 "자율 CEO 에이전트"라고 불러 실물보다 큰 서사를 입힌 것.
- 에이전트가 되려면 4가지가 필요하다: (1) 도구 (2) 루프 (3) 환경 피드백 (4) 정지조건/가드레일.
  현재 1~2는 씨앗만, 3~4는 사실상 없음. 그래서 2장의 본체가 바로 이 4가지다.

## 2. 프로젝트 종합 정리

### 2-1. 무엇을 만들었나 (기능)

대화형 인터뷰(CEO 첫 상담) -> CEO 목표 해석 -> 4개 워커 순차 실행
(대상분석 / 경쟁분석 / 플랫폼추천 / 컨셉기획) -> 품질 게이트(검증기 + LLM 저지)
-> CEO LLM 종합 리포트(6섹션) -> 실행계획 파서(4주 캘린더/다음액션/KPI)
-> 관리중심 웹 대시보드 -> 성과 입력 -> 재분석 루프 -> 매니저 알림.

### 2-2. 어떤 구조인가 (아키텍처)

- 빌드 레이어(docs/) / 런타임 레이어(src/) 분리 (ADR 014).
- src/ = core(인프라) / agents(CEO+매니저+워커4) / departments(기획본부) /
  validators / api(FastAPI+SSE) / prompts(영문경로) / knowledge(재귀로딩) / outputs.
- 패턴: OOP(SRP/LSP/DIP/OCP), Agent Registry(SSoT), DAG 스케줄러,
  AgentOutput/AgentContext(구조화 핸드오프), MVC 프론트(model/actions/api/views).

### 2-3. 기술 스택

- Python 3.11+, 프레임워크 없음 (의도적 — 프리미티브 학습).
- 생성 = gpt-4o (워커4 + CEO판단 + 인터뷰), 품질저지 = Groq, 폴백 = Gemini.
- 검색 = Serper. DB 없음(마크다운 파일 기반). UI = CLI + 웹.
- 테스트 268 pass.

### 2-4. 방향 (이번 세션 전환)

- 고정 파이프라인(워크플로우) -> 도구 호출 루프(에이전트)로 전환 결정.
- 재사용 자산은 "지식과 관례"지 "오케스트레이션 코드"가 아님을 확인.
- src/의 인프라(call_llm, AgentContext, EventEmitter, file_manager, state_manager)는
  하네스의 차대(chassis)로 그대로 재사용. 비어있는 것은 엔진(도구+루프+기억).

### 종합

- **워크플로우 MVP로서: 90/100** — 좋은 엔지니어링 데모. 졸업 자격 충분.
- **자율 에이전트 하네스로서: 35/100** — 차대 완성, 엔진 미착수.
- **검증된 제품으로서: 미시작** — (B) 실사용 0.

한 줄 평: 과정과 엔지니어링은 상위권, 자율성과 실사용 검증은 미착수.
"통과 != 검증" 규율을 코드엔 적용했으나 정작 자기 서사엔 늦게 적용했다.

## 4. 자체 피드백

### 잘한 것

1. 프레임워크 없이 끝까지 손으로 — LLM 시스템의 프리미티브를 실제로 이해.
2. 규율이 최상위 — ADR/loop/소크라테스 검증/Write-Through. 대부분의 포트폴리오에 없는 자산.
3. 컨텍스트 엔지니어링 자산 축적 — knowledge/ 도메인 프레임워크가 진짜 금.
4. 자기 반성 능력 — 확장_로드맵에서 "V2 완료가 아니다", "조용한 폴백" 스스로 발견.

### 못한 것 / 빠진 함정

1. **서사가 실물을 초과** — "자율 CEO"라 불렀지만 선형 파이프라인이었다.
2. **검증 안 된 핵심 위에 증축** — 워커 품질은 쟀으나 CEO 판단 품질은 안 쟀고, 그 위에 UI/부서를 올림.
3. **두 목표(A/B) 혼재** — 어느 쪽도 완결 못 함. (A)는 자율성 미구현, (B)는 실사용 0.
4. **조직 은유의 유혹** — CEO/부서/매니저 은유가 사용자에 안 닿는 기능(감독층, 브리핑)을 짓게 함.
5. **도구/루프 없이 "에이전트"라 명명** — 용어 혼동이 30세션 내내 방향을 흐림.

### 핵심 교훈

워크플로우와 에이전트를 구분하지 못한 채 "에이전트"를 목표로 워크플로우를 지었다.
이제 그 구분이 명확하다. 2장은 그 구분 위에서 진짜 하네스를 짓는다.

## 7. 후속 갱신 (세션 31 — cross-verify 종합)

목표·전달·성공 기준이 확정됐다. 본 선언의 상위 권위는 2장 폴더 문서다.

- **목표 1순위:** "멀티 에이전트 구현"이 아니라 "실제 미용사 1명 성장 + 포폴 증명". 실사용 검증 우선.
- **전달 모델:** 컨시어지 — 핑구가 운영, 미용사는 서비스 수혜(배포 불필요). 1장으로 즉시 시작 가능.
- **성공 기준(2축):** (실사용) 팔로워 0->300+ / (포폴) 측정 가능한 실험 + 의사결정 로그 +
  최소 자율 에이전트의 실데이터 조정 기록. 팔로워 미달이어도 데이터로 설명 가능하면 부분 성공.
- **우선순위:** 측정·실험 설계(1) -> 컨시어지 온보딩(2) -> 최소 자율 루프(3) -> 트랙0(4).
- **핵심 보정(GPT):** "데이터가 생기면 검증된다"는 틀림. 측정 가능한 실험 설계가 검증을 만든다.
- 상세: docs/workflow/2장_에이전트하네스/{00 개요,01 로드맵,02 설계스케치,03 측정실험설계}.
