## 1. 레이어 아키텍처

```
┌─────────────────────────────────────────────────────┐
│  Layer 0   main.py                                  │  입력 수집 + 객체 조립 (DI 진입점)
├─────────────────────────────────────────────────────┤
│  Layer 1   agents/ceo.py           CEO              │  오케스트레이션 (판단·지휘·상태·보고)
├─────────────────────────────────────────────────────┤
│  Layer 2   agents/*_analyst.py     BaseAgent 상속   │  도메인 에이전트 (각 직무 단일 책임)
├─────────────────────────────────────────────────────┤
│  Layer 3   core/                   모듈 함수         │  인프라 유틸 (LLM·검색·파일 I/O)
├─────────────────────────────────────────────────────┤
│  Layer 4   validators/             OutputValidator  │  Cross-cutting: 검증 규칙
└─────────────────────────────────────────────────────┘
```

**의존 방향 (단방향 강제):**
```
main.py → CEO → [agents, validators, core]
agents/ → core/
validators/ → (의존 없음)
core/ → (의존 없음)
```
레이어 역참조 금지. `core/`가 `agents/`를 알면 설계 위반.

## 2. 클래스 상속 구조

```
object
│
├── ABC
│   └── BaseAgent                       agents/base_agent.py
│       ├── SubjectAnalystAgent          agents/subject_analyst.py
│       ├── CompetitionAnalystAgent      agents/competition_analyst.py  ← run() 오버라이드
│       ├── PlatformRecommenderAgent     agents/platform_recommender.py
│       └── ConceptPlannerAgent          agents/concept_planner.py
│
├── CEO                                 agents/ceo.py          (상속 없음, 독립 클래스)
├── FileManager                         core/file_manager.py   (상속 없음)
│
└── OutputValidator                     validators/output_validator.py
    └── ValidationResult  (@dataclass)  (OutputValidator 내부에서 생성·반환)
```

**모듈 수준 함수 (클래스 없음):**
```
core/llm_client.py      call_llm(provider, model, system, user) → str
core/serper_client.py   search(query) → list[dict]
```
→ 상태가 없는 유틸은 클래스 불필요. 모듈 함수가 더 명확.

## 3. 클래스별 상세

### 3-1. `BaseAgent` (ABC)

**책임:** 4개 도메인 에이전트의 공통 계약 + 공통 실행 흐름 제공

**적용 패턴:** Template Method
```
__init__()
  ├── _load_template()   ← prompts/{prompt_file}.md 로드
  └── _load_knowledge()  ← knowledge/{knowledge_dir}/*.md 병합 로드 (ADR 014)

run() ← 알고리즘 골격 (BaseAgent가 정의)
  ├── build_prompt()     ← 추상 메서드 (서브클래스 구현 강제)
  └── call_llm()         ← 공통 (BaseAgent)
```

**클래스 속성 (서브클래스에서 오버라이드 필수):**
| 속성 | 타입 | 역할 |
|------|------|------|
| `display_name` | str | 한글 표시명. AGENT_KEY_MAP·OutputValidator.RULES 키와 반드시 일치 |
| `prompt_file` | str | `prompts/{prompt_file}.md` 경로. `"부서/기획본부/대상분석"` 형식 (ADR 014) |
| `knowledge_dir` | str | `knowledge/{knowledge_dir}/` 경로. `""` 이면 지식 없음 (ADR 014) |

**추상 메서드 (미구현 시 인스턴스 생성에서 TypeError):**
| 메서드 | 반환 | 설명 |
|--------|------|------|
| `build_prompt(context)` | str | 에이전트별 user_prompt 조합 방식 |
| `get_context_keys()` | list[str] | 이 에이전트가 필요로 하는 context 키 목록 |

**Public API:**
```python
agent.run(context: dict) -> str
```
외부(CEO)에서 호출하는 유일한 메서드. 내부 흐름 외부 노출 없음.

### 3-2. `SubjectAnalystAgent` / `PlatformRecommenderAgent` / `ConceptPlannerAgent`

**책임:** 각자 담당 직무 하나의 분석 결과 생성

**구현 요구사항:** `build_prompt()` + `get_context_keys()` 두 메서드만 구현하면 됨.
나머지(LLM 호출, 프롬프트 파일 로딩)는 BaseAgent 자동 상속.

```python
class SubjectAnalystAgent(BaseAgent):
    display_name  = "대상분석"
    prompt_file   = "부서/기획본부/대상분석"    # prompts/부서/기획본부/대상분석.md (ADR 014)
    knowledge_dir = "부서/기획본부/대상분석"    # knowledge/부서/기획본부/대상분석/*.md

    def get_context_keys(self) -> list[str]:
        return ["대상자"]

    def build_prompt(self, context: dict) -> str:
        payload = {k: context[k] for k in self.get_context_keys()}
        return json.dumps(payload, ensure_ascii=False, indent=2)
```

| 클래스 | display_name | prompt_file / knowledge_dir | get_context_keys() |
|--------|-------------|----------------------------|-------------------|
| `SubjectAnalystAgent` | `"대상분석"` | `"부서/기획본부/대상분석"` | `["대상자"]` |
| `PlatformRecommenderAgent` | `"플랫폼추천"` | `"부서/기획본부/플랫폼추천"` | `["대상자", "대상_분석", "경쟁_분석"]` |
| `ConceptPlannerAgent` | `"컨셉기획"` | `"부서/기획본부/컨셉기획"` | `["대상자", "대상_분석", "경쟁_분석", "플랫폼_추천"]` |

### 3-3. `CompetitionAnalystAgent` ← 주의: `run()` 오버라이드

**책임:** 경쟁 분석 (유사 채널 포지셔닝 + 시장 공백) + Serper 웹 검색 직접 호출

**BaseAgent와 차이점:** Serper 결과를 build_prompt에 주입해야 하므로 `run()` 자체를 오버라이드.

```python
def run(self, context: dict) -> str:             # 부모 run() 교체
    query = f"{context['대상자']['직업']} {context['대상자']['특기']} 유튜버"
    search_results = serper_client.search(query)  # 직접 호출 (ADR 010)
    user_prompt = self.build_prompt(context, search_results)
    return call_llm(DEFAULT_PROVIDER, DEFAULT_MODEL, self.system_prompt, user_prompt)

def build_prompt(self, context: dict, search_results: list = None) -> str:  # 시그니처 확장
    payload = {k: context[k] for k in self.get_context_keys()}
    payload["serper_results"] = search_results or []
    return json.dumps(payload, ensure_ascii=False, indent=2)
```

**⚠️ 설계 검토 포인트:**
`build_prompt(context, search_results)` 시그니처가 부모의 `build_prompt(context)`와 다름.
LSP 관점에서 반환값과 기본 계약(str 반환)은 동일하나, BaseAgent.run()이 `build_prompt(context)`로만 호출하도록 설계됐기 때문에 **run() 전체를 오버라이드**하는 방식 선택.
→ 대안으로 Composition(SerperMixin) 또는 `__init__`에 serper_client 주입 방식도 가능. 현재는 단순성 우선.

### 3-4. `CEO`

**책임:** 오케스트레이션 전체 (전략 수립 → 에이전트 루프 → 검증 → 상태 관리 → 보고 → 인수인계)

**생성자 파라미터:**
```python
CEO(file_manager: FileManager, dry_run: bool = False)
```
- `FileManager` DI 주입 (main.py에서 조립, CEO 내부에서 생성 금지)
- `dry_run=True`: Spiral 0-A 모드 — API 호출 없이 프롬프트 파일만 저장

**Public API:** `run(context: dict) -> None` 하나뿐.

**내부 구조 (Phase 분리):**
```
run(context)
 ├── _interpret_goal(context)     Phase 1: 목표 해석 → plan.md + state.md 초기화
 ├── _agent_loop(context)         Phase 2: while 루프 (LLM 재판단)
 │    └── _run_one_agent(name, context)
 │         ├── _assert_state_saved()
 │         ├── _update_agent_state(name, "🔄", start=True)
 │         ├── agent.run(context)
 │         ├── OutputValidator.validate()      ← 1단계 자동 검증
 │         ├── _judge_quality()               ← 2단계 CEO LLM 판단
 │         ├── context[AGENT_KEY_MAP[name]] = result  ← OK시만 누적
 │         ├── _update_plan()
 │         └── _check_llm_chairman_conditions()
 └── _finalize(context)           Phase 3: 승인 게이트 → 최종 저장 → 인수인계

```

**주요 상수:**
```python
MAX_RETRY = 1       # 에이전트당 최대 재작업 횟수
AUTO_CONDITIONS  = {5, 9}            # 코드 자동 감지 보고 조건
LLM_CONDITIONS   = {2,3,4,6,7,8,10} # CEO LLM 판단 보고 조건

AGENT_KEY_MAP = {         # display_name → context 키 매핑 (SSoT)
    "대상분석":  "대상_분석",
    "경쟁분석":  "경쟁_분석",
    "플랫폼추천": "플랫폼_추천",
    "컨셉기획":  "컨셉_기획",
}

# CEO 판단 유형 → knowledge/ 경로 매핑 (ADR 014 — 전체 로드 금지, 판단 시점마다 선택 주입)
KNOWLEDGE_MAP = {
    "전략수립": ["경영/01_전략경영", "경영/06_운영관리"],
    "품질판단": ["경영/05_품질관리", "경영/08_의사결정"],
    "보고판단": ["경영/07_리스크관리", "경영/09_커뮤니케이션"],
    "ROI판단":  ["경영/03_재무관리", "경영/04_조직인사관리"],
}
```

**knowledge 로드 전략 (ADR 014):**
- 하위 에이전트: `__init__` 시 `knowledge_dir` 경로의 .md 파일 전부 1회 로드 → `system_prompt`에 병합
- CEO: 판단 유형(`전략수립`, `품질판단` 등)에 따라 `_load_knowledge_for(judgment_type)` 호출 → 필요한 카테고리만 그때그때 로드
- 이유: Groq llama3-70b 컨텍스트 한계(8K~32K). 경영 지식 9카테고리 전체 로드 시 토큰 초과 위험

**Write-Through 패턴:**
```python
def _update_state(self, key, value):
    self.state[key] = value
    self.fm.save_state(self.state)   # 즉시 파일 저장 (메모리 의존 금지)
```

**⚠️ 설계 검토 포인트:**
CEO가 오케스트레이션 관련 여러 관심사(전략·루프·검증·상태·보고·인수인계)를 담당하므로 SRP 관점에서 "느슨한 적용" 상태.
→ 의도적 선택: 오케스트레이터는 본질적으로 여러 관심사 연결이 역할. 대신 메서드 레벨에서 단일 책임 분리.
→ 규모가 커지면 `CEOValidator`, `CEOReporter`, `CEOStateManager`로 분리 가능.

### 3-5. `FileManager`

**책임:** 모든 파일 I/O의 단일 진입점 (SSoT). 다른 모듈에서 직접 `open()` 금지.

**생성자:** `FileManager(influencer_name: str)` → `outputs/{influencer_name}/` 기준 경로 고정

**Public API:**
| 메서드 | 저장 위치 | 버전 관리 |
|--------|----------|----------|
| `save_output(agent_name, content)` | `산출물/` | ✅ `.versions/` 백업 |
| `save_raw(agent_name, content)` | `.system/agents/{name}/raw_output.md` | ✅ |
| `save_validation(agent_name, result)` | `.system/agents/{name}/validation.md` | ❌ |
| `save_rework(agent_name, instruction)` | `.system/agents/{name}/rework.md` | ❌ |
| `save_state(state: dict)` | `.system/ceo/state.md` | snapshots/ |
| `save_plan(content)` | `.system/ceo/plan.md` | snapshots/ |
| `save_final_report(content)` | `산출물/최종리포트.md` | ✅ |
| `save_briefing(condition_no, content)` | `.system/briefings/briefing_{datetime}_{no}.md` | ❌ (날짜로 구분) |
| `save_prompt_output(agent_name, content)` | `.system/prompts/` | ❌ (Spiral 0-A 전용) |
| `save_handover(content)` | `인수인계/{datetime}_인수인계.md` | ❌ (날짜로 구분) |
| `save_snapshot()` | `.system/ceo/snapshots/{datetime}/` | — |
| `append_log(event: dict)` | `.system/logs/{날짜}.jsonl` | append only |

**Private:**
- `_versioned_write(path, content)`: 덮어쓰기 전 `.versions/v{N}` 백업 후 저장
- `_ensure_dirs()`: `__init__` 시 전체 폴더 구조 자동 생성

### 3-6. `OutputValidator` + `ValidationResult`

**책임:** 1단계 자동 검증 — 에이전트 결과의 형식·구조 규칙 체크

**`@classmethod` 선택 이유:** 인스턴스 상태 없음. `OutputValidator.validate(...)` 직접 호출.
`@staticmethod`가 아닌 이유: `cls.RULES` 참조로 서브클래스 오버라이드 여지 확보.

**`RULES` dict (SSoT — 에이전트 추가 시 여기만 수정):**
```python
RULES = {
    "대상분석":  { "required_sections": [...], "required_keywords": [...],
                  "count_checks": {"강점": 3, "약점": 3}, "min_length": 300 },
    "경쟁분석":  { ... "min_length": 400 },
    "플랫폼추천": { ... "min_length": 200 },
    "컨셉기획":  { ... "count_checks": {"아이디어": 5}, "min_length": 600 },
}
```
⚠️ **키 = `agent.display_name`** — AGENT_KEY_MAP, CEO 내부와 반드시 일치해야 함.

**`ValidationResult` (dataclass):**
```python
@dataclass
class ValidationResult:
    passed: bool
    agent_name: str
    failed_rules: list[str] = field(default_factory=list)
    retry_required: bool = False
```

**검증 순서:**
1. required_sections 존재 여부
2. subject_name 포함 여부
3. required_keywords 포함 여부
4. count_checks (있는 경우)
5. min_length

## 4. OOP 원칙 적용 현황

### SOLID 점검

| 원칙 | 적용 위치 | 평가 |
|------|----------|------|
| **SRP** (단일 책임) | FileManager(I/O만), OutputValidator(검증만), 도메인 에이전트 4개(각 직무만) | ✅ |
| **SRP** | CEO (전략+루프+검증+상태+보고+인수인계) | ⚠️ 느슨한 적용 — 의도적 선택 |
| **OCP** (확장 개방, 수정 폐쇄) | BaseAgent: 새 에이전트 추가 시 base_agent.py 수정 불필요 | ✅ |
| **OCP** | OutputValidator.RULES + AGENT_KEY_MAP: 에이전트 추가 시 두 곳 수정 | ⚠️ |
| **LSP** (치환 가능) | BaseAgent 서브클래스 전부 `run() → str` 계약 유지 | ✅ |
| **LSP** | CompetitionAnalystAgent.build_prompt() 시그니처 확장 | ⚠️ 부모와 다름 (하지만 run()이 직접 호출하지 않으므로 실용적 문제 없음) |
| **ISP** (인터페이스 분리) | BaseAgent 추상 메서드 2개 — 최소한으로 유지 | ✅ |
| **DIP** (의존 역전) | CEO ← FileManager DI (main.py에서 주입) | ✅ |
| **DIP** | BaseAgent → call_llm() 모듈 직접 import | ⚠️ 인터페이스 없음 (Mock 교체 시 monkeypatch 필요) |

### 캡슐화

| 클래스 | Public | Private (`_`) |
|--------|--------|--------------|
| CEO | `run()` | `_interpret_goal`, `_agent_loop`, `_run_one_agent`, `_decide_next`, `_judge_quality`, `_finalize`, `_report_to_chairman`, `_update_state`, 등 |
| FileManager | save_*(), append_log() 전부 | `_versioned_write`, `_ensure_dirs` |
| BaseAgent | `run()` | `_load_template` |
| OutputValidator | `validate()` (classmethod) | — |

CEO의 유일한 public API = `run(context)`. `main.py`는 CEO 내부 구조를 알 필요 없음.

### 추상화

| 추상화 대상 | 방법 | 효과 |
|------------|------|------|
| LLM provider | `call_llm(provider, model, ...)` 단일 시그니처 | Groq→GPT 교체 = 상수 1줄 변경 |
| 에이전트 실행 | `agent.run(context)` 단일 인터페이스 | CEO는 어떤 에이전트인지 몰라도 됨 |
| 파일 경로 | `FileManager` 메서드 | 경로 구조 변경 시 FM만 수정 |
| 검증 규칙 | `RULES` dict | 규칙 추가/수정 = dict만 변경 |
| 프롬프트 | `.md` 파일 분리 | 코드 변경 없이 프롬프트 튜닝 |

## 5. 적용된 디자인 패턴

| 패턴 | 적용 위치 | 설명 |
|------|----------|------|
| **Template Method** | `BaseAgent.run()` | 알고리즘 골격 고정, `build_prompt()` 서브클래스 위임 |
| **Dependency Injection** | `CEO(file_manager)` | main.py가 FileManager 생성 후 주입. 테스트 시 Mock 교체 가능 |
| **Null Object 유사** | `context` dict 초기화 | 모든 키를 None으로 초기화 → NullPointerError 방지 |
| **Write-Through** | `CEO._update_state()` | state 변경 즉시 파일 저장 (세션 크래시 복구용) |
| **Factory-like** | `CEO.__init__` agents dict | `display_name`을 키로 에이전트 인스턴스 딕셔너리 구성 |
| **Classmethod Singleton-like** | `OutputValidator.validate()` | 상태 없는 검증 로직, 인스턴스 생성 없이 직접 호출 |
| **Strategy-like** | `AGENT_KEY_MAP` | display_name → context key 매핑으로 에이전트별 저장 위치 분기 |

## 6. 데이터 흐름 (context dict)

**초기화 (main.py):**

```python
context = {

               "타겟연령대": str, "SNS경험": str, "목표": str},
    "대상_분석":   None,  # SubjectAnalystAgent 완료 후 채워짐
    "경쟁_분석":   None,  # CompetitionAnalystAgent 완료 후
    "플랫폼_추천": None,  # PlatformRecommenderAgent 완료 후
    "컨셉_기획":   None,  # ConceptPlannerAgent 완료 후

```

**에이전트별 수신 키 (get_context_keys() 기준):**

## 7. 검증 2단계 흐름

```
agent.run(context)
        │
        ▼
[1단계] OutputValidator.validate(display_name, output, subject_name)
        │                     규칙 기반 자동 체크 (형식·구조)
        ├── PASS ──────────────────────────────────────────────────┐
        │                                                          ▼
        └── FAIL → save_rework() → retry_count += 1    [2단계] CEO._judge_quality()
                    ├── retry_count <= MAX_RETRY → 재실행            │   LLM 품질 판단
                    └── retry_count > MAX_RETRY → 회장보고(조건5)    ├── OK → context 누적
                                                                    └── NG → save_rework()
                                                                         → retry_count += 1
                                                                         → MAX_RETRY 초과 시 조건5
```

## 10. 설계 검토 포인트 (수정 가능 영역)

### ⚠️ 1. CEO 클래스 비대화

현재 CEO가 담당: 전략 수립 / 에이전트 루프 / 1단계 검증 호출 / 2단계 LLM 판단 / 상태 관리 / 회장 보고 / 인수인계

MVP 규모에서는 허용 범위지만, Spiral 2 이후 회장 보고 로직이 복잡해지면 분리 고려:
```
CEOOrchestrator  ← 루프 + 에이전트 호출
CEOReporter      ← 회장 보고 조건 감지 + 브리핑 생성
CEOStateManager  ← plan.md + state.md 관리
```

### ⚠️ 2. CompetitionAnalystAgent.build_prompt() 시그니처 불일치

부모: `build_prompt(self, context: dict) -> str`
자식: `build_prompt(self, context: dict, search_results: list = None) -> str`

현재는 run()을 통째로 오버라이드해서 실용적 문제는 없음.
더 엄격하게 하려면:
```python
# 대안 A: Composition
class CompetitionAnalystAgent(BaseAgent):
    def __init__(self):
        super().__init__()
        self._serper = serper_client  # 주입 가능하게 분리

# 대안 B: build_prompt에 **kwargs 추가
def build_prompt(self, context: dict, **kwargs) -> str:
    search_results = kwargs.get("search_results", [])
```

### ⚠️ 3. AGENT_KEY_MAP 중복 관리

에이전트를 추가하면 3곳을 동시에 맞춰야 함:
1. 에이전트 클래스 `display_name`
2. `CEO.AGENT_KEY_MAP`
3. `OutputValidator.RULES`

→ 이상적으로는 에이전트 클래스에 `context_key`와 `validation_rules`를 함께 정의하고, CEO와 Validator가 그것을 읽는 구조. 현재는 단순성 우선으로 분리.

### ⚠️ 4. context dict mutable 공유

`context`가 pass-by-reference로 CEO와 에이전트 사이를 오감.
에이전트가 context를 직접 수정하지 않는 걸 관례로만 강제 중. 엄격하게 하려면:
```python
# 에이전트 호출 시 필요한 키만 복사본 전달
agent_context = {k: context[k] for k in agent.get_context_keys()}
result = agent.run(agent_context)   # 원본 context 수정 불가
```

### ✅ 5. dry_run 분기 (Spiral 0-A)

`CEO.dry_run=True`일 때 `_interpret_goal()`과 `_agent_loop()` 내부에서 API 호출 대신 프롬프트 파일 저장.
→ 깔끔한 분리는 아님. 대안으로 Strategy 패턴 (`DryRunStrategy` vs `LiveStrategy`)이 있으나 MVP 오버엔지니어링.
