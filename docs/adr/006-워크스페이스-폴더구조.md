# ADR 006 — 워크스페이스 폴더 구조 확정
date: 2026-05-12
status: 확정
세션: 4

---

## 컨텍스트

outputs/ 폴더 구조를 설계하면서 두 가지 충돌하는 관점이 존재했다.
- 시스템 중심: 에이전트가 파일을 읽고 쓰기 편한 구조
- 사용자 중심: 핑구가 파일 탐색기에서 직접 열고 작업하기 편한 구조

또한 에이전트 산출물이 반복 실행으로 계속 업데이트될 때 이전 버전을 어떻게 보관할지 결정이 필요했다.

---

## 결정

### 폴더 구조

```
outputs/{인플루언서명}/
├── 산출물/          ← 핑구가 보는 최종 결과 (사람 중심)
│   └── .versions/  ← 버전 보관
├── 인수인계/        ← 세션별 인수인계
└── .system/         ← 시스템 내부 (핑구 직접 수정 금지)
    ├── ceo/         ← plan.md, state.md, snapshots/
    ├── agents/      ← 에이전트별 raw_output, validation, rework, .versions/
    ├── briefings/   ← 회장 보고 이력
    ├── prompts/     ← Spiral 0-A 전용
    └── logs/        ← UI 스트리밍용 .jsonl
```

### 버전 관리
- 방식: `.versions/` 서브폴더, `_{v숫자}` suffix
- 대상: 산출물/ 전체 + .system/agents/ raw_output.md
- 규칙: 덮어쓰기 전 자동 복사 → 최신은 항상 이름 고정

### 로그 형식
- `.system/logs/{날짜}.jsonl` — `{"timestamp", "agent", "event", "status"}`
- Spiral 3 웹 UI가 스트리밍으로 읽는 용도

---

## 고려한 대안

| 대안 | 제외 이유 |
|------|-----------|
| 시스템 중심 구조 (reports/, states/ 등) | 핑구가 파일 탐색기에서 직접 작업하는 시나리오 무시 |
| 날짜 suffix 버전 관리 | 최신 파일 이름이 계속 바뀌어 참조 어려움 |
| Git 기반 버전 관리 | 핑구가 git CLI 사용해야 하는 부담 |
| .meta/ 폴더 | .system/ 대비 의미 불명확 |

---

## 근거

핑구가 직접 파일 탐색기에서 열고 편집하는 실제 사용 시나리오 (미용사 가이드, 편집자 가이드 등 폴더 구조 참고) 기반.
시스템 파일은 .system/으로 격리해 혼란 방지.
플랫폼·컨셉 기획은 반복 실행이 잦으므로 버전 관리 필수.
