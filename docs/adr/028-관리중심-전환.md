# ADR 028 — 관리 중심 전환 (종합 리포트 + 실행 관리 하이브리드)
date: 2026-06-01 (세션 29)
status: accepted

## Context
세션 29에서 Claude Design v3 UI 통합 후 핑구가 시스템을 처음 직접 사용.
진단: (1) 분석 위주라 사용자가 다음 행동을 모름 (2) 최종 리포트가 산출물
단순 연결(report_builder.py:50, LLM 미사용)이라 가치가 낮음 (3) UI가 결과물
뷰어라 "관리"로 느껴지지 않음. 실전적용_로드맵의 명제와 충돌:
"산출물을 받은 미용사가 '그래서 내일 뭘 찍지?'에 답을 못 주면 시스템은 실패다."

핵심 사실: 실행계획(4주 캘린더·촬영가이드·해시태그)은 이미 컨셉기획이 생성하고
매니저가 주간 카드로 쪼갠다. 즉 기능은 있으나 (a) 종합 부재 (b) UI 노출 부재.

## Decision
제품 정체성을 "분석 도구"에서 "분석→전략→실행관리 하이브리드"로 고정한다.
1. CEO 최종 리포트를 LLM 종합 보고서로 재설계 (6섹션: 결론/핵심결정/강점기회/
   4주 로드맵/지금 할 일/KPI). 단순 이어붙이기는 폴백으로만 유지.
2. 실행계획을 결정론적 파서로 구조화하고 `GET /api/plan/{name}` 로 노출.
3. 대시보드를 관리 중심으로 재편 — 할 일/로드맵/다음 액션 전면, 분석4는 근거자료.

구현은 코덱스, UI 뷰는 Claude Design. 본 ADR과 명세 문서가 단일 기준.

## Consequences
+ 사용자가 화면에서 즉시 "다음 행동"을 안다 (실전 적용 가능성 상승).
+ 최종 리포트가 의사결정 가능한 문서가 된다.
+ plan API 로 향후 진행률/체크 추적·재분석 트리거 확장 용이.
- LLM 종합 1회 추가 → 비용·시간 소폭 증가.
- final_report 포맷이 plan_extractor 와 결합 → 헤더 계약을 깨면 파서 회귀.
  완화: 컨셉기획 `#### Week N` 및 리포트 섹션 헤더를 명세에 계약으로 고정.

## Alternatives 기각
- 실행 코치 전용(분석 제거): 분석은 전략 근거 → 제거 시 리포트 신뢰도 하락. 기각.
- 전략 컨설턴트 전용(실행 약화): 핑구의 1번 진단(행동 부재)을 해결 못 함. 기각.
- LLM 없이 템플릿 종합: 대상자 맞춤 종합 불가 → 결국 또 generic. 기각.
