## 발생 이슈 및 해결

### 이슈 1 — llama3-70b-8192 모델 decommissioned

**증상:** Groq API 400 Bad Request — "model has been decommissioned"

**원인:** CLAUDE.md에 명시된 기본 모델 `llama3-70b-8192`가 Groq에서 서비스 종료됨.

**해결:** `core/llm_client.py`의 `DEFAULT_MODEL`을 `llama-3.3-70b-versatile`로 교체.

**교훈:** Groq 모델명은 서비스 종료가 빠름. `llm_client.py`가 SSoT이므로 한 곳만 수정하면 전체 반영됨 — 설계가 올바름.

### 이슈 2 — Write 툴로 한글 경로 파일 생성 실패

**증상:** `src/prompts/부서/기획본부/*.md` 파일이 Write 툴 실행 후 디스크에 존재하지 않음.

**원인:** Claude Code Write 툴이 한글 폴더명 경로에서 파일 생성에 실패. 폴더는 생성되나 파일은 저장되지 않음.

**해결:** PowerShell `[System.IO.File]::WriteAllText(path, content, UTF8)` 직접 호출로 우회.

**교훈:** 한글 경로 파일은 Write 툴 대신 PowerShell로 작성해야 함. 향후 세션에서도 동일 패턴 유지.

### 이슈 3 — save_briefing 타임스탬프 충돌

**증상:** `test_save_briefing_unique_filename` — 같은 조건으로 2회 호출 시 파일 1개만 생성됨.

**원인:** 타임스탬프 포맷이 초 단위(`%H%M%S`)여서 빠른 연속 호출 시 동일 파일명 생성.

**해결:** `%H%M%S%f` (마이크로초 포함)로 변경.

### 이슈 4 — OutputValidator 미등록 에이전트에 subject_name 체크 실행

**증상:** `OutputValidator.validate("없는에이전트", ...)` 호출 시 "대상자 이름 누락"으로 fail.

**원인:** `subject_name` 체크가 `rules` 유무와 무관하게 항상 실행됨.

**해결:** `if not rules: return ValidationResult(passed=True, ...)` early return 추가.

**설계 결정:** 미등록 에이전트는 검증 없이 pass — 알려진 에이전트만 FORMAT 규칙 강제.

### 이슈 6 — PowerShell stdin 파이프 한글 인코딩

**증상:** `echo "한글" | python main.py` 실행 시 `OSError: 파일 이름 구문이 잘못됨` — 한글이 깨진 폴더명으로 처리됨.

**원인:** PowerShell 기본 stdin 인코딩(cp949)과 Python 파일 경로 처리 불일치.

**해결:** 파이프 대신 스크립트 파일(`test_dryrun.py`)에 입력값 하드코딩 후 직접 실행.

**장기 대응:** `main.py`에 `--from-file` 옵션 추가 (Spiral 0-B 때).

### 이슈 7 — 프롬프트 파일에 [INPUT] 섹션 누락

**증상:** 에이전트가 `build_prompt()`에서 JSON을 `user_prompt`로 전달하는데, 프롬프트 파일(시스템 프롬프트)에 어떤 JSON 구조가 입력으로 오는지 설명이 없었음.

**원인:** 프롬프트 설계 시 시스템 프롬프트 vs 유저 프롬프트 구분 없이 FORMAT 섹션만 작성.

**해결:** 4개 에이전트 프롬프트 파일에 `[INPUT]` 섹션 추가 — JSON 키 구조와 각 필드 용도 명시.

**교훈:** 에이전트 시스템 프롬프트에는 항상 유저 프롬프트(입력) 구조를 `[INPUT]` 섹션으로 문서화해야 함.

### 이슈 8 — 유니코드 코드포인트 혼재 (석 U+C11D vs U+6790)

**증상:** `OutputValidator.RULES` 키에서 "분석"의 "석"이 Korean hangul `석` (U+C11D)인데, PowerShell here-string으로 작성한 파일에서는 CJK 한자 `석` (U+6790)으로 저장됨. 테스트 fixture에서도 동일 문제 발생.

**원인:** PowerShell `@'...'@` here-string 과 Edit/Write 툴이 "석"를 CJK(U+6790)로 처리. Python 소스 파일은 Korean(U+C11D)으로 처리. 두 문자는 육안으로 동일하나 코드포인트가 다름.

**증상 재현:**
```
FAILED TestValidate대상분석::test_pass_with_valid_output
failed_rules=['필수 섹션 누락: ## 대상 분석 결과']
```
(fixture의 "분석"이 U+6790이나 RULES의 "분석"은 U+C11D — 동일 검색 실패)

**해결:**
1. 한글 경로 파일 — Python `pathlib.Path.write_text(encoding="utf-8")` 전용 스크립트(`_write_prompts.py`)로 작성
2. 테스트 fixture — `OutputValidator.RULES`에서 직접 섹션 헤더를 꺼내 `repr()` 리터럴로 임베딩 (`_write_tests.py`)
3. 중복 파일(U+6790 버전) Python으로 탐지 후 삭제

**교훈:** 시각적으로 동일한 문자라도 Unicode 코드포인트가 다를 수 있음. 한글 파일명·한글 포함 내용은 Python `write_text()`로만 작성. PowerShell/Edit/Write 툴은 CJK 범위로 혼입 가능.

**장기 대응:** 한글 경로 파일 작성이 필요한 세션에는 Python helper 스크립트 우선 사용.
