# 품질관리 — CEO 판단 프레임워크
updated: 2026-05-16

## 사고흐름: 에이전트 결과물 품질 판단

CEO가 에이전트 산출물을 받았을 때 판단하는 순서.

### Step 1. 구조 검증 결과 확인
```
질문: OutputValidator 자동 검증을 통과했는가?
판단:
  - 통과 → Step 2로
  - 실패 → 즉시 rework (CEO 판단 불필요)
```

### Step 2. 대상자 반영도 체크
```
질문: 이 산출물에서 대상자 이름/정보를 빼면 여전히 성립하는가?
판단 방법:
  - 대상자 이름을 "홍길동"으로 바꿔도 내용이 똑같다면 → Generic
  - 대상자의 직업/경험/성격이 분석 근거로 사용되었다면 → Specific
출력: Generic / Specific 판정
```

### Step 3. 구체성 체크
```
질문: 조언이 즉시 실행 가능한 수준인가?
판단 방법:
  - "무엇을" 하라는지 명확한가?
  - "언제/얼마나" 구체적인가?
  - "어떻게" 시작하는지 알 수 있는가?
  - 셋 중 하나라도 모호하면 → 구체성 부족
출력: 충분 / 부족 (부족 항목 명시)
```

### Step 4. 에이전트별 핵심 기준 체크
```
질문: 해당 에이전트의 고유 품질 기준을 충족하는가?
판단 방법: 에이전트별 품질 초점 표 참조
출력: 충족 / 미충족 (미충족 항목 명시)
```

### Step 5. 최종 판정
```
판단 로직:
  - Step 2~4 모두 통과 → "합격" → 컨텍스트에 추가, 다음 에이전트로
  - 하나라도 실패 → "rework" → 구체적 개선 지시와 함께 재작업 요청
  - rework 후에도 실패 → 보고 조건 5번 (반복 실패) → 회장 보고
```

## 판단 원칙
- 의심스러우면 불합격 — 품질 기준은 보수적으로 적용
- rework 지시는 구체적이어야 한다 ("더 잘 해"는 지시가 아니다)
- "이 정도면 괜찮지 않나"는 CEO의 타협 — 크리에이터에게 줄 수 없으면 불합격
- 합격 기준: "핑구가 이걸 읽고 크리에이터에게 직접 전달할 수 있는가?"
