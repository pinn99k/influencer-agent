# 조직인사관리 — CEO 판단 프레임워크
updated: 2026-05-16

## 사고흐름: 에이전트 관리 판단

CEO가 에이전트 배정, 평가, 재배치를 판단하는 흐름.

### Step 1. 에이전트 배정
```
질문: 다음에 어떤 에이전트를 실행할 것인가?
판단 방법:
  - 기본 순서: 대상분석 → 경쟁분석 → 플랫폼추천 → 컨셉기획
  - 단, 상황에 따라 순서 변경 가능 (next_decision 프롬프트)
  - 변경 근거 예시:
    - 경쟁 분석 결과가 대상 분석 결과와 충돌 → 대상분석 재실행
    - 대상자 정보에 플랫폼 선호가 명확 → 플랫폼추천을 앞으로
  - 순서 건너뛰기는 금지 (4개 에이전트 모두 실행 필수)
출력: 다음 에이전트 + 핵심 지시
```

### Step 2. 결과 평가
```
질문: 이 에이전트의 산출물이 기준을 충족하는가?
판단 방법: 품질관리 프레임워크 참조
출력: 합격 / rework / 회장 보고
```

### Step 3. rework 판단
```
질문: rework 시 개선될 가능성이 있는가?
판단 방법:
  - 실패 원인이 프롬프트 미준수 → rework 효과 높음
  - 실패 원인이 정보 부족 → rework보다 컨텍스트 보강 필요
  - 실패 원인이 모델 한계 → rework 무의미, 모델 교체 검토
출력:
  - rework 진행 + 구체적 개선 지시
  - rework 무의미 → 회장 보고 (조건 5)
```

### Step 4. 성과 추적
```
질문: 이 에이전트의 누적 성과는 어떠한가?
판단 방법:
  - 1회 통과율 70% 이상 → 정상
  - 1회 통과율 50% 미만 → 프롬프트 재설계 검토
  - 특정 유형의 대상자에서만 실패 → 프롬프트에 해당 케이스 보강
출력: 성과 요약 + 필요 시 개선 방향
```

## 판단 원칙
- CEO는 에이전트를 "관리"하지 "대체"하지 않는다 — 직접 산출물 수정 금지
- rework 지시는 1회만 — 2회 실패면 시스템 문제로 판단, 회장 보고
- 에이전트 순서 변경은 근거가 있어야 한다 — 직감으로 변경 금지
- 에이전트 성과가 지속적으로 낮으면 에이전트 탓이 아니라 프롬프트 탓으로 본다
