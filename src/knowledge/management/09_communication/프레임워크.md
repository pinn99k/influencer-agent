# 커뮤니케이션 — CEO 판단 프레임워크
updated: 2026-05-16

## 사고흐름: 회장 보고문 작성

CEO가 회장 보고가 필요하다고 판단한 후, 보고문을 구성하는 흐름.

### Step 1. 보고 조건 확정
```
질문: 10가지 보고 조건 중 정확히 어디에 해당하는가?
판단: 리스크관리 프레임워크의 Step 4 결과 참조
출력: 조건 번호 + 조건명
```

### Step 2. 팩트 정리
```
질문: 객관적으로 무슨 일이 일어났는가?
원칙:
  - 해석이나 의견 배제, 사실만
  - 수치가 있으면 수치로
  - 에이전트 출력 원문 중 관련 부분 인용
출력: 현재 상황 2~3문장
```

### Step 3. 영향 평가
```
질문: 이대로 진행하면/하지 않으면 어떻게 되는가?
원칙:
  - 최선/최악 시나리오 모두 언급
  - 크리에이터에게 미치는 영향 중심
출력: 영향 1~2문장
```

### Step 4. 선택지 구성
```
질문: 회장이 선택할 수 있는 현실적 옵션은?
원칙:
  - 2~3개 선택지
  - 각 선택지에 장단점(트레이드오프) 명시
  - "아무것도 안 한다"도 고려
  - CEO 추천 1개 + 추천 근거
출력: 선택지 A/B(/C) + CEO 추천
```

### Step 5. 보고문 조립
```
형식:
  ## 회장 보고
  보고 조건: {번호}. {조건명}
  현재 상황: {Step 2 결과}
  영향: {Step 3 결과}
  선택지:
    A. {행동} — 장점: ..., 단점: ...
    B. {행동} — 장점: ..., 단점: ...
  CEO 추천: {A/B} — 이유: ...
```

## 사고흐름: rework 지시 작성

### Step 1. 실패 원인 특정
```
질문: 품질 판단에서 정확히 어떤 항목이 실패했는가?
출력: 실패 항목 목록 (예: "대상자 반영도 부족", "경쟁자 실명 누락")
```

### Step 2. 개선 지시 구성
```
질문: 에이전트가 정확히 무엇을 어떻게 고쳐야 하는가?
원칙:
  - 항목별 구체적 지시 (무엇을 + 어떻게)
  - 좋은 예시 제시 가능하면 포함
  - "전체 다시 해"가 아니라 "이 섹션의 이 부분을 이렇게"
출력: 개선 지시 목록
```

## 판단 원칙
- 결론 먼저 — 배경 설명으로 시작하지 않는다
- 모호한 표현 금지 — "약간", "아마", "그럴 수도" 대신 구체적 수치/사실
- 선택지 없는 보고는 보고가 아니라 푸념이다
- rework 지시는 1회로 끝나게 — 구체적이지 않은 지시가 반복 실패의 원인
